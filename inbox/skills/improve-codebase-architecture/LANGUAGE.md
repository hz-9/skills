# 语言（Language）

该技能中每条建议使用的共享词汇。请精确使用这些术语——不要替换为"component"、"service"、"API"或"boundary"。一致的用语是全部要点所在。

## 术语

**模块（Module）**
任何具有接口和实现的东西。故意保持规模无关——同样适用于函数、类、包或跨层切片。
_避免使用_：unit、component、service。

**接口（Interface）**
调用者为了正确使用模块而需要了解的一切。包括类型签名，但也包括不变量、顺序约束、错误模式、必需的配置和性能特征。
_避免使用_：API、signature（过于狭隘——它们仅指类型层面的表面）。

**实现（Implementation）**
模块内部的东西——它的代码主体。与**适配器（Adapter）**不同：一个东西可以是带有大量实现的小适配器（一个 Postgres 仓库），也可以是带有少量实现的大适配器（一个内存 fake）。当接缝是主题时，使用"adapter"；否则使用"implementation"。

**深度（Depth）**
接口处的杠杆效应——调用者（或测试）每学习一个单位的接口所能使用的行为量。当一个模块在小型接口背后隐藏了大量行为时，它是**深的（deep）**。当一个模块的接口几乎和其实现一样复杂时，它是**浅的（shallow）**。

**接缝（Seam）** _（源自 Michael Feathers）_
一个可以在不修改该处代码的情况下改变行为的位置。模块接口所在的*位置*。选择在哪里放置接缝本身就是一个设计决策，与接缝后面有什么是分开的。
_避免使用_：boundary（与 DDD 的限界上下文产生过载）。

**适配器（Adapter）**
在接缝处满足接口的具体实现。描述*角色*（它填充哪个槽），而非实质（里面有什么）。

**杠杆效应（Leverage）**
调用者从深度中获得的东西。每学习一个单位的接口，就能获得更多能力。一个实现跨越 N 个调用点和 M 个测试进行回报。

**局部性（Locality）**
维护者从深度中获得的东西。变更、缺陷、知识和验证集中在一个地方，而不是分散在调用者之间。一处修复，处处生效。

## 原则

- **深度是接口的属性，而非实现的属性。**一个深层模块可以在内部由小的、可模拟的、可替换的部件组成——只是它们不是接口的一部分。一个模块可以有**内部接缝**（对其实现私有，由其自身测试使用），也可以有其接口处的**外部接缝**。
- **删除测试。**想象一下删除该模块。如果复杂性消失了，说明该模块没有隐藏任何东西（它只是一个透传）。如果复杂性在 N 个调用者处重新出现，说明该模块在发挥其作用。
- **接口就是测试面。**调用者和测试跨越同一个接缝。如果你想在接口*之后*进行测试，说明该模块的形状可能有问题。
- **一个适配器意味着一个假设性的接缝。两个适配器才意味着一个真实的接缝。**除非某个东西确实在接缝处变化，否则不要引入接缝。

## 关系

- 一个**模块**有且仅有一个**接口**（它呈现给调用者和测试的表面）。
- **深度**是一个**模块**的属性，相对于其**接口**来衡量。
- **接缝**是**模块**的**接口**所在的位置。
- **适配器**位于**接缝**处，满足**接口**的要求。
- **深度**为调用者产生**杠杆效应**，为维护者产生**局部性**。

## 已拒绝的框架

- **深度作为实现行数与接口行数的比率**（Ousterhout）：会奖励填充实现。我们改用深度即杠杆效应。
- **"Interface"作为 TypeScript 的 `interface` 关键字或类的公共方法**：过于狭隘——这里的接口包括调用者必须了解的每一个事实。
- **"Boundary"**：与 DDD 的限界上下文产生过载。使用**seam**或**interface**。
