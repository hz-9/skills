# 接口设计

当用户希望为选定的深化候选项探索替代接口时，使用这种并行子代理模式。基于"设计两次"原则（Ousterhout）——你的第一个想法不太可能是最好的。

使用 [LANGUAGE.md](LANGUAGE.md) 中的词汇——**模块（module）**、**接口（interface）**、**接缝（seam）**、**适配器（adapter）**、**杠杆效应（leverage）**。

## 流程

### 1. 界定问题空间

在生成子代理之前，为选定的候选项编写面向用户的问题空间说明：

- 任何新接口需要满足的约束条件
- 它将依赖的依赖项，以及这些依赖项所属的类别（参见 [DEEPENING.md](DEEPENING.md)）
- 一个粗略的示意性代码草图，用于将约束具体化——不是一个提案，只是让约束变得具体的一种方式

将其展示给用户，然后立即进入第 2 步。用户在子代理并行工作时阅读和思考。

### 2. 生成子代理

使用 Agent 工具并行生成 3 个以上的子代理。每个子代理必须为深化后的模块生成一个**截然不同**的接口。

为每个子代理提供独立的技术简报（文件路径、耦合细节、[DEEPENING.md](DEEPENING.md) 中的依赖类别、接缝背后是什么）。该简报与第 1 步中面向用户的问题空间说明是独立的。为每个代理提供不同的设计约束：

- 代理 1："最小化接口——最多 1-3 个入口点。最大化每个入口点的杠杆效应。"
- 代理 2："最大化灵活性——支持多个用例和扩展。"
- 代理 3："为最常见的调用者优化——使默认用例变得简单。"
- 代理 4（如适用）："围绕跨接缝依赖的端口与适配器进行设计。"

在简报中同时包含 [LANGUAGE.md](LANGUAGE.md) 词汇和 CONTEXT.md 词汇，以便每个子代理的命名与架构语言及项目的领域语言保持一致。

每个子代理输出：

1. 接口（类型、方法、参数——以及不变量、顺序约束、错误模式）
2. 使用示例，展示调用者如何使用它
3. 实现隐藏在接缝背后的内容
4. 依赖策略和适配器（参见 [DEEPENING.md](DEEPENING.md)）
5. 权衡——杠杆效应高的地方和薄弱的地方

### 3. 展示和比较

依次展示设计方案，让用户逐个吸收，然后用文字进行比较。通过**深度**（接口处的杠杆效应）、**局部性**（变更集中的位置）和**接缝位置**进行对比。

比较后，给出你自己的推荐：你认为哪个设计最强以及原因。如果不同设计中的元素可以很好地组合，提出一个混合方案。要有主见——用户想要的是有见地的判断，而不是一个菜单。
