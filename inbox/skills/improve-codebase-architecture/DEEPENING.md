# 深化（Deepening）

如何安全地深化一个由浅层模块组成的集群，同时考虑其依赖关系。假设已掌握 [LANGUAGE.md](LANGUAGE.md) 中的词汇——**模块（module）**、**接口（interface）**、**接缝（seam）**、**适配器（adapter）**。

## 依赖类别

在评估候选模块进行深化时，对其依赖进行分类。类别决定了深化后的模块如何在其接缝处进行测试。

### 1. 进程内（In-process）

纯计算、内存状态、无 I/O。始终可以深化——合并模块，直接通过新接口进行测试。无需适配器。

### 2. 本地可替代（Local-substitutable）

具有本地测试替代品的依赖（PGLite 替代 Postgres，内存文件系统）。如果存在替代品，则可以深化。深化后的模块在测试套件中使用该替代品进行测试。接缝是内部的；模块外部接口上没有端口。

### 3. 远程但自有的（端口与适配器）

你自有的跨网络边界的服务（微服务、内部 API）。在接缝处定义一个**端口（port）**（接口）。深层模块拥有逻辑；传输层作为**适配器（adapter）**注入。测试使用内存适配器。生产环境使用 HTTP/gRPC/队列适配器。

推荐的表述：*"在接缝处定义一个端口，实现一个 HTTP 适配器用于生产环境，一个内存适配器用于测试，这样即使部署在网络边界上，逻辑也能位于一个深层模块中。"*

### 4. 真正的外部服务（模拟）

你不控制的第三方服务（Stripe、Twilio 等）。深化后的模块将外部依赖作为注入的端口接收；测试提供模拟适配器。

## 接缝规范

- **一个适配器意味着一个假设性的接缝。两个适配器才意味着一个真实的接缝。**除非至少有两个适配器是合理的（通常是生产环境 + 测试环境），否则不要引入端口。单一适配器的接缝只是间接层。
- **内部接缝 vs 外部接缝。**深层模块可以有内部接缝（对其实现私有，由其自身测试使用），也可以有其接口处的外部接缝。不要仅仅因为测试使用它们就将内部接缝暴露到接口中。

## 测试策略：替换，而非叠加

- 一旦在深化模块的接口处有了测试，浅层模块上的旧单元测试就变成了废物——删除它们。
- 在深化模块的接口处编写新测试。**接口就是测试面**。
- 测试通过接口断言可观察到的结果，而不是内部状态。
- 测试应当能够承受内部重构——它们描述的是行为，而非实现。如果一个测试在实现变更时必须修改，那它就是在测试接口之后的东西。
