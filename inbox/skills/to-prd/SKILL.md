该 skill 获取当前对话上下文和代码库理解，并生成一个 PRD。**不要**追问用户——只需综合你已经知道的信息。

Issue 跟踪器和 triage 标签词汇表应该已经提供给你了——如果没有，运行 `/setup-matt-pocock-skills`。

## 流程

1. 浏览仓库以了解代码库的当前状态，如果你还没有这样做的话。在整个 PRD 中使用项目的领域术语表词汇，并尊重你正在接触的任何区域的 ADR。

2. 勾勒出你需要构建或修改以完成实现的主要模块。积极寻找可以提取出可以独立测试的深模块的机会。

深模块（相对于浅模块）是将大量功能封装在一个简单、可测试且很少变化的接口中的模块。

与用户确认这些模块是否符合他们的期望。与用户确认他们希望为哪些模块编写测试。

3. 使用下面的模板编写 PRD，然后将其发布到项目 issue 跟踪器。应用 `ready-for-agent` triage 标签——无需额外的 triage。

<prd-template>

## 问题陈述

用户面临的问题，从用户的角度出发。

## 解决方案

问题的解决方案，从用户的角度出发。

## 用户故事

一个**长**的、编号的用户故事列表。每个用户故事的格式应为：

1. 作为一个<角色>，我想要一个<功能>，以便<收益>

<user-story-example>
1. 作为一个手机银行客户，我想要查看我账户上的余额，以便我能更明智地做出消费决策
</user-story-example>

这个用户故事列表应该非常详尽，涵盖该功能的各个方面。

## 实现决策

所做的实现决策列表。可以包括：

- 将要构建/修改的模块
- 将要修改的模块的接口
- 来自开发者的技术澄清
- 架构决策
- Schema 变更
- API 契约
- 特定交互

不要包含具体的文件路径或代码片段。它们可能很快就会过时。

例外：如果原型产生了一个比文字更精确地编码决策的片段（状态机、reducer、schema、类型形状），将其内联在相关决策中，并简要说明它来自原型。裁剪到包含决策的部分——不是工作演示，只是重要的部分。

## 测试决策

所做的测试决策列表。包括：

- 什么构成了一个好的测试的描述（只测试外部行为，不测试实现细节）
- 哪些模块将被测试
- 测试的现有类似示例（即代码库中类似类型的测试）

## 范围之外

本 PRD 范围之外的内容的描述。

## 进一步说明

关于该功能的任何进一步说明。

</prd-template>
