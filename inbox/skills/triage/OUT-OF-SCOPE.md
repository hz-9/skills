# 范围外知识库

仓库中的 `.out-of-scope/` 目录存储被拒绝的功能请求的持久记录。它有两个目的：

1. **机构记忆**——功能为什么被拒绝，这样当 Issue 关闭时推理不会丢失
2. **去重**——当新 Issue 与先前的拒绝匹配时，技能可以呈现先前的决定，而不是重新争论

## 目录结构

```
.out-of-scope/
├── dark-mode.md
├── plugin-system.md
└── graphql-api.md
```

每个**概念**一个文件，而不是每个 Issue 一个文件。多个请求相同内容的 Issue 被归组到同一个文件下。

## 文件格式

文件应以轻松、可读的风格编写——更像一个简短的设计文档而不是数据库条目。使用段落、代码示例和示例来使推理清晰明了，对首次遇到它的人有用。

```markdown
# 深色模式

本项目不支持深色模式或面向用户的主题设置。

## 为什么这不在范围内

渲染管道假定在 `ThemeConfig` 中定义了单一调色板。
支持多个主题需要：

- 一个主题上下文提供者包裹整个组件树
- 每个组件都需要支持主题的样式解析
- 用户主题偏好的持久化层

这是一个重大的架构变更，与项目专注于内容创作的方向不符。
主题设置是嵌入或重新分发输出的下游消费者的关注点。

```ts
// 当前的 ThemeConfig 接口不是为运行时切换设计的：
interface ThemeConfig {
  colors: ColorPalette; // 单一调色板，在构建时解析
  fonts: FontStack;
}
```

## 先前的请求

- #42 — "添加深色模式支持"
- #87 — "用于无障碍的夜间主题"
- #134 — "深色主题选项"
```

### 文件命名

使用简短、描述性的 kebab-case 名称来命名概念：`dark-mode.md`、`plugin-system.md`、`graphql-api.md`。名称应该足够可识别，使得浏览目录的人无需打开文件就能理解被拒绝的是什么。

### 编写理由

理由应该是实质性的——不是"我们不想要这个"，而是为什么。好的理由引用：

- 项目范围或理念（"本项目专注于 X；主题设置是下游的关注点"）
- 技术约束（"支持这将需要 Y，这与我们的 Z 架构冲突"）
- 战略决策（"我们选择使用 A 而不是 B，因为..."）

理由应该是持久的。避免引用临时情况（"我们现在太忙了"）——那些不是真正的拒绝，而是推迟。

## 何时检查 `.out-of-scope/`

在分类期间（第 1 步：收集上下文），读取 `.out-of-scope/` 中的所有文件。在评估新 Issue 时：

- 检查请求是否匹配现有的范围外概念
- 匹配基于概念相似性，而不是关键词——"夜间主题"匹配 `dark-mode.md`
- 如果有匹配，向维护者呈现："这与 `.out-of-scope/dark-mode.md` 类似——我们以前拒绝过这个，因为 [理由]。你仍然这么认为吗？"

维护者可能会：

- **确认**——新 Issue 被添加到现有文件的"Prior requests"列表中，然后关闭
- **重新考虑**——范围外文件被删除或更新，Issue 通过正常分类流程进行
- **不同意**——这些 Issue 相关但不同，按正常分类流程进行

## 何时写入 `.out-of-scope/`

仅当**enhancement**（非 bug）被标记为 `wontfix` 拒绝时。流程：

1. 维护者决定某个功能请求不在范围内
2. 检查是否存在匹配的 `.out-of-scope/` 文件
3. 如果存在：将新 Issue 追加到"Prior requests"列表
4. 如果不存在：使用概念名称、决定、理由和第一个先前的请求创建新文件
5. 在 Issue 上发布评论，解释决定并提及 `.out-of-scope/` 文件
6. 使用 `wontfix` 标签关闭 Issue

## 更新或删除范围外文件

如果维护者改变了之前拒绝的概念的想法：

- 删除 `.out-of-scope/` 文件
- 技能不需要重新打开旧 Issue——它们是历史记录
- 触发重新考虑的新 Issue 通过正常分类流程进行
