# 编写 Agent Brief

Agent brief 是一个结构化的评论，在 Issue 移入 `ready-for-agent` 时发布在 GitHub Issue 上。它是 AFK agent 将依据的权威规范。原始的 Issue 正文和讨论是上下文——agent brief 是契约。

## 原则

### 持久性优于精确性

Issue 可能在 `ready-for-agent` 中停留数天或数周。在此期间代码库会发生变化。编写 brief 时要使其即使在文件被重命名、移动或重构后仍然有用。

- **要**描述接口、类型和行为契约
- **要**指定 agent 应该查找或修改的具体类型、函数签名或配置形状
- **不要**引用文件路径——它们会过时
- **不要**引用行号
- **不要**假设当前的实现结构将保持不变

### 行为性，而非过程性

描述系统**应该做什么**，而不是**如何实现**。Agent 将重新探索代码库并自行做出实现决策。

- **好：**"`SkillConfig` 类型应该接受一个可选的 `schedule` 字段，类型为 `CronExpression`"
- **差：**"打开 src/types/skill.ts 并在第 42 行添加一个 schedule 字段"
- **好：**"当用户不带参数运行 `/triage` 时，他们应该看到需要关注的 Issue 摘要"
- **差：**"在主处理函数中添加一个 switch 语句"

### 完整的验收标准

Agent 需要知道何时完成。每个 agent brief 必须有具体的、可测试的验收标准。每个标准应可独立验证。

- **好：**"运行 `gh issue list --label needs-triage` 返回已通过初始分类的 Issue"
- **差：**"分类应该正常工作"

### 明确的范围边界

说明什么是范围之外的。这可以防止 agent 过度设计或对相邻功能做出假设。

## 模板

```markdown
## Agent Brief

**类别：** bug / enhancement
**摘要：** 一行描述需要做什么

**当前行为：**
描述现在发生什么。对于 bug，这是有问题的行为。
对于 enhancement，这是该功能所基于的现状。

**期望行为：**
描述在 agent 工作完成后应该发生什么。
对边界情况和错误条件要具体。

**关键接口：**
- `TypeName` — 需要更改什么以及为什么
- `functionName()` 返回类型 — 它当前返回什么 vs 应该返回什么
- 配置形状 — 任何需要的新配置选项

**验收标准：**
- [ ] 具体的、可测试的标准 1
- [ ] 具体的、可测试的标准 2
- [ ] 具体的、可测试的标准 3

**范围之外：**
- 不应在此 Issue 中更改或解决的事项
- 可能看起来相关但实际上是独立的相邻功能
```

## 示例

### 好的 agent brief（bug）

```markdown
## Agent Brief

**类别：** bug
**摘要：** 技能描述的截断在单词中间断开，产生损坏的输出

**当前行为：**
当技能描述超过 1024 个字符时，它会在恰好 1024 个字符处截断，
不考虑单词边界。这会产生在单词中间结束的描述
（例如"Use when the user wants to confi"）。

**期望行为：**
截断应在 1024 个字符之前的最后一个单词边界处断开，
并附加"..."以表示截断。

**关键接口：**
- `SkillMetadata` 类型的 `description` 字段——不需要类型变更，
  但填充它的验证/处理逻辑需要尊重单词边界
- 任何读取 SKILL.md 前置元数据并提取描述的函数

**验收标准：**
- [ ] 少于 1024 个字符的描述不变
- [ ] 超过 1024 个字符的描述在 1024 个字符之前的最后一个
      单词边界处截断
- [ ] 截断后的描述以"..."结尾
- [ ] 包含"..."的总长度不超过 1024 个字符

**范围之外：**
- 更改 1024 个字符的限制本身
- 多行描述支持
```

### 好的 agent brief（enhancement）

```markdown
## Agent Brief

**类别：** enhancement
**摘要：** 添加 `.out-of-scope/` 目录支持，用于跟踪被拒绝的功能请求

**当前行为：**
当功能请求被拒绝时，Issue 以 `wontfix` 标签和一条评论关闭。
没有关于该决定或推理的持久记录。
未来的类似请求需要维护者回忆或搜索先前的讨论。

**期望行为：**
被拒绝的功能请求应记录在 `.out-of-scope/<concept>.md` 文件中，
这些文件捕获决定、推理以及请求该功能的所有 Issue 的链接。
在分类新 Issue 时，应检查这些文件以查找匹配项。

**关键接口：**
- `.out-of-scope/` 中的 Markdown 文件格式——每个文件应有一个
  `# Concept Name` 标题、一个 `**Decision:**` 行、一个 `**Reason:**` 行，
  以及一个包含 Issue 链接的 `**Prior requests:**` 列表
- 分类工作流应尽早读取所有 `.out-of-scope/*.md` 文件，
  并通过概念相似性将传入的 Issue 与它们进行匹配

**验收标准：**
- [ ] 将功能以 wontfix 关闭会在 `.out-of-scope/` 中创建/更新一个文件
- [ ] 该文件包含决定、推理和已关闭 Issue 的链接
- [ ] 如果匹配的 `.out-of-scope/` 文件已存在，新 Issue 会被追加到
      其"Prior requests"列表中，而不是创建重复文件
- [ ] 在分类期间，检查现有的 `.out-of-scope/` 文件，并在新 Issue
      与先前的拒绝匹配时将其呈现出来

**范围之外：**
- 自动匹配（由人类确认匹配）
- 重新打开以前被拒绝的功能
- Bug 报告（只有 enhancement 拒绝才进入 `.out-of-scope/`）
```

### 坏的 agent brief

```markdown
## Agent Brief

**摘要：** 修复分类 bug

**要做什么：**
分类功能有问题。查看主文件并修复它。
第 150 行附近的函数有问题。

**要更改的文件：**
- src/triage/handler.ts（第 150 行）
- src/types.ts（第 42 行）
```

这是坏的原因：
- 没有类别
- 描述模糊（"分类功能有问题"）
- 引用会过时的文件路径和行号
- 没有验收标准
- 没有范围边界
- 没有描述当前行为与期望行为
