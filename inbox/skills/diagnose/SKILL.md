# 诊断

处理疑难 bug 的一套纪律。仅在明确有充分理由时才跳过某个阶段。

在浏览代码库时，使用项目的领域术语表来建立相关模块的清晰心智模型，并检查你正在接触的区域的 ADR。

## 第一阶段 — 构建反馈循环

**这是核心技能。** 其他一切都是机械性的。如果你有一个快速、确定性、可由 agent 运行的成功/失败信号来判断 bug，你就会找到原因——二分查找、假设检验和检测都只是消耗那个信号。如果你没有这样一个信号，盯着代码看再多也救不了你。

在此处投入不成比例的精力。**要积极。要有创意。绝不放弃。**

### 构建反馈循环的方法——大致按此顺序尝试

1. **失败的测试**——在任何能触及 bug 的接缝处——单元测试、集成测试、e2e 测试。
2. **Curl / HTTP 脚本**——对着正在运行的开发服务器执行。
3. **CLI 调用**——使用固定输入，将 stdout 与已知正确的快照进行 diff。
4. **无头浏览器脚本**（Playwright / Puppeteer）——驱动 UI，在 DOM/控制台/网络上进行断言。
5. **回放捕获的跟踪**——将真实的网络请求/负载/事件日志保存到磁盘；在隔离环境中通过代码路径回放它。
6. **一次性测试工具**——启动系统的一个最小子集（一个服务，mock 的依赖项），通过单个函数调用即可执行 bug 代码路径。
7. **属性/模糊测试循环**——如果 bug 是"有时输出错误"，运行 1000 个随机输入并查找失败模式。
8. **二分查找工具**——如果 bug 出现在两个已知状态（提交、数据集、版本）之间，自动化"在状态 X 启动，检查，重复"，以便你可以用 `git bisect run` 来运行它。
9. **差异循环**——将相同的输入分别通过旧版本和新版本（或两种配置）运行，并 diff 输出。
10. **HITL bash 脚本**——最后的手段。如果必须由人类点击操作，用 `scripts/hitl-loop.template.sh` 来_驱动他们_，使循环仍然是有结构的。捕获的输出反馈给你。

构建正确的反馈循环，bug 就解决了 90%。

### 在循环本身上迭代

将循环当作一个产品来对待。一旦你有了_一个_循环，问问：

- 我能让它更快吗？（缓存设置，跳过不相关的初始化，缩小测试范围。）
- 我能让信号更清晰吗？（针对具体症状进行断言，而不是"没有崩溃"。）
- 我能让它更确定吗？（固定时间，设置随机种子，隔离文件系统，冻结网络。）

一个 30 秒的不稳定循环比没有循环好不了多少。一个 2 秒的确定性循环是调试的超能力。

### 非确定性 bug

目标不是完美的复现，而是**更高的复现率**。循环触发 100 次，并行化，增加压力，缩小时间窗口，注入 sleep。一个 50% 概率触发的 bug 是可调试的；1% 则不是——持续提高复现率直到它变得可调试。

### 当你确实无法构建循环时

停下来并明确说明。列出你尝试过的方法。向用户请求：(a) 访问能复现该问题的环境，(b) 一个捕获的制品（HAR 文件、日志转储、核心转储、带时间戳的屏幕录制），或 (c) 添加临时生产环境检测的权限。**在没有反馈循环的情况下，不要继续提出假设。**

在拥有你确信有效的循环之前，不要进入第二阶段。

## 第二阶段 — 重现

运行循环。观察 bug 出现。

确认：

- [ ] 循环产生了**用户**描述的失败模式——而不是恰好附近的另一个不同失败。错误的 bug = 错误的修复。
- [ ] 失败在多次运行中是可重现的（或者，对于非确定性 bug，以足够高的复现率可调试）。
- [ ] 你已经捕获了确切的症状（错误消息、错误输出、缓慢的计时），以便后续阶段可以验证修复是否真正解决了问题。

在重现 bug 之前不要继续。

## 第三阶段 — 提出假设

在测试任何假设之前，生成 **3–5 个排序后的假设**。单一假设的生成会锚定在第一个看似合理的想法上。

每个假设必须是**可证伪的**：陈述它所做出的预测。

> 格式："如果 <X> 是原因，那么 <改变 Y> 将使 bug 消失 / <改变 Z> 将使 bug 更严重。"

如果你无法陈述预测，那这个假设只是一种感觉——丢弃它或使其更精确。

**在测试之前向用户展示排序列表。** 他们通常拥有可以立即重新排序的领域知识（"我们刚刚部署了对 #3 的修改"），或者知道他们已经排除过的假设。便宜的检查点，能节省大量时间。如果用户不在，可按你的排序继续。

## 第四阶段 — 检测

每个探测必须对应第三阶段中的一个特定预测。**一次只改变一个变量。**

工具偏好：

1. **调试器 / REPL 检查**——如果环境支持的话。一个断点胜过十条日志。
2. **针对性日志**——在能区分不同假设的边界处。
3. 永远不要"记录所有内容然后 grep"。

**为每个调试日志打上标签**，使用唯一前缀，例如 `[DEBUG-a4f2]`。最后的清理只需一个 grep 即可完成。未打标签的日志保留；打标签的日志删除。

**性能分支。** 对于性能回退，日志通常是不对的。替代方案：建立基线测量（计时工具、`performance.now()`、性能分析器、查询计划），然后进行二分查找。先测量，再修复。

## 第五阶段 — 修复 + 回归测试

在**修复之前**编写回归测试——但前提是存在一个**正确的接缝**。

正确的接缝是指测试能像在调用点发生的那样，对**真正的 bug 模式**进行测试。如果唯一可用的接缝太浅（当 bug 需要多个调用者时只有一个调用者测试，无法复制触发 bug 的链条的单元测试），那么在那里的回归测试会带来虚假的信心。

**如果没有正确的接缝存在，这本身就是一个发现。** 记录下来。代码库架构阻止了该 bug 被锁定。将此标记给下一阶段。

如果存在正确的接缝：

1. 将最小化的复现转化为该接缝上的一个失败测试。
2. 观察它失败。
3. 应用修复。
4. 观察它通过。
5. 针对原始（未最小化的）场景重新运行第一阶段的反馈循环。

## 第六阶段 — 清理 + 事后分析

在宣布完成之前需要完成以下事项：

- [ ] 原始复现不再复现（重新运行第一阶段的循环）
- [ ] 回归测试通过（或接缝缺失已记录）
- [ ] 所有 `[DEBUG-...]` 检测已被移除（用 `grep` 搜索前缀）
- [ ] 一次性原型已删除（或移至标记清晰的调试位置）
- [ ] 被证明正确的假设已在提交/PR 消息中陈述——以便下一个调试者学习

**然后问：什么本来可以防止这个 bug？** 如果答案涉及架构变更（没有好的测试接缝、调用者纠缠不清、隐藏的耦合），将具体细节移交给 `/improve-codebase-architecture` skill。在**修复完成之后**提出建议，而不是之前——你现在拥有的信息比开始时更多。
