# 转换为 Issues

使用垂直切片（tracer bullets）将计划分解为可独立处理的 issues。

Issue 跟踪器和 triage 标签词汇表应该已经提供给你了——如果没有，运行 `/setup-matt-pocock-skills`。

## 流程

### 1. 收集上下文

根据对话上下文中已有的内容进行工作。如果用户传递了一个 issue 引用（issue 编号、URL 或路径）作为参数，从 issue 跟踪器中获取它，并阅读其完整正文和评论。

### 2. 浏览代码库（可选）

如果你还没有浏览过代码库，那么去了解代码的当前状态。Issue 标题和描述应使用项目的领域术语表词汇，并尊重你正在接触的区域的 ADR。

### 3. 起草垂直切片

将计划分解为 **tracer bullet** issues。每个 issue 是一个薄的垂直切片，贯穿**所有**集成层端到端，**而不是**一个层次的水平切片。

切片可以是 'HITL' 或 'AFK'。HITL 切片需要人类交互，例如架构决策或设计评审。AFK 切片可以在没有人类交互的情况下实现和合并。尽可能优先选择 AFK 而非 HITL。

<vertical-slice-rules>
- 每个切片提供一条狭窄但**完整**的路径，贯穿每一层（schema、API、UI、测试）
- 一个完成的切片可以独立演示或验证
- 优先选择多个薄切片而不是少数厚切片
</vertical-slice-rules>

### 4. 询问用户

将提议的分解展示为一个编号列表。对于每个切片，显示：

- **标题**：简短的描述性名称
- **类型**：HITL / AFK
- **被阻塞**：哪些其他切片（如果有）必须先完成
- **覆盖的用户故事**：这解决了哪些用户故事（如果源材料中有的话）

询问用户：

- 粒度感觉合适吗？（太粗 / 太细）
- 依赖关系正确吗？
- 是否有任何切片应该合并或进一步拆分？
- HITL 和 AFK 的标记是否正确？

迭代直到用户批准该分解。

### 5. 发布 issues 到 issue 跟踪器

对于每个已批准的切片，向 issue 跟踪器发布一个新 issue。使用下面的 issue 正文模板。这些 issues 被认为是准备好供 AFK agents 处理的，因此除非另有指示，使用正确的 triage 标签发布它们。

按依赖顺序发布 issues（阻塞者优先），以便你可以在"被阻塞"字段中引用真实的 issue 标识符。

<issue-template>
## 父级

指向 issue 跟踪器上父级 issue 的引用（如果源材料是现有 issue，否则省略此部分）。

## 要构建什么

对此垂直切片的简洁描述。描述端到端的行为，而不是逐层的实现。

避免具体的文件路径或代码片段——它们很快就会过时。例外：如果原型产生了一个比文字更精确地编码决策的片段（状态机、reducer、schema、类型形状），在此处内联它，并简要说明它来自原型。裁剪到包含决策的部分——不是工作演示，只是重要的部分。

## 验收标准

- [ ] 标准 1
- [ ] 标准 2
- [ ] 标准 3

## 被阻塞

- 指向阻塞工单的引用（如果有）

或者"无 - 可以立即开始"如果没有阻塞者。

</issue-template>

不要关闭或修改任何父级 issue。
