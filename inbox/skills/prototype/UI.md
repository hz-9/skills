# UI 原型

在单个路由上生成**几个截然不同的 UI 变体**，可通过浮动底部栏切换。用户在浏览器中切换变体，选择一个（或从每个变体中摘取部分），然后丢弃其余部分。

如果问题是关于逻辑/状态而不是外观——走错了分支。请使用 [LOGIC.md](LOGIC.md)。

## 何时适合这种形式

- "这个页面应该长什么样？"
- "我想在确定之前看一下这个仪表盘的几个选项。"
- "为设置页面尝试一个不同的布局。"
- 任何用户本来会花一天时间在他们头脑中的三个模糊线框图之间做选择的情况。

## 两种子形式——强烈推荐子形式 A

UI 原型在**与应用的其他部分互相结合**时更容易判断——真实的头部、真实的侧边栏、真实的数据、真实的密度。一个单独的一次性路由是一个真空：每个变体在孤立状态下看起来都不错。只要存在一个合理的现有页面来承载这些变体，就默认使用子形式 A。只有当原型确实没有附近的归属页面时才使用子形式 B。

### 子形式 A——对现有页面的调整（首选）

路由已存在。变体在**同一路由**上渲染，通过 `?variant=` URL 查询参数控制。现有的数据获取、参数和认证全部保留——只有渲染部分被替换。这是默认选择；除非有特定原因不这样做。

如果原型针对的是还没有页面但*自然应该位于某个页面内*的东西（仪表盘的新部分、设置页面的新卡片、现有流程中的新步骤）——那仍然是子形式 A。将变体挂载到宿主页面内。

### 子形式 B——新页面（最后手段）

仅当被原型化的东西确实没有现有页面可以放入时才使用——例如，一个全新的顶层页面，或者一个无法合理嵌入任何地方的流程。

创建一个**一次性路由**，遵循项目已有的任何路由约定——不要创建新的顶层结构。命名要明显表明是原型（例如，在路径或文件名中包含 `prototype` 一词）。使用相同的 `?variant=` 模式。

在确定使用子形式 B 之前，做一次合理性检查：真的没有现有页面可以嵌入这个原型吗？一个空的路由会隐藏设计问题，而一个有内容的路由则会暴露它们。

在两种子形式中，浮动底部栏是相同的。

## 流程

### 1. 明确问题并选择 N

默认选择 **3 个变体**。超过 5 个就不再是截然不同，而开始变成噪音——以此为上限。

在原型的位置或文件顶部注释中用一行写下计划：

> "设置页面的三个变体，通过 `?variant=` 切换，在现有的 `/settings` 路由上。"

无论用户是否在此提出反对意见，这都有效。

### 2. 生成截然不同的变体

起草每个变体。让每个变体满足：

- 页面的目的及其可访问的数据。
- 项目的组件库/样式系统（TailwindCSS、shadcn、MUI、纯 CSS 等）。
- 清晰的导出组件名称，例如 `VariantA`、`VariantB`、`VariantC`。

变体必须是**结构上不同的**——不同的布局、不同的信息层次、不同的主要交互方式，而不仅仅是不同的颜色。三个略微调整的卡片网格不是 UI 原型，而是壁纸。如果两个草稿过于相似，用明确的"不要使用卡片网格"指导重做其中一个。

### 3. 将它们连接起来

在路由上创建一个切换器组件：

```tsx
// 伪代码——适配到项目的框架
const variant = searchParams.get('variant') ?? 'A';
return (
  <>
    {variant === 'A' && <VariantA {...data} />}
    {variant === 'B' && <VariantB {...data} />}
    {variant === 'C' && <VariantC {...data} />}
    <PrototypeSwitcher variants={['A','B','C']} current={variant} />
  </>
);
```

对于子形式 A（现有页面）：保持切换器上方的所有现有数据获取不变；只有渲染的子树随变体变化。

对于子形式 B（新页面）：`/prototype/<name>` 下的一次性路由挂载相同的切换器。

### 4. 构建浮动切换器

一个固定在屏幕底部中央的小型位置栏，包含三个部分：

- **左箭头** — 循环到上一个变体（循环到头时回到最后一个）。
- **变体标签** — 显示当前变体键，如果变体导出名称，也显示该名称。例如 `B — 侧边栏布局`。
- **右箭头** — 循环到下一个（循环到尾时回到第一个）。

行为：

- 点击箭头更新 URL 查询参数（使用框架的路由器——Next 用 `router.replace`，React Router 用 `navigate` 等），以便变体可分享且刷新后保持。
- 键盘：`←` 和 `→` 方向键也可循环。当 `<input>`、`<textarea>` 或 `[contenteditable]` 获得焦点时，不要拦截方向键。
- 视觉上区别于页面（例如高对比度药丸形状、微妙的阴影），以明显表明它不是被评估的设计的一部分。
- 在生产构建中隐藏——通过 `process.env.NODE_ENV !== 'production'` 或等效检查来控制，以免意外的原型合并将工具栏发布给用户。

将切换器放在一个共享组件中，以便两种子形式都能重用它。将其放在项目中共享 UI 所在的位置。

### 5. 交给用户

给出 URL（以及 `?variant=` 键）。用户会在有空时翻看查看。有趣的反馈通常是**"我想要 B 的头部和 C 的侧边栏"**——那才是他们真正想要的设计。

### 6. 记录答案并清理

一旦某个变体胜出，记录下来是哪一个以及为什么（提交信息、ADR、issue，或者如果不在旁且用户尚未回应，在原型旁边放一个 `NOTES.md`）。然后：

- **子形式 A** — 删除失败的变体和切换器；将胜出者合并到现有页面中。
- **子形式 B** — 将胜出的变体提升为正式路由，删除一次性路由和切换器。

不要留下变体组件或切换器。它们会迅速腐化并让后续的读者感到困惑。

## 反模式

- **仅在颜色或文字上不同的变体。**那是一个调整，不是原型。真正的变体在结构上存在分歧。
- **在变体之间共享过多代码。**共享的 `<Header>` 没问题；共享的 `<Layout>` 则违背了目的。每个变体应该可以自由地抛弃布局。
- **将变体连接到真实的修改操作。**只读原型完全可以。如果变体需要修改，指向一个桩——问题是"这应该长什么样"，而不是"后端是否工作"。
- **直接将原型提升到生产环境。**变体代码是在原型约束下编写的（没有测试、最小限度的错误处理）。在将其合并时重新正确地编写它。
