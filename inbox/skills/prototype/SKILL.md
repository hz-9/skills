# 原型

原型是**回答问题的临时代码。** 问题决定了形态。

## 选择一个分支

确定正在回答哪个问题——从用户的提示、周围的代码，或者如果用户在场的话询问他们：

- **"这个逻辑/状态模型感觉对吗？"** → [LOGIC.md](LOGIC.md)。构建一个微小的交互式终端应用，推动状态机通过那些难以在纸上推理的案例。
- **"这应该是什么样的？"** → [UI.md](UI.md)。在单个路由上生成几个截然不同的 UI 变体，通过 URL 搜索参数和浮动的底部栏进行切换。

这两个分支产生非常不同的产物——选错了会浪费整个原型。如果问题确实含糊不清且用户联系不上，默认选择与周围代码更匹配的分支（后端模块 → logic；页面或组件 → UI），并在原型顶部说明这个假设。

## 适用于两者的规则

1. **从第一天起就是一次性的，并明确标记为这样。** 将原型代码放在它将实际使用的位置附近（在它正在为其做原型的模块或页面旁边），以便上下文是清晰的——但命名它时让随意读者能看出它是原型，不是生产代码。对于一次性 UI 路由，遵循项目已有的路由约定；不要发明新的顶层结构。
2. **一条命令即可运行。** 无论项目现有的任务运行器支持什么——`pnpm <name>`、`python <path>`、`bun <path>` 等。用户必须能够无需思考就启动它。
3. **默认无持久化。** 状态保存在内存中。持久化是原型要_检查_的东西，而不是它应该依赖的东西。如果问题明确涉及数据库，则使用一个临时数据库或本地文件，并带有清晰的"PROTOTYPE — wipe me"名称。
4. **跳过打磨。** 没有测试，没有超出使原型_可运行_所需的错误处理，没有抽象。重点是快速学到东西然后删除它。
5. **展示状态。** 每次操作后（逻辑分支）或每次变体切换时（UI 分支），打印或渲染完整的相关状态，以便用户可以看到发生了什么变化。
6. **完成后删除或吸收。** 当原型回答了它的问题时，要么删除它，要么将验证过的决策合并到真实代码中——不要让它腐烂在仓库里。

## 完成时

原型中唯一值得保留的是_答案_。将答案连同它所回答的问题一起捕获到某个持久的载体中（提交消息、ADR、issue、或原型旁边的 `NOTES.md`）。如果用户在场，这个捕获是一个快速的对话；如果不在，留下占位符，以便他们（或你，在下次处理时）可以在删除原型之前填写结论。
