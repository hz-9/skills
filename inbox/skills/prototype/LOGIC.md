# 逻辑原型

一个微型交互式终端应用，让用户手动驱动状态模型。当问题是关于**业务逻辑、状态转换或数据形态**时使用——就是那种在纸面上看起来合理，但只有在实际用例中运行才会发现问题的情况。

## 何时适合这种形式

- "我不确定这个状态机是否能处理 X 然后 Y 的边界情况。"
- "这个数据模型真的能让我表示...的情况吗？"
- "我想在真正编写之前感受一下 API 应该是什么样的。"
- 任何用户想要**按下按钮并观察状态变化**的情况。

如果问题是"这应该长什么样"——走错了分支。请使用 [UI.md](UI.md)。

## 流程

### 1. 明确问题

在编写代码之前，写下你要原型化的状态模型和问题。一个段落，放在原型的 README 或文件顶部的注释中。回答了错误问题的逻辑原型完全是浪费——明确问题，以便之后可以检查，无论用户是在旁观看还是之后才回来看。

### 2. 选择语言

使用宿主项目所用的语言。如果项目没有明显的运行时（例如文档仓库），则询问。

匹配项目现有的工具约定——不要仅仅为了原型而添加新的包管理器或运行时。

### 3. 将逻辑隔离在可移植模块中

将实际逻辑——即回答问题的部分——放在一个小的、纯的接口后面，这个接口可以在以后被提取出来并放入真实的代码库中。围绕它的 TUI 是一次性的；逻辑模块则不应如此。

合适的形式取决于问题：

- **纯 reducer** — `(state, action) => state`。适用于当 action 是离散事件且状态是单个值时。
- **状态机** — 显式的状态和转换。适用于当"哪些 action 在现在是合法的"本身就是问题的一部分时。
- **一组纯函数**，操作一个普通数据类型。适用于没有隐式当前状态——只有转换时。
- **一个具有清晰方法面的类或模块**，当逻辑真正持有持续的内部状态时。

选择最适合被问及问题的形式，*而不是*最容易连接到 TUI 的形式。保持纯函数式：无 I/O、无终端代码、无用于控制流的 `console.log`。TUI 导入并调用它；没有任何东西流向相反方向。

这就是使原型在其生命周期之外仍然有用的原因。当问题得到解答后，经过验证的 reducer/状态机/函数集可以被提取到真正的模块中——TUI shell 则被删除。

### 4. 构建暴露状态的最小 TUI

将其构建为**轻量级 TUI**——在每个 tick 上，清屏（`console.clear()` / `print("\033[2J\033[H")` / 等效操作）并重新渲染整个帧。用户应该始终看到一个稳定的视图，而不是不断增长的滚动缓冲区。

每个帧包含两个部分，按此顺序：

1. **当前状态**，美观打印且方便对比（每行一个字段，或格式化的 JSON）。使用**加粗**表示字段名或章节标题，使用**暗淡**表示不太重要的上下文（时间戳、ID、派生值）。原生 ANSI 转义码即可——`\x1b[1m` 加粗、`\x1b[2m` 暗淡、`\x1b[0m` 重置。除非项目中已有样式库，否则无需引入。
2. **键盘快捷键**，列在底部：`[a] 添加用户  [d] 删除用户  [t] 时钟滴答  [q] 退出`。键名加粗、描述暗淡，或反之——只要看起来清晰即可。

行为：

1. **初始化状态**——单个内存中的对象/结构体。启动时渲染第一帧。
2. **每次读取一个按键（或一行）**，调度给一个处理函数来修改状态。
3. **重新渲染**每次操作后的完整帧——不要追加，要替换。
4. **循环直到退出。**

整个帧应能容纳在一个屏幕内。

### 5. 使其可通过一个命令运行

在项目现有的任务运行器（`package.json` 的 scripts、`Makefile`、`justfile`、`pyproject.toml`）中添加一个脚本。用户应该运行 `pnpm run <prototype-name>` 或等效命令——永远不需要记住路径。

如果宿主项目没有任务运行器，只需将命令放在原型 README 的顶部。

### 6. 交给用户

给用户运行命令。他们自己会驱动；有趣的时刻是他们说"等等，那应该是不可能的"或"哈，我原以为 X 会不同"——这些是*想法*中的缺陷，而这正是全部意义所在。如果他们想要添加新的操作，就添加。原型是会演变的。

### 7. 记录答案

当原型完成其任务后，问题的答案是唯一值得保留的东西。如果用户在旁，问问他们学到了什么。如果不在，在原型旁边留下一个 `NOTES.md`，以便稍后填写答案（或者如果你观察了会话，由你来填写），然后在删除原型之前完成。

## 反模式

- **不要添加测试。**需要测试的原型就不再是原型了。
- **不要连接到真实数据库。**使用内存存储，除非问题专门关于持久化。
- **不要泛化。**没有"如果我们以后想支持 X 呢"。原型回答一个问题。
- **不要模糊逻辑和 TUI 的界限。**如果 reducer/状态机引用了 `console.log`、提示或终端转义码，它就不再是可移植的了。保持 TUI 作为纯模块之上的薄壳。
- **不要将 TUI shell 发布到生产环境。**Shell 是为手动从终端驱动而优化的。其背后的逻辑模块才是值得保留的部分。
