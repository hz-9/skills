## 领域感知

在浏览代码库时，也要查找现有的文档：

### 文件结构

大多数仓库只有一个 context（上下文）：

```
/
├── CONTEXT.md
├── docs/
│   └── adr/
│       ├── 0001-event-sourced-orders.md
│       └── 0002-postgres-for-write-model.md
└── src/
```

如果根目录存在 `CONTEXT-MAP.md`，则仓库有多个 contexts。该映射指向每个 context 所在位置：

```
/
├── CONTEXT-MAP.md
├── docs/
│   └── adr/                          ← 系统级决策
├── src/
│   ├── ordering/
│   │   ├── CONTEXT.md
│   │   └── docs/adr/                 ← 特定 context 的决策
│   └── billing/
│       ├── CONTEXT.md
│       └── docs/adr/
```

惰性创建文件——只有当你需要写入内容时才创建。如果 `CONTEXT.md` 不存在，则在第一个术语被确定时创建。如果 `docs/adr/` 不存在，则在需要第一个 ADR 时创建。

## 在会话期间

### 对照术语表进行挑战

当用户使用的术语与 `CONTEXT.md` 中已有的语言冲突时，立即指出来。"你的术语表将 'cancellation' 定义为 X，但你似乎指的是 Y——到底是哪个？"

### 澄清模糊语言

当用户使用模糊或过载的术语时，提出一个精确的规范术语。"你说 'account'——你指的是 Customer 还是 User？它们是不同的东西。"

### 讨论具体场景

当讨论领域关系时，用具体的场景来压力测试它们。设计能够探测边缘情况并迫使用户精确说明概念之间边界的场景。

### 与代码交叉引用

当用户陈述某事物如何运作时，检查代码是否同意。如果你发现矛盾，将其摆到台面上："你的代码取消了整个 Orders，但你刚才说部分取消是可能的——哪个是正确的？"

### 即时更新 CONTEXT.md

当一个术语被确定后，立即更新 `CONTEXT.md`。不要批量处理——在它们发生时立即捕获。使用 [CONTEXT-FORMAT.md](./CONTEXT-FORMAT.md) 中的格式。

不要将 `CONTEXT.md` 与实现细节耦合。只包含对领域专家有意义的术语。

### 谨慎提供 ADR

仅在以下三个条件都满足时才提供创建 ADR：

1. **难以逆转**——以后改变主意的成本很高
2. **无上下文的会令人惊讶**——未来的读者会想知道"他们为什么这样做？"
3. **是真正权衡的结果**——存在真正的替代方案，你出于特定原因选择了一个

如果缺少任意一个条件，跳过 ADR。使用 [ADR-FORMAT.md](./ADR-FORMAT.md) 中的格式。
