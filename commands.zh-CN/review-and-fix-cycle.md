# Review and Fix Cycle

## 定义

### 严重度分级

| 级别   | 定义                                           | 示例                                        |
| ------ | ---------------------------------------------- | ------------------------------------------- |
| **高** | 结构性问题或语义错误，会导致功能失效或安全隐患 | 逻辑错误、安全漏洞、API 误用、死链接        |
| **中** | 可维护性问题或非关键性遗漏                     | 命名不规范、缺少边界检查、注释与代码不一致  |
| **低** | 优化建议，不影响正确性                         | 代码风格微调、冗余注释、推荐的但非强制的最佳实践 |

### 收敛条件

**收敛**：本轮审查发现 **0 个新问题**，满足循环退出条件。

> **新问题定义**：相对本轮操作的上一次等价判定结果而言，本次新发现且之前未被记录的问题。同一问题被修复后再次出现的属于复现，计入新问题。

### 迭代上限

| 层级                   | 上限 |
| ---------------------- | ---- |
| 修复循环（内层）       | 10 次 |
| Code Review 外圈循环   | 30 次 |

## Prerequisites

- 当前工作空间是一个 Git 仓库（用于识别变更范围）
- CodeReview Agent 子代理可用
- `docs/review-and-fix-cycle/` 目录存在于当前工作空间（不存在则创建）；UTC 时间子目录由 Step 0 自动创建

## Workflow

### Step 0: 前置检查

0.1 检查当前工作空间是否为 Git 仓库：
  - 是 -> 下一步；
  - 否 -> 报告"非 Git 仓库，无法识别变更范围"，终止流程；

0.2 检查 CodeReview Agent 子代理是否可用（通过 `Agent` 工具尝试调用）：
  - 是 -> 下一步；
  - 否 -> 报告"CodeReview Agent 不可用"，终止流程；

0.3 获取当前 UTC 时间并创建 `docs/review-and-fix-cycle/{UTC时间}/` 目录（父目录不存在则一并创建），本轮运行所有报告均保存至此目录：
  - 创建成功 -> 下一步；
  - 创建失败 -> 报告"目录创建失败"，终止流程；

0.4 确定审查范围：
  - 扫描当前暂存区和工作区的变更（`git diff --cached` 和 `git diff`）：
    - 仅存在暂存变更 -> 审查范围为暂存变更；
    - 仅存在工作区变更 -> 审查范围为工作区变更；
    - 同时存在暂存和工作区变更 -> 通过 AskUserQuestion 提供选项，阻塞等待用户选择：
      - 暂存变更（推荐） -> 审查范围为暂存变更；
      - 工作区变更 -> 审查范围为工作区变更；
      - 全部变更 -> 审查范围包含暂存区和工作区变更；
    - 无任何变更 -> 通过 AskUserQuestion 提供选项，阻塞等待用户选择：
      - 审查指定文件/目录 -> 由用户指定路径，执行后进入下一步；
      - 审查整个仓库 -> 审查范围为整个仓库；
      - 审查最近提交 -> 审查范围为最近一次提交的变更；
      - 终止 -> 终止流程；

### Step 1: Code Review 审查

并行派出 **恰好 3 个** CodeReview Agent（Completeness/完整性、Correctness/正确性、Impact/影响），各自以全新的视角独立审查变更范围，**禁止参考本轮之前的审查结果或历史报告（回归审查场景除外，见 Step 5），不得以本步骤已被执行过或问题数已减少为由降级**，各自记录发现的问题：

- **3 个 Agent 均报告 0 个新问题** -> **收敛**，进入 Step 6；
- **任一 Agent 发现问题** -> 汇总去重后进入 Step 2；
- **任一 Agent 因环境故障不可用时** -> 视为环境故障，流程终止并在报告中标注"CodeReview Agent 不可用"；

### Step 2: 记录问题

将每个问题记录到审查日志中，标注严重度（高/中/低），生成审查日志文件 `review-cycle-{外圈迭代}.md` 保存到 `docs/review-and-fix-cycle/{UTC时间}/`，格式如下：

```markdown
# Review Cycle - 第 {N} 轮审查

## 发现的问题

| # | 严重度 | 来源 | 文件 | 问题描述 |
|---|--------|------|------|----------|
| 1 | 高 | Completeness | src/example.ts:42 | ... |
| 2 | 中 | Correctness | src/example.ts:88 | ... |

## 问题详情

### [高] 问题 1：{问题标题}

- **来源**：Completeness Agent
- **文件**：`src/example.ts:42`
- **描述**：{详细描述}

### [中] 问题 2：{问题标题}
...
```

### Step 3: 修复问题

逐个修复所有发现的问题：

- 每次修复后生成修复报告 `review-cycle-{外圈迭代}-fix{修复轮次}.md`，记录修复方式和结果，保存到 `docs/review-and-fix-cycle/{UTC时间}/`；
- 全部问题修复完成后，修复轮次自增 1，进入 Step 4；

### Step 4: 修复轮次上限判断

检查是否达到 10 次修复上限：
- 是 -> 标记"修复已达上限，部分问题未解决"，中止修复流程，进入 Step 6；
- 否 -> 进入 Step 5；

### Step 5: 回归审查

并行派出 **恰好 3 个** CodeReview Agent（Completeness/完整性、Correctness/正确性、Impact/影响），回归审查已修复的问题（回归审查 Agent 不受 Step 1 禁止参考历史约束，可携带本轮修复摘要上下文），验证修复是否正确以及是否引入了新问题，各自记录发现：

- **3 个 Agent 均报告 0 个新问题** -> **修复循环收敛**，生成迭代报告 `review-cycle-{外圈迭代}.md`，保存到 `docs/review-and-fix-cycle/{UTC时间}/`，进入 Step 6；
- **任一 Agent 发现问题** -> 将新问题汇总去重后回到 Step 2（修复轮次+1）；

### Step 6: 外圈循环控制

检查本轮外圈审查的问题数量：

- **3 个 Agent 均报告 0 个新问题** -> **收敛**，进入 Step 7；
- **存在问题** -> 判断外圈循环是否达到 30 次上限：
  - 是 -> 标记"外圈循环-未完全收敛"，强制退出，进入 Step 7；
  - 否 -> 返回 Step 1（外圈迭代计数+1）；

### Step 7: 总结输出

汇总本轮所有报告，输出审查与修复总结，内容包括：

- 审查总览（审查范围、审查轮次、总修复问题数等）
- 按严重度分类的问题统计
- 所有生成的报告文件列表

生成最终总结报告 `final-summary.md` 保存到 `docs/review-and-fix-cycle/{UTC时间}/`。

## Rules

- **行为规范**
  - **自动持续运行**：各步骤之间必须自动推进，不得中断等待用户确认后手动触发（AskUserQuestion 场景除外）。
  - **Agent 类型锁定规则**：整个流程中，Code Review 阶段（含 Step 1 首次审查和 Step 5 回归审查）**必须**使用 CodeReview Agent 执行审查。严禁以任何理由切换为 GeneralPurpose Agent 或其他 Agent 类型。若环境客观上不支持 CodeReview Agent，流程终止并在报告中标注"CodeReview Agent 不可用"，不得降级替代后继续执行。
  - **并行数硬约束规则**：每轮 Code Review（含 Step 1 首次审查和 Step 5 回归审查）必须并行派出 **恰好 3 个** CodeReview Agent，分别独立覆盖 Completeness、Correctness、Impact 三个视角。3 个 Agent 的审查结果均需独立记录到审查日志中。
  - **禁止流程降级规则**：各步骤必须严格按照 Workflow 定义的顺序、内容和执行方式执行，不允许以任何理由跳过、简化、合并或加速任何步骤。
  - **修复方式选择**：修复问题时，根据问题类型自行决定修改方式（直接修改代码或使用相关工具），无需逐一询问用户。
  - **不影响当前工作区**：执行过程中不得执行 `git commit`、`git add` 等会影响 Git 历史的操作，除非用户明确要求。
- **报告规范**
  - **报告持久化**：所有审查日志、修复报告和总结报告均保存到 `docs/review-and-fix-cycle/{UTC时间}/` 目录。
  - **命名规则**：
    - 审查日志：`review-cycle-{外圈迭代}.md`
    - 修复报告：`review-cycle-{外圈迭代}-fix{修复轮次}.md`
    - 最终总结：`final-summary.md`
  - **禁止引入流程外标签**：流程执行和报告中**严禁**出现 Workflow 未定义的流程标签或状态标识。
- **交互规范**
  - 涉及变更范围选择等需要用户决策的场景，**必须**使用 `AskUserQuestion` 工具，不得以纯文本追问形式向用户咨询问题。

## Examples

### 对话交互示例

```markdown
用户 > 执行代码审查和修复

AI   > 开始 Review and Fix Cycle：
       Step 0 前置检查…Git 仓库确认完毕，UTC 时间目录已创建。

       Step 0.4 确定审查范围…检测到暂存区和工作区均有变更。
       通过 AskUserQuestion 提供选项：
          - 暂存变更（推荐） -> 审查范围为暂存变更；
          - 工作区变更 -> 审查范围为工作区变更；
          - 全部变更 -> 审查范围为全部变更；
用户 > 暂存变更

AI   > Step 1 Code Review 审查…派出 3 个 CodeReview Agent（Completeness、Correctness、Impact）。
       Completeness 发现 2 个问题，Correctness 发现 1 个问题，Impact 发现 0 个问题。
       汇总去重后共 2 个问题。

       Step 2 记录问题…已记录到审查日志 review-cycle-1.md。

       Step 3 修复问题…修复问题 1（高-逻辑错误）、问题 2（中-命名不规范）。
       生成修复报告 review-cycle-1-fix1.md。

       Step 5 回归审查…派出 3 个 CodeReview Agent 回归审查。
       3 个 Agent 均报告 0 个新问题 -> 修复循环收敛。

       Step 6 外圈循环控制…检查是否收敛：
       当前外圈发现 2 个问题 -> 未收敛，外圈迭代+1，返回 Step 1。

       Step 1（第 2 轮）Code Review 审查…派出 3 个 CodeReview Agent。
       3 个 Agent 均报告 0 个新问题 -> 收敛。

       Step 7 总结输出…审查与修复完成！共发现并修复 2 个问题（高 1 / 中 1 / 低 0）。
```
