# [Rush.js](#rushjs) -> [Nx](#nx) + [Changesets](#changesets) Migration

## Overview

Automated migration from [Rush.js](#rushjs) monorepo to [Nx](#nx) + pnpm workspace + [Changesets](#changesets): analyze existing structure -> initialize [Nx](#nx) configuration -> copy source code and restructure directories -> create project.json -> migrate Git hooks -> update CI/CD -> verify. Suitable for scenarios with a moderate number of packages (<20) where the team wants to move away from the [Rush.js](#rushjs) ecosystem toward the standard pnpm+[Nx](#nx) toolchain.

## Definitions

- <a id="rushjs"></a>**Rush.js**: A monorepo management tool by Microsoft, using `rush.json` to define project structure, with built-in pnpm and version publishing strategies.
- <a id="nx"></a>**Nx**: A general-purpose monorepo build system providing task orchestration, caching, and dependency graph analysis capabilities.
- <a id="changesets"></a>**Changesets**: A semantic version management tool that records change intent through changeset files, automatically bumping versions, generating changelogs, and publishing to npm during execution.

## Prerequisites

- A local [Rush.js](#rushjs) monorepo repository (containing `rush.json`).
- Node.js >= 18.15.0 (<19) or >=20.9.0 (<21), pnpm >= 8.15.9.
- Write permissions for the target directory, Git installed and configured.
- To create an [Nx](#nx) + [Changesets](#changesets) project from scratch (not a migration), please read the detailed steps first and adjust manually.

## Workflow

0. **Pre-flight Check** — Ensure migration conditions are met;
  - Check if `rush.json` exists:
    - Yes -> next step;
    - No -> report "rush.json not found, not a [Rush.js](#rushjs) project", terminate flow;
  - Check if Node.js version meets >=18.15.0 (<19) or >=20.9.0 (<21):
    - Yes -> next step;
    - No -> prompt to update Node.js to a supported version (>=18.15.0 <19 or >=20.9.0 <21), terminate flow;
  - Check if pnpm version >= 8.15.9:
    - Yes -> next step;
    - No -> prompt to update pnpm, terminate flow;
  - Check if Git is installed and configured:
    - Yes -> next step;
    - No -> prompt to install/configure Git, terminate flow;
  - Check if `pnpm-workspace.yaml` or `nx.json` already exists:
    - Yes -> provide options via AskUserQuestion, block and wait for user selection:
      - Existing Nx workspace config detected, continue override -> continue execution, proceed to next step;
      - Terminate -> terminate flow;
    - No -> next step;
  - Check if target directory has write permissions:
    - Yes -> next step;
    - No -> report no permissions, terminate flow;

1. **Analysis** — Read `rush.json`, understand project structure;
  - Count number of projects, `workspace:*` dependency relationships, custom [Rush.js](#rushjs) commands;
  - Check if project count is 0:
    - Yes -> report "No projects found in `rush.json`, cannot migrate", terminate flow;
    - No -> next step;
  - Check if package count >= 20:
    - Yes -> provide options via AskUserQuestion, block and wait for user selection:
      - Known risks, continue -> continue execution, proceed to next step;
      - Do not continue -> terminate flow;
    - No -> next step;
  - Provide options via AskUserQuestion, block and wait for user selection:
    - [Dynamic options generated by AI based on [analysis results]] -> confirm plan, proceed to next step;

2. **Initialization** — Create temp directory, create [Nx](#nx) workspace base config in new repo, see [migration steps](references/migration-steps.md#Step-2-Initialize-Nx-Workspace);

3. **Copy Source Code** — rsync excluding [Rush.js](#rushjs) cache, restructure directories, see [migration steps](references/migration-steps.md#Step-3-Copy-Source-Packages);

4. **Create project.json** — Create [Nx](#nx) project.json for each package, see [migration steps](references/migration-steps.md#Step-4-Create-projectjson-for-each-package);

5. **Migrate Git Hooks** — Create Husky configuration, see [migration steps](references/migration-steps.md#Step-5-Migrate-Git-Hooks);

6. **Update Configuration** — Update `.gitignore`, `.prettierrc.js`, etc., see [migration steps](references/migration-steps.md#Step-6-Update-Configuration-Files);

7. **Update CI/CD** — Migrate CI/CD configuration, see [migration steps](references/migration-steps.md#Step-7-Update-CICD);

8. **Verification** — Run build verification, see [migration steps](references/migration-steps.md#Step-8-Install-Dependencies-and-Verify);
  - Check if build passes:
    - Yes -> next step;
    - No -> record failure count, check if max retries reached (3 times):
      - Yes -> terminate flow, report unfixable build errors;
      - No -> report error reason, return to specific step (corresponding step in steps 2~7) to fix, after fix return to current step to re-verify;

9. **Review Check** — Check against [Review List](#review-list) to confirm migration results;
  - Check if Review List has content:
    - No -> go directly to next step (output results);
    - Yes -> next step;
  - Check each item in [Review List](#review-list) in order, check if passed:
    - Display output content based on "Review Check Example";
  - If any check fails, check if max retries reached (3 times):
    - Yes -> terminate flow, report failed items;
    - No -> return to specific step (corresponding step in steps 2~7) to fix, after fix return to current step to re-check;
  - All passed -> next step;

10. **Output Results** — Output migration summary, notify completion;
  - Output structured summary (package count, new files, verification results, etc.);
  - Notify migration complete;

## Rules

- **Operational Conventions**
  - For all configuration file creation and modification operations, first present the plan to the user, proceed only after confirmation;
  - Preserve the user's existing non-[Rush.js](#rushjs) configuration files (`.markdownlint.json`, `.nvmrc`, `.editorconfig`, etc.), do not delete;
  - For every file move operation, synchronously update related path references;
- **Safety Conventions**
  - Before migration, first build the new repository structure in a temp directory, apply to the target directory only after verification passes;
- **Interaction Conventions**
  - All user decision-making interactions **must** use the `AskUserQuestion` tool; plain text follow-ups are prohibited; pass questions and options into AskUserQuestion in a structured way, no more than 4 questions per call;
- **Self-Consistency Conventions**
  - [Review List](#review-list) check items must cover all binding rules in Rules, ensuring one-to-one correspondence between Rules and [Review List](#review-list);

## Review List

- **Migration Completeness Check**
    - [ ] All packages restructured from [Rush.js](#rushjs) structure to `packages/` flat layout

# Rush.js → Nx + Changesets Migration Skill

## Quick Start

```bash
# 1. Analyze existing Rush project structure
cat rush.json | jq '.projects[] | {packageName, projectFolder}'

# 2. Create new repository and initialize base configuration
mkdir my-repo-nx && cd my-repo-nx
pnpm init
pnpm add -D -w nx @changesets/cli @changesets/changelog-git husky lint-staged prettier

# 3. Copy source code, reorganize directory structure
rsync -av --exclude='node_modules' --exclude='common' --exclude='.git' /path/to/rush-repo/ .
mkdir -p packages
# Reorganize Rush's category/package structure into packages/

# 4. Install dependencies and verify
pnpm install
pnpm nx run-many --target=lint --all
```

## Workflow

### Migration Process (8 Steps)

1. **Analyze** — Identify Rush project count, dependency relationships, and custom commands
2. **Initialize** — Create `pnpm-workspace.yaml`, `nx.json`, `.changeset/config.json`, root `package.json`
3. **Copy Source** — rsync to new repository, reorganize directory into `packages/`
4. **Create project.json** — Create Nx project.json for each package, specify build and lint commands
5. **Migrate Git Hooks** — Create `.husky/` (commit-msg + pre-commit) and `commitlint.config.js`
6. **Update Configuration** — `.gitignore`, `.prettierrc.js`, `.npmrc`, `.vscode/settings.json`
7. **Update CI/CD** — `actions-rush` → `pnpm/action-setup`
8. **Verify** — `pnpm install && pnpm nx run-many --target=build --all`

### Release Process

```bash
# Record changes (interactive)
pnpm changeset

# Execute release (version bump → commit → tag → publish → push)
bash scripts/release.sh
```
