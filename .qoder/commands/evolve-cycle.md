## 定义

### 收敛条件

| 层级 | 收敛条件 | 判定位置 |
|------|----------|----------|
| **小循环A 收敛** | 当前 skill-optimizer 优化发现 **0 个新问题** | 目标 SKILL 已对齐标准结构，无待优化项 |
| **小循环B 收敛** | 当前 Code Review 发现 **0 个新问题** | 分两种情况：① 首次从全新角度审查无问题 → 目标 SKILL 已稳定；② 修复后返回全新审查无问题 → 本轮修复正确、未引入新问题 |
| **大循环收敛** | ① 本轮小循环A 已完全收敛（非跳过） **且** ② 大循环步骤 2 中小循环B 首次 Code Review 发现 **0 个新问题** | 两条均满足时说明目标 SKILL 已稳定，无待修复问题 |

> **新问题定义**：相对本轮操作的上一次等价判定结果而言，本次新发现且之前未被记录的问题。同一问题被修复后再次出现的属于复现，计入新问题。

### 严重度分级

| 级别 | 定义 | 示例 |
|------|------|------|
| **高** | 结构性问题或语义错误，会导致用户误解或功能失效 | 缺少关键 section、描述格式错误、死链接 |
| **中** | 可维护性问题或非关键性遗漏 | 标点不规范、术语不一致、超过 300 行但未拆分 |
| **低** | 优化建议，不影响正确性 | 冗余内容、推荐的但非强制的最佳实践 |

### 小循环A（优化）

每轮小循环A 由以下步骤组成，循环运行直至收敛：

1. **skill-optimizer 优化** — 使用 skill-optimizer 优化目标 SKILL
   - **发现 0 个问题** → **小循环A 收敛**，进入大循环步骤 2（小循环B）
   - **发现问题** → 继续步骤 2
2. **记录问题** — 将每个问题记录到本轮报告中，标注严重度（高/中/低）
3. **修复问题** — 逐个修复所有发现的问题
4. **返回步骤 1**，重新运行优化

> 每轮小循环A 最大 30 次迭代。达到上限仍未收敛时，强制退出小循环A，
> 进入大循环步骤 2，在报告中标注「小循环A-未完全收敛」。

### 小循环B（审查）

每轮小循环B 由以下步骤组成，循环运行直至收敛。收敛结果根据"是否执行过修复"分化去向。

> **Code Review 执行方式**：优先使用专用 CodeReview Agent（从 completeness / correctness / impact
> 三个视角并行审查）；若当前环境不可用，则由 AI 自行全面审查。
> 步骤 4 的再审查同样遵循此规则。

> **状态标记**：「本轮是否执行过修复」的作用域为当前小循环B 的整个生命周期。
> step 6 返回 step 1 时不重置该标记——"全新角度"指审查视角切换，而非状态重置。
> 两轮大循环之间该标记自动复位（因小循环B 每次由大循环重新发起）。

1. **Code Review** — 全面审查当前文件状态，记录所有发现的问题
   - **发现 0 个问题**:
     - 若本轮未执行过修复（首次审查即无问题） → **小循环B 收敛**，等待大循环步骤 2 判定去向
     - 若本轮已执行过修复（修复后审查无问题） → **小循环B 收敛**，等待大循环步骤 2 判定去向
   - **发现问题** → 继续步骤 2
2. **记录问题** — 将每个问题记录到本轮报告中，标注严重度（高/中/低）
3. **修复问题** — 逐个修复所有发现的问题
4. **Code Review（再审查）** — 重新审查被修改的文件，仅聚焦于修复准确性和副作用，检查：
   - 修复是否正确
   - 修复是否引入了新问题
   - 修改后的文件是否自指一致（自身通过自己定义的规则）
5. 若发现**任何问题**（修复不正确、引入新问题、自指不一致），回到**步骤 2**，将新问题记录到本轮报告后再进入步骤 3 继续修复
6. 未发现新问题，**返回步骤 1**，从全新角度重新审查

> 每轮小循环B 最大 30 次外层循环迭代（步骤 1→...→返回步骤 1 计为一次）。
> 内层修复循环（步骤 2→3→4→5→2）单独计数，最多 30 次，超限时强制退出小循环B，继续大循环步骤 3（报告标注「小循环B-修复卡死」）。
> 达到上限仍未收敛时，强制退出小循环B，继续大循环步骤 3（报告标注「小循环B-未完全收敛」）。

### 大循环

每轮大循环按以下顺序执行：

1. **小循环A** — 按上述定义运行直至收敛
   - 若目标就是 skill-optimizer 本身，小循环A 跳过优化，视为已完全收敛，直接进入步骤 2
     ⚠️ 跳过后大循环收敛条件（非跳过）不满足，大循环只能通过 30 次上限终止

2. **小循环B** — 按上述定义运行直至收敛
   - 小循环B 收敛后检查：
     - 本轮小循环A 已完全收敛（非跳过） **且** 小循环B 首次审查发现 0 个新问题 → **大循环收敛**，终止全部流程
     - 否则 → 继续步骤 3

3. **合并与整理** — 汇总本轮小循环A 和 小循环B 发现的所有问题，按类型统计

4. **反哺到 skill-optimizer** — ⚠️ 本步骤会修改 skill-optimizer SKILL.md 自身，后续所有轮次的小循环A 都会受此影响。将本轮 Code Review 学到的经验整合到 skill-optimizer 自身：
   - 更新 Rules（补充缺失的约束规则）
   - 更新 Workflow（优化执行步骤）
   - 更新 Review List 检查项（补全遗漏的验证维度）

5. **skill-optimizer 自优化** — 使用 skill-optimizer 自身的流程对其自身的 SKILL.md 执行一次完整 Workflow（不进入小循环A 的迭代循环），确保自指一致性。自优化中所有需交互确认的步骤遵循「交互模式覆盖」原则。

6. **返回步骤 1**，进入下一轮大循环

> 每轮大循环最大 30 次外层迭代（步骤 1→6 计为一次）。达到上限仍未触发大循环收敛时，终止全部流程，
> 汇总所有轮次报告，标记最终状态为「大循环-未完全收敛」（目标为 skill-optimizer 自身时此状态为预期行为，非异常）。


## 运行原则

- **自动持续运行**：大循环之间不允许等待用户确认，必须自动连续执行直至收敛或达到上限
- **交互模式覆盖**：调用 skill-optimizer 时，原本需要用户交互确认的步骤由 AI 自行决策（优先采用采纳/自动迁移策略），不得等待用户确认
- **小循环收敛即推进**：小循环A 收敛后进入大循环步骤 2；小循环B 收敛后由大循环步骤 2 判定去向（大循环收敛或进入步骤 3）。均不允许中断
- **目录准备**：开始前确保 `docs/` 目录存在（不存在则创建）
- **报告持久化**：每个小循环、每轮大循环的合并报告，均保存到 `docs/` 目录。
  命名规则：小循环A 报告 `cycle{轮次}-A-{迭代}.md`，小循环B 报告 `cycle{轮次}-B-{迭代}.md`，
  大循环合并报告 `cycle{轮次}-summary.md`，最终汇总报告 `final-summary.md`
