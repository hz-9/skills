# NestJS 关系型数据库集成（TypeORM）

## Overview

当 AI 在 NestJS 项目中遇到 TypeORM 相关代码时，自动执行以下工作：审查实体定义的正确性和关系映射的完整性，检查 TypeORM 模块配置和仓库模式使用，评估事务管理的可靠性，识别常见数据库性能问题，并提供改进建议。

## Definitions

- <a id="目标代码"></a>**目标代码**：当前对话中 NestJS 的 TypeORM 实体类、模块配置或数据库操作代码。
- <a id="实体关系"></a>**实体关系**：TypeORM 中实体间的 @OneToOne、@OneToMany、@ManyToOne、@ManyToMany 关联装饰器。
- <a id="仓库模式"></a>**仓库模式**：通过 @InjectRepository 注入 EntityRepository 或 Repository 实例的方式。
- <a id="是否分析完成"></a>**是否分析完成**：标记对目标代码的分析是否已得出完整结果。

## Prerequisites

- NestJS 项目环境（包含 @nestjs/typeorm 和 typeorm 依赖）；
- 实体、模块或数据库操作代码文件可访问；
- 了解关系型数据库和 ORM 基本概念。

## Workflow

0. **前置检查** — 确保目标代码和运行环境可达；
   - 判断目标代码是否存在且可读取：
     - 是 -> 下一步；
     - 否 -> 提示用户提供目标代码或文件路径，阻塞等待用户输入；
   - 初始化全局变量 [是否分析完成](#是否分析完成)：
     - 判断代码是否完整可解析：
       - 满足 -> 初始化变量为 true；
       - 不满足 -> 初始化变量为 false；

1. **分析 TypeORM 代码** — 读取并理解实体定义和数据库操作；
   - 读取目标代码，识别以下核心要素：
     - 实体的 @Entity 定义和列装饰器（@Column、@PrimaryGeneratedColumn 等）；
     - 关系映射的装饰器配置（@OneToMany、@ManyToOne 等）；
     - TypeORM 模块配置（TypeOrmModule.forRoot / forFeature）；
     - 服务中数据库操作方法（find、save、query builder 等）；

2. **逐项审查** — 对照审查清单检查 TypeORM 代码质量；
   - 依次判断以下审查项是否通过：
     - 实体是否使用 @Entity 装饰器且指定了正确的表名：
       - 通过 -> 记录通过，继续下一个审查项；
       - 未通过 -> 记录“实体缺少 @Entity 装饰器或表名不明确”，继续；
     - 关系映射是否在两端都正确配置（双向关系需在双方都添加装饰器）：
       - 通过 -> 记录通过，继续下一个审查项；
       - 未通过 -> 记录“单向关系可能导致查询效率低下，建议配置双向关系”，继续；
     - 查询是否使用 select 限制返回字段（避免 SELECT * 全表查询）：
       - 通过 -> 记录通过，继续下一个审查项；
       - 未通过 -> 记录“建议使用 select 指定返回字段以减少数据传输”，继续；
     - 事务处理是否使用 @Transaction 或 QueryRunner 确保原子性：
       - 通过 -> 记录通过，继续下一个审查项；
       - 未通过 -> 记录“多步数据库操作未使用事务，可能导致数据不一致”，继续；
     - TypeOrmModule.forRoot() 配置是否正确（数据库类型、主机、凭证等）：
       - 通过 -> 记录通过，继续下一个审查项；
       - 未通过 -> 记录“数据库配置不完整或使用了不推荐的方式”，继续；
     - 迁移配置是否正确（migrationsRun、migrations 表名等）：
       - 通过 -> 记录通过，继续下一个审查项；
       - 未通过 -> 记录“迁移配置不完善，可能导致生产环境表结构不一致”，继续；
   - 判断是否有任何问题记录：
     - 是 -> 汇总问题列表，进入下一步；
     - 否 -> 直接进入步骤 4（复核检查）；

3. **提供修改建议** — 对发现的问题给出具体修复方案；
   - 依次对每个问题提供修复建议；
   - 通过 AskUserQuestion 提供选项，阻塞等待用户选择：
     - 采纳全部建议 -> 生成修正后的代码，进入步骤 4；
     - 逐条确认 -> 逐条由用户决定采纳或忽略，全部确认后进入步骤 4；
     - 仅查看不修改 -> 不影响代码，直接进入步骤 4；

4. **复核检查** — 对照 [Review List](#review-list)，确认执行结果；
   - 判断 Review List 是否有内容：
     - 否 -> 直接进入下一步（成果输出）；
     - 是 -> 下一步；
   - 依次判断 [Review List](#review-list) 中每个检查项，是否通过：
     - 基于“复核检查示例”来显示输出内容；
   - 有任一检查失败，终止流程；
   - 全部通过后，进入下一步；

5. **成果输出** — 输出审查摘要，告知完成；
   - 输出结构化摘要表格（审查文件 / 审查项数 / 通过 / 发现问题 / 风险等级 / 已采纳建议）；
   - 告知审查完成；

## Rules

- **内容规范**
  - description 须遵循格式：第一句说明审查内容，第二句说明触发条件，使用第三人称，不超过 1024 字符；
  - 审查项须引用 TypeORM 和 NestJS 官方文档；
  - SKILL.md 不超过 300 行；超过时拆分到 references/ 目录；
  - 引用层次不超过一层；
  - 保持术语一致，Definitions 中已定义术语在正文中通过锚点链接引用；
  - 删除任何时效性信息；

- **行为规范**
  - 审查时仅输出问题摘要和定位，不直接修改代码，除非用户通过 AskUserQuestion 明确要求；
  - 所有涉及代码修改的交互环节，必须使用 AskUserQuestion 工具；
  - 关系设计和查询优化建议标注为架构建议而非代码缺陷；

- **防御规范**
  - 如目标代码为空或不可读，直接报告并终止；
  - 事务必要性仅在多步操作明确时报告，单步操作不要求事务；

- **验证规范**
  - Examples 内容必须自洽于 Rules；
  - Examples 必须包含复核检查示例；
  - 对话交互示例仅聚焦步骤 0~3（不含复核检查和成果输出）；
