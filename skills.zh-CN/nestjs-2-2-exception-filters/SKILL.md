# NestJS 全局与局部异常处理

## Overview

当 AI 在 NestJS 项目中遇到异常过滤器相关代码时，自动执行以下工作：审查全局和局部异常过滤器的注册方式，检查自定义异常类的定义规范性，评估异常响应格式的统一性，诊断未捕获异常的潜在来源，并提供改进建议。

## Definitions

- <a id="目标代码"></a>**目标代码**：当前对话中 NestJS 的异常过滤器实现代码或异常类定义代码。
- <a id="过滤器层级"></a>**过滤器层级**：异常过滤器的三种作用范围 —— 全局过滤器（app.useGlobalFilters）、控制器级过滤器（@UseFilters）、方法级过滤器（@UseFilters）。
- <a id="HttpException"></a>**HttpException**：NestJS 内置的基础异常类，包含 HTTP 状态码和错误消息，所有自定义异常应继承此类。
- <a id="是否分析完成"></a>**是否分析完成**：标记对目标代码的分析是否已得出完整结果。

## Prerequisites

- NestJS 项目环境；
- 异常过滤器或异常类的代码文件可访问；
- 理解异常过滤器在 NestJS 请求处理链中的位置。

## Workflow

0. **前置检查** — 确保目标代码和运行环境可达；
   - 判断目标代码是否存在且可读取：
     - 是 -> 下一步；
     - 否 -> 提示用户提供目标代码或文件路径，阻塞等待用户输入；
   - 初始化全局变量 [是否分析完成](#是否分析完成)：
     - 判断代码是否完整可解析：
       - 满足 -> 初始化变量为 true；
       - 不满足 -> 初始化变量为 false；

1. **分析异常过滤器代码** — 读取并理解过滤器实现和异常体系；
   - 读取目标代码，识别以下核心要素：
     - 过滤器实现方式（实现 ExceptionFilter 接口或继承 BaseExceptionFilter）；
     - 过滤器的注册层级（全局 / 控制器 / 方法）；
     - 自定义异常类的定义（是否继承 HttpException）；
     - 响应格式的结构（状态码、消息、时间戳等）；

2. **逐项审查** — 对照审查清单检查异常处理代码质量；
   - 依次判断以下审查项是否通过：
     - 自定义异常类是否继承 HttpException（保持与 NestJS 异常体系一致）：
       - 通过 -> 记录通过，继续下一个审查项；
       - 未通过 -> 记录“自定义异常应继承 HttpException，以利用内置异常处理机制”，继续；
     - 全局过滤器是否注入了依赖（全局过滤器在模块外注册时无法通过 DI 注入）：
       - 通过 -> 记录通过，继续下一个审查项；
       - 未通过 -> 记录“全局过滤器如需 DI 注入，应通过模块内的提供者方式注册”，继续；
     - 异常过滤器是否捕获了特定异常类型（而非对所有异常统一处理，避免信息丢失）：
       - 通过 -> 记录通过，继续下一个审查项；
       - 未通过 -> 记录“建议针对不同异常类型使用多个过滤器或区分处理逻辑”，继续；
     - 响应格式是否一致（所有过滤器返回的结构是否对齐）：
       - 通过 -> 记录通过，继续下一个审查项；
       - 未通过 -> 记录“响应格式不统一，建议定义标准响应接口”，继续；
     - 是否处理了未预期的异常（全局过滤器是否作为兜底存在）：
       - 通过 -> 记录通过，继续下一个审查项；
       - 未通过 -> 记录“缺少全局异常过滤器兜底，未预期异常可能暴露内部信息”，继续；
   - 判断是否有任何问题记录：
     - 是 -> 汇总问题列表，进入下一步；
     - 否 -> 直接进入步骤 4（复核检查）；

3. **提供修改建议** — 对发现的问题给出具体修复方案；
   - 依次对每个问题提供修复建议；
   - 通过 AskUserQuestion 提供选项，阻塞等待用户选择：
     - 采纳全部建议 -> 生成修正后的代码，进入步骤 4；
     - 逐条确认 -> 逐条由用户决定采纳或忽略，全部确认后进入步骤 4；
     - 仅查看不修改 -> 不影响代码，直接进入步骤 4；

4. **复核检查** — 对照 [Review List](#review-list)，确认执行结果；
   - 判断 Review List 是否有内容：
     - 否 -> 直接进入下一步（成果输出）；
     - 是 -> 下一步；
   - 依次判断 [Review List](#review-list) 中每个检查项，是否通过：
     - 基于“复核检查示例”来显示输出内容；
   - 有任一检查失败，终止流程；
   - 全部通过后，进入下一步；

5. **成果输出** — 输出审查摘要，告知完成；
   - 输出结构化摘要表格（审查文件 / 审查项数 / 通过 / 发现问题 / 风险等级 / 已采纳建议）；
   - 告知审查完成；

## Rules

- **内容规范**
  - description 须遵循格式：第一句说明审查内容，第二句说明触发条件，使用第三人称，不超过 1024 字符；
  - 审查项须引用 NestJS 官方关于异常过滤器的文档；
  - SKILL.md 不超过 300 行；超过时拆分到 references/ 目录；
  - 引用层次不超过一层；
  - 保持术语一致，Definitions 中已定义术语在正文中通过锚点链接引用；
  - 删除任何时效性信息；

- **行为规范**
  - 审查时仅输出问题摘要和定位，不直接修改代码，除非用户通过 AskUserQuestion 明确要求；
  - 所有涉及代码修改的交互环节，必须使用 AskUserQuestion 工具；
  - 响应格式统一性建议标注为建议而非强制；

- **防御规范**
  - 如目标代码为空或不可读，直接报告并终止；
  - DI 注入问题仅在可确认注册方式时报告，不确定时标注为需手动确认；

- **验证规范**
  - Examples 内容必须自洽于 Rules；
  - Examples 必须包含复核检查示例；
  - 对话交互示例仅聚焦步骤 0~3（不含复核检查和成果输出）；

## Examples

### 对话交互示例

**示例：用户请求审查异常过滤器配置**

```markdown
用户 > 帮我检查项目的异常处理是否完善
AI   > 检测到用户需要 NestJS 异常过滤器审查，触发 nestjs-2-2-exception-filters 技能
AI   > 正在分析异常处理代码...

审查结果：
- 🟩 自定义异常类 XxxException 继承自 HttpException
- 🟥 全局过滤器在 main.ts 中使用 app.useGlobalFilters()，但需要注入 Logger 服务
- 🟩 控制器级过滤器精确捕获了特定异常类型
- 🟥 不同过滤器的响应格式不一致（有的返回 timestamp，有的没有）
- 🟩 存在全局异常过滤器作为兜底

总评：2 个问题需要关注
用户 > 帮我修复全局过滤器的注入问题
AI   > 建议将全局过滤器在 AppModule 中作为提供者注册。是否应用修改？
用户 > 是
```
