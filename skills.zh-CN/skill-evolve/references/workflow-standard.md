# Workflow 编写规范 — 定义 SKILL.md 中 `## Workflow` 节的完整编写标准

## 概述

定义 SKILL.md 中 `## Workflow` 节的完整编写标准，涵盖固定步骤结构、步骤编号与格式、分支逻辑及交互范式。遵循此规范可确保各 SKILL 的 Workflow 结构一致、层次清晰、便于维护。

## 术语

- **Secure 步骤**：Workflow 中固定出现的三个标准化步骤：前置检查（首个步骤）、复核检查（倒数第二个步骤）、成果输出（末尾步骤），统称为 Secure 步骤。
- **前置检查**：Workflow 的第 0 步（首个步骤），在执行核心逻辑前，验证环境完整性和前置条件。
- **复核检查**：Workflow 的倒数第二个步骤，在执行完成后，对照 Review List 逐项验收结果。
- **成果输出**：Workflow 的末尾步骤，在复核通过后，输出成果摘要并告知完成。

每个 Secure 步骤包含**基础（必选）**和**补充（可选）**两部分：

- **基础**：该步骤的核心必需内容，由步骤本身的核心职责决定；
- **补充**：由目标 SKILL 根据自身场景在 Workflow 步骤中按需添加；

## 固定步骤规范

### 1. 前置检查

**名称**：`0. **前置检查** — ［职责描述］`

**职责**：在开始核心工作流之前，确保后续任务的先决条件已达成。

**标准内容**：

**基础（必选）：**

- 确保前置条件已达成（环境、工具、权限等）：
  - 是 -> 下一步；
  - 否 -> 报告未满足的条件，终止流程或提示用户处理；

**补充（可选）：**

- 保存目标文件的完整原始内容副本（作为回滚基线）；
- 验证模板（如有依赖）存在且结构可解析：
  - 是 -> 下一步；
  - 否 -> 报告”模板文件缺失或损坏”，终止流程；
- 验证 `references/` 下被引用的文件存在，与 References 节保持同步：
  - 是 -> 下一步；
  - 否 -> 列出缺失文件清单，通过 AskUserQuestion 提供选项，阻塞等待用户选择：
    - 跳过缺失项继续 -> 下一步；
    - 终止修复后再运行 -> 终止流程；
- 初始化全局变量（由 Definitions 中声明的变量确定列表），使用通用格式：
  ```
  - 初始化全局变量 [变量A](#锚点A)、[变量B](#锚点B)、...：
      - 依次判断每个变量的条件：
          - 满足 -> 初始化该变量为 true；
          - 不满足 -> 初始化该变量为 false；
  ```

**自动补全规则：**

- 若目标 SKILL.md 的 Workflow 首个步骤不是”前置检查”，AI 应自动在最前面插入该步骤；
- 自动生成的内容直接复刻上述”基础（必选）”部分的完整内容，无需额外修改；
- 补充部分由 AI 根据目标 SKILL 的具体场景判断是否添加；
- 前置检查步骤编号固定为 `0.`；
- 失败处理原则：可恢复的条件提示用户处理；不可恢复的条件直接终止流程。

### 2. 复核检查

**名称**：`N. **复核检查** — 对照 [Review List](#review-list)，确认执行结果`

- 其中 N = 步骤总数 - 1（倒数第二个步骤）

**职责**：在所有操作完成后，对照 Review List 逐项验收执行结果，确保质量达标。

**标准内容**：

**基础（必选）：**

- 判断 Review List 是否有内容：
  - 否 -> 直接进入下一步（成果输出）；
  - 是 -> 下一步；
- 依次判断 [Review List](#review-list) 中每个检查项，是否通过：
  - 基于 ”复核检查示例” 来显示输出内容；
- 有任一检查失败，终止流程；
- 全部通过后，进入下一步（成果输出）；

**自动生成规则：**

- 若目标 SKILL.md 的 Workflow 倒数第二个步骤不是”复核检查”，AI 应自动在末尾步骤之前插入该步骤；
- 自动生成的内容直接复刻上述”基础（必选）”部分的完整内容，无需额外修改；
- 步骤编号自动计算为当前步骤总数 - 1。

### 3. 成果输出

**名称**：`N. **成果输出** — 输出执行摘要，告知完成`

- 其中 N = 步骤总数（末尾步骤）

**职责**：在复核通过后，汇总执行结果，输出成果摘要，告知用户执行完成。

**标准内容**：

**基础（必选）：**

- 输出结构化摘要（具体格式请参考目标 SKILL 的”成果输出示例”）；
- 告知用户完成；

**补充（可选）：**

- 仅列出本次有变化的维度，无变化的维度不输出；

**自动生成规则：**

- 若目标 SKILL.md 的 Workflow 末尾步骤不是”成果输出”，AI 应自动追加该步骤；
- 自动生成的内容必须包含基础（必选）部分：
  - 输出结构化摘要（具体格式请参考目标 SKILL 的”成果输出示例”）；
  - 告知用户完成；
- 补充（可选）部分由 AI 根据目标 SKILL 的上下文判断是否添加；
- 步骤编号自动计算为当前步骤总数。

### 自动补全算法

AI 在检查目标 SKILL.md 的 Workflow 结构时，执行以下逻辑：

1. **解析**：提取 Workflow 中所有 `N. **名称**` 格式的步骤行；
2. **检查首个步骤**：
  - 若首个步骤名称不是“前置检查” -> 判断 Workflow 中是否存在名为“前置检查”的步骤：
    - 存在 -> 将该步骤移动到首位，无需新增；
    - 不存在 -> 在最前面插入前置检查步骤（编号 `0.`）；
3. **检查末尾步骤**：
  - 若末尾步骤名称不是“成果输出” -> 判断 Workflow 中是否存在名为“成果输出”的步骤：
    - 存在 -> 将该步骤移动到末尾，无需新增；
    - 不存在 -> 在末尾追加成果输出步骤；
4. **重新计算步骤总数** N；
5. **检查倒数第二个步骤**：
  - 若倒数第二个步骤名称不是“复核检查” -> 判断 Workflow 中是否存在名为“复核检查”的步骤：
    - 存在 -> 将该步骤移动到成果输出之前，无需新增；
    - 不存在 -> 在成果输出之前插入复核检查步骤；
6. **重新编号**所有步骤（从 0 开始依次递增）；
7. **无需用户确认**，自动执行上述补全操作。

### 一致性要求

- 所有 SKILL.md 的 Workflow 必须包含这三个 Secure 步骤；
- 三个步骤在各 SKILL.md 中的核心职责保持一致：
  - **前置检查**：验证环境，防止中途失败；
  - **复核检查**：逐项验收，确保质量达标；
  - **成果输出**：汇总展示，告知执行完成；
- 允许根据目标 SKILL 的具体场景对子步骤进行适配（如前置检查中是否包含模板验证），但不允许改变步骤的核心职责；
- 自动补全的步骤内容应与目标 SKILL 的上下文相关（如引用文件列表应与该 SKILL 的 References 节同步）。

### 修复重试规范

Workflow 中涉及"修复→重新检查→仍不通过"的迭代循环时：

- 应设置最大重试次数上限（建议 2-3 次）
- 超限后必须在 Workflow 中显式标注处理方案（如"记录失败项，继续下一步"或"终止流程"）
- 降级（记录失败后继续下一步）不是默认行为，须在 Workflow 中显式声明
- 返回点应精确到触发该操作的子步骤，而非整个步骤开头

> 此规范约束的是 Workflow 编写行为，而非 AI 执行行为。AI 在编写/修改 Workflow 时应遵循此规范。

### 示例

#### 前置检查自动补全

```markdown
修改前（缺少前置检查）：

1. **元数据结构** — 检查 frontmatter 内容正确性；
2. **结构对齐** — 对照模板补齐缺失目录；

修改后（自动插入前置检查）：0. **前置检查** — 验证环境完整性；- 验证目标 SKILL.md 存在且可读取：- 是 -> 下一步；- 否 -> 报告“目标文件不存在或无法读取”，终止流程；- 在内存中保存原始内容副本；

1. **元数据结构** — 检查 frontmatter 内容正确性；
2. **结构对齐** — 对照模板补齐缺失目录；
```

#### 复核检查 + 成果输出自动补全

```markdown
修改前（缺少复核检查和成果输出）：4. **内容精简** — 检查并清理 SKILL.md 内容；

修改后（自动追加复核检查和成果输出）：4. **内容精简** — 检查并清理 SKILL.md 内容；5. **复核检查** — 对照 Review List，确认优化结果；- 依次判断 Review List 中每个检查项，是否通过：- 是 -> 下一步；- 否 -> 输出未通过项，终止流程；- 全部通过后，进入下一步；6. **成果输出** — 输出优化总结，告知完成；
```

## 步骤编写格式

本节定义 Workflow 步骤的编号体系、标题命名、子步骤结构等通用格式规则。关于条件分支的详细规范（树形箭头、极性匹配等）由第 3 节完整定义。

### 编号体系

#### 顶层步骤

- 步骤编号从 0 开始，依次递增：`0.`、`1.`、`2.`、`3.`…
- `0.` 保留给 Secure 步骤”前置检查”
- 编号后紧跟一个空格，再接标题

```
0. **前置检查** — 确保后续任务的先决条件已达成；
1. **主流程** — 执行核心逻辑；
```

#### 子步骤编号

- **常规子步骤**：使用缩进项目符号列表（`-`），**不使用**数字编号
- **需定位引用的子步骤**：使用 1.1、1.2、1.3（或 2.1、2.2）格式，**仅当**子步骤间存在循环跳转或需在文档其他位置引用时使用

```
3. **修复循环** — 逐个修复发现的问题；
  3.1 修复当前问题；
  3.2 重新验证修复结果；
  3.3 返回入口判断；
  - 待修复列表为空 -> 下一步；
```

### 标题命名

#### 格式

```
N. **标题** — 描述；
```

| 部分       | 要求                           |
| ---------- | ------------------------------ |
| `N.`       | 步骤编号，后跟一个空格         |
| `**标题**` | 加粗的步骤核心动作，3~10 字    |
| `—`        | 中文全角破折号，前后各一个空格 |
| `描述`     | 简要说明步骤目的，10~30 字     |
| `；`       | 中文分号结尾                   |

标题与描述之间用中文全角破折号 `—` 连接，不使用冒号 `：`、连字符 `-` 或英文破折号 `--`。

#### 示例

```
0. **前置检查** — 确保后续任务的先决条件已达成；
1. **元数据结构** — 检查 frontmatter 内容正确性；
2. **结构对齐** — 对照模板，补齐缺失目录、调整目录顺序；
6. **复核检查** — 对照 Review List，确认优化结果；
7. **成果输出** — 输出优化总结，告知完成；
```

### 子步骤结构

#### 项目符号列表（常规）

子步骤使用 `-` 开头，逐层缩进表示嵌套层次：

```
N. **标题** — 描述；
  - 判断某个条件：
    - 是 -> 下一步；
    - 否 -> 处理方式；
  - 执行某个操作：
    - 成功 -> 下一步；
    - 失败 -> 终止流程；
```

每层缩进 2 个空格。条件分支（`是 -> / 否 ->`）再缩进 2 个空格。

#### <a id="数字子编号（引用场景）"></a>数字子编号（引用场景）

当子步骤之间存在循环跳转或在文档其他位置有交叉引用时，使用 `N.M` 格式：

```
4. **内容精简** — 检查并清理 SKILL.md 内容；
  4.1 判断行数是否超过 300 行：
    - 是 -> 迁移到 references/；
    - 否 -> 下一步；
  4.2 删除时效性信息，保持术语一致；
  4.3 检查抽象变量名：
    - 发现问题 -> 替换后返回 4.1 重新评估；
    - 无问题 -> 下一步；
```

### 条件分支

条件分支的详细规范（树形箭头格式、分支终点、极性匹配、循环迭代、跨步骤引用等）由第 3 节 [分支逻辑写作规范](#分支逻辑写作规范) 完整定义。

本节仅列出在步骤编写格式上下文中最常用的几个要点：

- 树形箭头格式：`- 是 -> 动作；` / `- 否 -> 动作；`
- 分支终点显式化：每个分支必须以 `下一步；`、`终止流程；` 或 `返回 N.M` 结尾
- 每个条件判断必须有明确的 `是` 和 `否` 两个分支，不得缺少否定分支

### 循环与迭代

#### 循环标记

当 Workflow 包含迭代循环时，在循环入口处明确说明循环边界：

```
3. **修复循环** — 逐个修复发现的问题；
  - 判断待修复列表是否为空：
    - 是 -> 下一步（全部修复完成）；
    - 否 -> 继续修复；
  - 修复当前问题；
  - 执行验证，判断验证结果：
    - 通过 -> 继续下一个问题；
    - 未通过 -> 返回当前问题，重新修复；
```

#### 迭代处理

涉及多元素迭代时，显式说明对每个元素的处理路径：

```
- 依次对每个文件执行步骤 4.1~4.3：
  - 全部完成 -> 下一步；
  - 任一失败 -> 终止流程，报告失败文件名；
```

避免仅使用”分别生成””逐个处理”等模糊表述。

### 跨步骤引用

#### 引用格式

在子步骤中引用其他步骤时，使用统一格式：

| 引用类型 | 格式       | 示例           |
| -------- | ---------- | -------------- |
| 顶层步骤 | `步骤 N`   | `步骤 3`       |
| 子步骤   | `步骤 N.M` | `步骤 4.2`     |
| 范围     | `步骤 N~M` | `步骤 4.1~4.3` |

#### 示例

```
- 发现问题 -> 记录后返回步骤 3 重新评估；
- 全部修复后，进入步骤 5（复核检查）；
```

### 常见错误

| ❌ 错误写法                      | ✅ 正确写法                                    | 原因                               |
| -------------------------------- | ---------------------------------------------- | ---------------------------------- |
| `### 1. 标题`                    | `1. **标题** — 描述；`                         | 避免使用 Markdown 标题级别定义步骤 |
| `1) 执行操作`                    | `- 执行操作`                                   | 子步骤统一使用项目符号             |
| `如果条件成立，则继续；否则终止` | `- 是 -> 下一步；否 -> 终止流程；`             | 条件分支使用树形箭头               |
| `循环执行直到完成`               | `- 待处理列表为空 -> 下一步；否 -> 继续处理；` | 循环边界显式化                     |

### 整体示例

以下是一个符合前述所有规范的完整 Workflow：

```markdown
0. **前置检查** — 确保后续任务的先决条件已达成；
  - 确保前置条件已达成：
    - 验证目标文件存在且可读取：
      - 是 -> 下一步；
      - 否 -> 报告“文件缺失”，终止流程；
  - 保存原始内容副本（作为回滚基线）；
1. **执行主流程** — 完成核心业务逻辑；
  - 执行操作 A：
    - 是 -> 下一步；
    - 否 -> 报告错误，终止流程；
2. **修复循环** — 逐个修复发现的问题；
  2.1 判断待修复列表是否为空：
    - 是 -> 下一步（进入复核检查）；
    - 否 -> 继续；
  2.2 修复当前问题；
  2.3 重新验证修复结果：
    - 通过 -> 返回 2.1 处理下一个问题；
    - 未通过 -> 返回 2.2 重新修复；
3. **复核检查** — 对照 Review List，确认执行结果；
  - 判断 Review List 是否有内容：
    - 否 -> 直接进入成果输出；
    - 是 -> 下一步；
  - 依次判断每个检查项，是否通过：
    - 基于 “复核检查示例” 显示输出内容；
  - 有任一检查失败，终止流程；
  - 全部通过后，进入成果输出；
4. **成果输出** — 输出执行摘要，告知完成；
  - 输出结构化摘要；
  - 告知完成；
```

## 分支逻辑写作规范

规范 SKILL.md 中条件分支判断的写法，将平铺的 if-else 转为树形箭头格式（`是 -> / 否 ->`），确保每种分支结果对应明确、可执行的操作指令。每个分支动作行末必须以 `；` 结尾，作为分支结束标记；引入子条件或子操作列表的非终止分支行末以 `：` 结尾。

### 规则 1：分支逻辑树形化

条件分支必须以树形箭头格式呈现，每种结果（是/否）独占一行、缩进表达层级，避免将多个分支结果压缩在同一行。

> **标准格式**：
>
> ```markdown
> - 判断条件：
>   - 是 -> 执行动作 A；
>   - 否 -> 执行动作 B；
> ```

#### 示例 1：简单二元判断

```markdown
**原文**

- 若缺失或不规范，交互式确认修复方案；

**优化后**

- 确认 `[字段名]` 是否存在：
  - 是 -> 下一步；
  - 否 -> 通过 AskUserQuestion 提供选项，阻塞等待用户选择：
  - 文件夹名 -> 使用文件夹名，执行后进入下一步；
  - 基于 SKILL.md 整理出来的文件名 -> 使用文件名，执行后进入下一步；
```

> **关键改进**：原文只有“若缺失“的隐含分支，AI 不知道满足条件时该做什么；优化后明确定义了“是->下一步“。

#### 示例 2：嵌套分支

```markdown
**原文**

- 非模板标准目录的内容 -> 迁移到 `references/` 下（若存在同名文件，交互式确认覆盖、合并或跳过）；

**优化后**

- 非模板标准目录的内容 -> 迁移到 `references/` 下：
  - 是否存在同名文件：
    - 是 -> 通过 AskUserQuestion 提供选项，阻塞等待用户选择：
      - 覆盖 -> 覆盖文件，执行后进入下一步；
      - 合并 -> 合并内容，执行后进入下一步；
      - 跳过 -> 保留原文件，执行后进入下一步；
    - 否 -> 下一步；
```

> **关键改进**：原文将“迁移“和“同名文件处理“压缩在一行，AI 可能混淆执行顺序；优化后通过缩进表达了父子判断关系。

#### 示例 3：存在性检测

```markdown
**原文**

- 若存在 `REFERENCE.md`，先展示拆分方案，确认后执行：

**优化后**

- 检测 `[目标文件]` 是否存在：
  - 是 -> 通过 AskUserQuestion 提供选项，阻塞等待用户选择：
    - [动态选项，由 AI 根据 [目标文件] 内容生成] -> 执行对应拆分方案，执行后进入下一步；
  - 否 -> 下一步；
```

> **关键改进**：原文缺少“不存在时“的行为定义；优化后补充了 `否 -> 下一步`，消除 AI 的行为盲区。

#### 示例 4：多步动作块的冒号对齐

```markdown
**原文**

- 拆分后更新 SKILL.md 中所有指向原文件的链接

**优化后**

- 拆分后更新 SKILL.md 中所有指向原文件的链接：
  - 扫描 SKILL.md 中所有指向原 REFERENCE.md 的链接；
  - 逐一更新为新的拆分文件路径；
  - 验证无死链；
```

> **关键改进**：原文的“更新“是一个模糊动作。优化后以 `：` 结尾引入子操作列表，每个子操作以 `；` 结尾，格式统一、边界清晰。

### 规则 2：幂等守卫

对”是否已符合某规范”的判断，必须在”是”分支标注跳过理由，防止二次执行时重复触发优化。

> **标准格式**：
>
> ```markdown
> - 判断 [元素] 是否已使用 [规范] 标准范式：
>   - 是 -> 下一步（当前内容已满足该目录的模板职责要求，标记为已合规，跳过本轮优化；当模板职责更新或内容发生实质偏离时，需重新执行此检查）；
>   - 否 -> 基于 [规范] 进行优化；
> ```

#### 示例：交互环节幂等检查

```markdown
**原文**

- 判断 `## Workflow` 中是否存在涉及用户抉择的交互环节
  - 是 -> 基于交互式操作写作规范进行优化
  - 否 -> 下一步

**优化后**

- 判断 `## Workflow` 中是否存在涉及用户抉择的交互环节：
  - 是 -> 判断是否已使用 `AskUserQuestion` 标准范式：
    - 是 -> 下一步（当前内容已满足该目录的模板职责要求，标记为已合规，跳过本轮优化；当模板职责更新或内容发生实质偏离时，需重新执行此检查）；
    - 否 -> 基于 [交互式操作写作规范] 进行优化；
  - 否 -> 下一步；
```

> **关键改进**：原版缺少“已合规->跳过“的二级判断，导致二次执行时会把已优化为正确格式的内容再次“优化“，可能造成格式变形。

### 规则 3：依次判断（集合迭代）

当需要对集合中每个元素逐一判断时，使用”依次判断...每个...”前缀，明确迭代范围。

> **标准格式**：
>
> ```markdown
> - 依次判断 [集合] 中每个 [元素]，是否已 [条件]：
>   - 是 -> 下一步（当前内容已满足该目录的模板职责要求，标记为已合规，跳过本轮优化；当模板职责更新或内容发生实质偏离时，需重新执行此检查）；
>   - 否 -> 基于 [规范] 进行优化；
> ```

#### 示例：批量交互环节检查

```markdown
**原文**

- 判断 `## Workflow` 中是否存在涉及用户抉择的交互环节
  - 是 -> 基于交互式操作写作规范进行优化
  - 否 -> 下一步

**优化后**

- 依次判断 `## Workflow` 中每个存在涉及用户抉择的交互环节，是否已使用 `AskUserQuestion` 标准范式：
  - 是 -> 下一步（当前内容已满足该目录的模板职责要求，标记为已合规，跳过本轮优化；当模板职责更新或内容发生实质偏离时，需重新执行此检查）；
  - 否 -> 基于 [交互式操作写作规范] 进行优化；
```

> **关键改进**：“依次判断...每个...“相比“判断...是否存在“，将“存在性检测“升级为“逐项合规检查“，AI 不会在第一个合规项后就停止检查。

### 规则 4：条件极性匹配

条件词的选择必须与分支语义对齐。条件的”肯定/否定”极性决定了分支的走向，错误的极性匹配会导致 AI 误判分支归属。

> **标准格式**：始终使用肯定态条件（“是否存在“、“是否满足“），避免使用否定态条件（“是否不满足“、“如果缺失“）。

#### 示例 1：极性对齐的正确做法

```markdown
**原文**

- 若 description 缺失，交互式确认修复方案；

**优化后**

- 判断 description 是否存在：
  - 是 -> 下一步；
  - 否 -> 通过 AskUserQuestion 提供选项，阻塞等待用户选择：
  - [动态选项，由 AI 根据缺失类型生成] -> 执行对应修复，执行后进入下一步；
```

> **关键改进**：原文的“若...缺失“是否定态条件，但只有“缺失时“的一个分支，缺少“不缺失时“的行为。改为“判断是否存在“后，正反分支的边界自然对齐。

#### 示例 2：肯定态 vs 否定态

```markdown
**错误**

- 判断 description 是否不满足格式要求：
  - 是 -> 通过 AskUserQuestion 提供选项（修复 / 保持现状），阻塞等待用户选择；
  - 否 -> 下一步；
```

> **问题**：条件本身是否定态（“是否不满足“），`是 ->` 对应“不满足->修复“，`否 ->` 对应“满足->跳过“。否定态条件增加 AI 阅读理解负担，可能将“是/否“与“好/坏“混淆。
> **正确**：始终使用肯定态条件 `判断 description 是否满足格式要求`，然后将 `是` / `否` 分支动作互换。

#### 示例 3：单路径句式不接二元分支

```markdown
**错误**

- 当 SKILL.md 超过 300 行时：
  - 是 -> 执行迁移；
  - 否 -> 下一步；
```

> **问题**：“当...时“是单路径条件，不适合接 `是 -> / 否 ->` 二元分支。AI 可能困惑：“当...时“的语义暗示只有满足时才执行动作，没有明确的“不满足“语义。
> **正确**：改用“判断 SKILL.md 是否超过 300 行“

**自检方法**：写下条件后，用自然语言读一遍”是->X”、”否->Y”，验证是否符合直觉。

### 规则 5：分支终点显式化

每个分支（`是 ->` / `否 ->`）必须以显式的流程控制动作结尾，不得仅有输出而无后续流向指令。非终止分支必须有明确的”下一步”指示。

> **标准格式**：
>
> - 终止分支：`是 -> 执行动作；终止流程；`
> - 继续流转：`是 -> 执行动作，执行后进入下一步；`
> - 跳过优化：`是 -> 下一步（当前内容已满足该目录的模板职责要求，标记为已合规，跳过本轮优化；当模板职责更新或内容发生实质偏离时，需重新执行此检查）；`
> - 引入子操作：`是 -> 进入子操作：`（以 `：` 结尾）

#### 示例 1：仅有输出缺少流向

```markdown
**错误**

- 判断文件是否存在：
  - 是 -> 读取内容；
  - 否 -> 提示用户；
```

> **问题**：`是 -> 读取内容；` 之后去了哪里？继续执行后续步骤还是终止？AI 无法确定，可能输出内容后继续执行后续步骤。

```markdown
**正确**

- 判断文件是否存在：
  - 是 -> 读取内容，执行后进入下一步；
  - 否 -> 提示用户文件不存在；终止流程；
```

#### 示例 2：分支终点与冒号配合

```markdown
**正确**（多步动作块）

- 非模板标准目录的内容 -> 迁移到 references/ 下：
  - 是否存在同名文件：
  - 是 -> 通过 AskUserQuestion 提供选项，阻塞等待用户选择：
      - 覆盖 -> 覆盖文件，执行后进入下一步；
      - 合并 -> 合并内容，执行后进入下一步；
      - 跳过 -> 保留原文件，执行后进入下一步；
```

> 非终止行“迁移到 references/ 下：“以全角冒号结尾，表示“此处有子操作展开“；子操作最终以 `；` 结尾，表示“此路径已结束“。

### 分支动作类型

树形分支的 `是 ->` / `否 ->` 后可接以下动作类型：

| 类型       | 格式                                                       | 含义                                                           |
| ---------- | ---------------------------------------------------------- | -------------------------------------------------------------- |
| 下一步     | `下一步；`                                                 | 继续当前步骤内的下一个判断，或进入下一步骤                     |
| 终止流程   | `终止流程；`                                               | 停止执行，不再继续后续步骤                                     |
| 跳过优化   | `下一步（当前内容已满足该目录的模板职责要求，标记为已合规，跳过本轮优化；当模板职责更新或内容发生实质偏离时，需重新执行此检查）；`                             | 当前项已合规，跳过并继续                                       |
| 多步动作块 | `迁移到 references/：` + 缩进子列表                        | 执行一组有序子操作；非终止行以 `：` 结尾，子操作行以 `；` 结尾 |
| 引入建议   | `建议引入对应目录；`                                       | 给出建议但不强制执行                                           |
| 阻塞交互   | `通过 AskUserQuestion 提供选项，阻塞等待用户选择：` + 缩进子列表 `- 选项 -> 动作，执行后进入下一步；` | 暂停执行，通过 UI 组件向用户提供结构化选项并等待选择           |

#### 示例：终止流程

```markdown
- 验证 target 是否存在：
  - 是 -> 下一步；
  - 否 -> 报告“目标不存在”，终止流程；
```

> **与“下一步“的区别**：`终止流程` 是流程级别的结束，后续所有步骤不再执行；`下一步` 仅结束当前判断，继续同一步骤内的后续操作。

### 危险场景分析

> **场景 1：缺少否定分支**
> _示例_：`若文件存在则读取内容` -> 缺少“不存在时“的行为
> _问题_：AI 遇到文件不存在时可能报错或跳过，行为不可预测
> _正确_：明确 `否 -> 提示用户文件不存在并退出`
>
> **场景 2：多分支压缩**
> _示例_：`若 A 则 X，若 B 则 Y` -> 两个独立条件写在同一行
> _问题_：AI 难以判断 A 和 B 是否互斥，可能同时执行 X 和 Y
> _正确_：拆分为两个独立的树形判断块
>
> **场景 3：层级混淆**
> _示例_：`若 A 成立则执行 B，若 C 成立则执行 D`
> _问题_：C 是 A 的子条件还是同级条件？AI 无法判断
> _正确_：用缩进区分父子层级
>
> **场景 4：规范文件自身违反规范**
> _示例_：本文本规范自身的示例 3 中 `- 否 -> 下一步` 曾缺少行末分号（现已修复），但标准格式和示例 1 均以 `；` 结尾
> _问题_：AI 以规范文件为基准学习格式时，会因示例不一致而产生随机行为（有时加分号有时不加）
> _正确_：规范文件自身必须通过其定义的验证清单，确保示例之间格式严格一致
>
> **场景 5：条件极性导致的 AI 行为偏差**
> _示例_：`若 description 格式不符合要求，通过 AskUserQuestion 提供选项`
> _问题_：否定态条件（“不符合要求“）+ 仅单分支，AI 可能遗漏“格式已合规“的路径
> _实际案例_：在 Code Review 中发现，AI 将格式正确的 description 也送入修复流程，因为条件原文没有定义“合规时“的行为
> _正确_：始终使用肯定态条件（“是否存在“、“是否满足“），正反分支边界自然对齐
>
> **场景 6：引号风格不一致导致格式学习紊乱**
> _示例_：本规范早期版本中某个示例使用 `「是」` 而其他示例使用 `“已合规“`，同一文件中出现直角引号与弯引号的混合
> _问题_：AI 以规范文件为基准学习格式时，引号风格的不一致会使 AI 在生成的新内容中随机选择一种风格
> _正确_：统一使用中文弯引号 `""`，禁止在中文正文中使用 `「」`（参照[标点符号使用规范](punctuation-convention.md)规则 3）
>
> **场景 7：新建符号规则后未反向验证规范文件自身**
> _示例_：punctuation-convention.md 规则 6.2 禁止连续全角破折号，但其自身规则 1.1 曾使用 `——`（两连破折号）
> _问题_：新增规则后未立即扫描规范文件自身，造成“规则禁止自身内容“的自相矛盾
> _正确_：新增标点规则后，立即对规范文件自身执行验证清单检查

### 常见反模式

#### 反模式 1：隐式 else

只有”是”分支，缺少”否”分支。

> **错误**：`若 [字段名] 缺失，交互式确认` -> 缺少“否“分支（[字段名] 存在时该如何？）
> **正确**：
>
> ```markdown
> - 判断 name 是否存在：
>   - 是 -> 下一步；
>   - 否 -> 通过 AskUserQuestion 提供确认选项（由 AI 根据上下文生成），阻塞等待用户选择；
> ```

#### 反模式 2：内联嵌套

将嵌套条件写在括号或同一行中。

> **错误**：`迁移到 references/ 下（若存在同名文件，交互式确认覆盖、合并或跳过）`
> **正确**：用缩进层级展开：
>
> ```markdown
> - 迁移到 references/ 下：
>   - 是否存在同名文件：
>     - 是 -> 通过 AskUserQuestion 提供选项，阻塞等待用户选择：
>       - 覆盖 -> 覆盖文件，执行后进入下一步；
>       - 合并 -> 合并内容，执行后进入下一步；
>       - 跳过 -> 保留原文件，执行后进入下一步；
>     - 否 -> 下一步；
> ```

#### 反模式 3：箭头缺失

有分支内容但未标注 `是 ->` / `否 ->`。

> **错误**：
>
> ```markdown
> - 检查文件是否存在；
>   - 读取内容；
>   - 提示用户；
> ```
>
> **问题**：缩进不等于分支，“不存在时“还提示用户吗？
> **正确**：显式标注箭头

#### 反模式 4：缺少幂等守卫

对”是否已符合规范”的判断不检查现有合规状态，直接把所有匹配项送入优化流程。

> **错误**：`判断 Workflow 中是否存在交互环节 -> 是 -> 基于交互式操作写作规范进行优化`
> **问题**：已合规的内容被二次“优化“，可能导致格式变形或产生新问题
> **正确**：增加二级判断 `已使用 AskUserQuestion 标准范式？-> 是 -> 下一步（当前内容已满足该目录的模板职责要求，标记为已合规，跳过本轮优化；当模板职责更新或内容发生实质偏离时，需重新执行此检查）`

#### 反模式 5：条件极性错位

条件词与分支语义不匹配。

> **错误**：`当 X 不符合要求时：` - 是 -> 修复；- 否 -> 下一步；
> **问题**：条件本身是否定态（“不符合要求“），`是 ->` 对应“不符合要求“，直觉上容易误读。且“当...时“是单路径句式，不适合二元分支。
> **正确**：`判断 X 是否符合要求：` - 是 -> 下一步；- 否 -> 修复；

#### 反模式 6：隐式流控制

分支未能显式声明执行后的流向。

> **错误**：
>
> ```markdown
> - 验证目标文件是否存在：
>   - 是 -> 读取内容；
>   - 否 -> 报错退出；
> ```
>
> **问题**：“读取内容“之后去哪里？“报错退出“之后流程是否终止？AI 只能猜测。
> **正确**：明确标注“下一步；“或“终止流程；“。

## 交互式操作写作规范

规范 SKILL.md 中涉及用户抉择的交互环节的写法，确保 AI 正确使用 `AskUserQuestion` 工具进行阻塞式 UI 确认，而非以纯文本追问代替。

### 规则 1：Rules 中的交互式确认规则

`## Rules` 中必须包含使用 `AskUserQuestion` 的行为兜底。

**必须包含的规则**：

```markdown
- 所有涉及用户抉择的交互环节（确认修复方案、选择处理方式、确认删除/覆盖/合并等），**必须**使用 `AskUserQuestion` 工具，禁止以纯文本追问代替；将问题与选项结构化传入 AskUserQuestion，每次调用 ≤4 个问题；
```

目的：为未进行明确设定的交互操作进行兜底；

### 规则 2：Workflow 中的交互编写范式

Workflow 中涉及用户抉择的步骤，必须以固定范式编写，确保 AI 生成的结构化选项清晰可执行。

> **标准范式**：`通过 AskUserQuestion 提供选项，阻塞等待用户选择：` + 缩进子列表
>
> ```markdown
> - 选项A -> 对应动作，执行后进入下一步；
> - 选项B -> 对应动作，执行后进入下一步；
> - 选项C -> 对应动作，执行后进入下一步；
> ```

#### 示例 1：单条件交互

```markdown
**原文**

交互式确认覆盖、合并或跳过；

**优化后**

通过 AskUserQuestion 提供选项，阻塞等待用户选择：
  - 覆盖 -> 覆盖文件，执行后进入下一步；
  - 合并 -> 合并内容，执行后进入下一步；
  - 跳过 -> 保留原文件，执行后进入下一步；
```

#### 示例 2：确认性交互

```markdown
**原文**

由用户确认是否采用、留空或跳过；

**优化后**

通过 AskUserQuestion 提供选项，阻塞等待用户选择：
  - 采用 -> 下一步；
  - 留空 -> 下一步；
  - 跳过 -> 下一步；
```

> **危险场景**：使用模糊表述替代范式
> _示例_：`询问用户如何处理` -> AI 可能输出纯文本问题而非调用工具
> _正确_：`通过 AskUserQuestion 提供选项，阻塞等待用户选择：` + 缩进子列表

### 规则 3：动态选项

当选项内容依赖上下文分析结果、无法在 SKILL.md 中预先枚举时，使用动态选项模式。

> **标准范式**：`通过 AskUserQuestion 提供选项，阻塞等待用户选择：`
>
> ```markdown
> - [动态选项，由 AI 根据 [分析对象] 生成] -> 执行选项对应操作，执行后进入下一步；
> ```

#### 示例：依赖上下文生成的选项

```markdown
**原文**

需要用户确认 description 格式的修复方案；

**优化后**

通过 AskUserQuestion 提供选项，阻塞等待用户选择：
  - [动态选项，由 AI 根据 description 缺失类型生成] -> 执行对应修复，执行后进入下一步；
```

> **适用条件**：选项取决于待优化内容的具体特征（如缺失类型、不符合项内容、拆分领域），预定义不可行时使用。

### 规则 4：选项流向标注

每个 `AskUserQuestion` 的选项后必须显式标注动作流向，确保用户选择后 AI 能明确下一步行为。

> **标准格式**：
>
> ```markdown
> 通过 AskUserQuestion 提供选项，阻塞等待用户选择：
>     - 覆盖 -> 覆盖文件，执行后进入下一步；
>     - 合并 -> 合并内容，执行后进入下一步；
>     - 跳过 -> 保留原文件，执行后进入下一步；
> ```

#### 示例：带流向标注的交互

```markdown
**原文**

通过 AskUserQuestion 提供选项（覆盖 / 合并 / 跳过），阻塞等待用户选择；

**优化后**

通过 AskUserQuestion 提供选项，阻塞等待用户选择：
  - 覆盖 -> 覆盖文件，执行后进入下一步；
  - 合并 -> 合并内容，执行后进入下一步；
  - 跳过 -> 保留原文件，执行后进入下一步；
```

> **关键改进**：无流向标注时，用户选择后 AI 行为方向不明确。增加 `-> 动作；下一步` 后每个选项的行为唯一确定。
>
> **动态选项适配**：动态选项同样需标注流向，在描述动态选项范围时附加流向模板：
> ```markdown
> 通过 AskUserQuestion 提供选项，阻塞等待用户选择：
>     - [动态选项，由 AI 根据 [分析对象] 生成，每个选项后标注流向] -> 对应动作，执行后进入下一步；
> ```

### 常见反模式

#### 反模式 1：纯文本追问

用普通消息替代 `AskUserQuestion` 工具调用。

> **错误**：`向用户确认是否删除`
> **问题**：AI 以文本输出代替 UI 确认，可能不阻塞等待
> **正确**：
>
> ```markdown
> 通过 AskUserQuestion 提供选项，阻塞等待用户选择：
>     - 删除 -> 删除文件，执行后进入下一步；
>     - 保留 -> 保留文件，执行后进入下一步；
> ```

#### 反模式 2：选项描述缺失

只提”询问用户”但未给出具体选项。

> **错误**：`询问用户选择处理方式`
> **问题**：AI 不知道应提供哪些选项，可能遗漏关键选项
> **正确**：
>
> ```markdown
> 通过 AskUserQuestion 提供选项，阻塞等待用户选择：
>     - 覆盖 -> 覆盖文件，执行后进入下一步；
>     - 合并 -> 合并内容，执行后进入下一步；
>     - 跳过 -> 保留原文件，执行后进入下一步；
> ```

#### 反模式 3：超量选项

单次调用超过 4 个问题。

> **错误**：一个 `AskUserQuestion` 调用包含 6 个问题
> **问题**：UI 展示混乱，用户决策负担重
> **正确**：拆分为多次调用，每次 ≤4 个问题

#### 反模式 4：缺少流向标注

选项仅列名称，未标注选择后的行为方向。

> **错误**：`通过 AskUserQuestion 提供选项（覆盖 / 合并 / 跳过），阻塞等待用户选择`
> **问题**：用户选择“覆盖”后 AI 不确定具体行为，可能执行错误的后续操作
> **正确**：
>
> ```markdown
> 通过 AskUserQuestion 提供选项，阻塞等待用户选择：
>     - 覆盖 -> 覆盖文件，执行后进入下一步；
>     - 合并 -> 合并内容，执行后进入下一步；
>     - 跳过 -> 保留原文件，执行后进入下一步；
> ```

## Review List

目标 SKILL.md 的 `## Review List` 节，包含执行完成后需验证的检查项清单。

### Review List 编写规则

- Review List 引用外部验证清单时，必须指向具体子分组锚点（如 `#交互规范验证`），不得引用未经分组的完整清单
- Review List 中每条检查项应覆盖 Rules 对应规则的所有子条件，不能仅覆盖部分

## 验证清单

### 结构验证

- [ ] Workflow 包含三个 Secure 步骤（前置检查、复核检查、成果输出）
- [ ] 步骤编号从 0 开始，依次递增
- [ ] 子步骤使用缩进项目符号列表（`-`），数字编号仅用于循环跳转/引用场景
- [ ] 每个步骤标题符合格式 `N. **标题** — 描述；`
- [ ] 步骤标题不使用 Markdown 标题级别（`###`）定义

### 分支逻辑验证

- [ ] 每个条件判断都有明确的 `是 ->` 和 `否 ->` 分支
- [ ] 嵌套分支通过缩进正确表达父子层级
- [ ] 每个分支指向具体、可执行的操作指令
- [ ] 不存在括号内嵌套条件（反模式 2）
- [ ] 不存在隐式 else（反模式 1）
- [ ] 不存在多个独立条件写入同一行的情况（危险场景 2）
- [ ] 对”是否已符合规范”的判断包含二级幂等守卫（已合规 -> 跳过）
- [ ] 集合迭代使用”依次判断...每个...”前缀，而非”判断...是否存在”

### 格式验证

- [ ] 每个终止分支行末以 `；` 结尾；引入子条件或子操作列表的非终止分支行末以 `：` 结尾
- [ ] 规范文件自身的所有示例格式严格一致（自指一致性）
- [ ] 条件极性匹配：条件词使用肯定态（”是否存在”、”是否满足”），非否定态（”是否不满足”、”如果缺失”）
- [ ] 分支终点显式化：每个 `是 ->` / `否 ->` 分支以具体动作结尾，非裸条件
- [ ] 多步动作块冒号对齐：非终止行（引入子操作）以 `：` 结尾，子操作行以 `；` 结尾
- [ ] 引号风格统一：引用文本时使用中文弯引号 `""`，不使用 `「」`（参照[标点符号使用规范](punctuation-convention.md)规则 3）
- [ ] Review List 引用外部验证清单时使用具体子分组锚点（`#锚点名`）；仅当目标文件的验证清单无子分组时引用 `#验证清单`

- [ ] Workflow 中每个交互环节使用标准范式 `通过 AskUserQuestion 提供选项，阻塞等待用户选择：` + 缩进子列表
