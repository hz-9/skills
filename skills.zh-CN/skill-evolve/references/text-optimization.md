# 文本简化规则与安全边界 — 在精简 SKILL.md 内容时遵循的压缩规则与安全约束

## 概述

在精简 SKILL.md 内容时，遵循以下规则压缩文字长度，同时保证不丢失关键语义，避免 AI 因指令过简而遗漏动作或误判行为。

## 核心原则

> 简化可以压缩文字的“长度”，但不能压缩信息的“语义密度”。

**判断标准**：去掉某句话后，AI 是否可能做出不同决策或遗漏某个动作？如果是，就不能省。

## 全局保护

> 任何包含 `AskUserQuestion` 的交互决策句免于简化。不适用本文的任何压缩规则（含规则 1 正向命令化）。各规则中的 AskUserQuestion 白名单作为补充说明，进一步阐述各规则下禁止简化的原因。

## 规则 1：正向命令化

把“如果不存在则创建”改为“确保存在”。

> **安全场景**：纯技术操作，无时机要求
> _示例_：`若目录不存在则创建` -> `确保目录存在`
>
> **危险场景**：涉及执行时机或顺序
> _示例_：`若 references/ 目录不存在，于迁移前创建` -> 不可简化为 `确保 references/ 目录存在`
> _原因_：丢失“迁移前”时间点，AI 可能在全流程末尾才创建

## 规则 2：去冗余动词

删除 AI 可隐含推理的步骤说明。

> **安全场景**：AI 默认会执行的隐含前置步骤
> _示例_：`读取文件，评估内容` -> `评估内容`
>
> **危险场景**：明确声明后 AI 才会执行的交互动作
> _示例_：`检查后有违规项则返回对应步骤修正` -> 不可简化为 `检查违规项`
> _原因_：删除了校验环节中的迭代修复语义，AI 可能直接标记失败而非返回修正
>
> **白名单**：`AskUserQuestion` 范式语句
> _示例_：`通过 AskUserQuestion 提供选项，阻塞等待用户选择：` + 缩进子列表 -> “阻塞等待用户选择”不可简化为“确认”
> _原因_：“阻塞等待”是工具调用的关键行为约束，简化后 AI 可能不等待用户响应

## 规则 3：省略自解释

删除括号内不加说明也能理解的补充描述。

> **安全场景**：括号内是纯补充性信息
> _示例_：`展示拆分方案（列出拟创建的文件名）` -> `展示拆分方案`
>
> **危险场景**：括号内是边界条件或例外情况
> _示例_：`交互式确认覆盖、合并或跳过（同名文件存在时）` -> 不可省略“同名文件存在时”
> _原因_：缺少边界条件限定，AI 可能对所有文件都执行确认流程，而非仅在冲突时
>
> **白名单**：`AskUserQuestion` 范式语句的选项列表
> _示例_：`通过 AskUserQuestion 提供选项，阻塞等待用户选择：` + 缩进子列表 -> 选项不可省略
> _原因_：选项是交互工具的结构化参数，省略后 AI 无法确定应提供哪些选项

## 规则 4：语义去重

合并指向同一关注点的多个子项。

> **安全场景**：两个子项的边界条件完全相同
> _示例_：`检查链接是否有效；修复无效链接` -> `修复无效链接`
>
> **危险场景**：两个子项表面相似但各有不同边界
> _示例_：`扫描 SKILL.md 中所有指向原 REFERENCE.md 的链接；更新为新的拆分文件路径` -> 不可合并为 `更新链接`
> _原因_：丢失“扫描范围”和“替换目标”两个独立操作指令
>
> **白名单**：包含 `AskUserQuestion` 的“定义 + 约束”闭环
> _示例_：Definitions 中的交互式确认定义 与 Rules 中的交互约束规则 -> 不可合并为一条
> _原因_：两者形成“定义 + 执行约束”闭环，分别约束工具用途和行为边界，合并后 AI 可能只遵循其一

## 规则应用优先级

4 条规则按风险从低到高排序，推荐按以下顺序逐条执行：

| 优先级 | 规则       | 风险等级 | 理由                                                                |
| ------ | ---------- | -------- | ------------------------------------------------------------------- |
| 1      | 正向命令化 | 🟢 低    | 纯句式转换，不改变信息量                                            |
| 2      | 去冗余动词 | 🟢 低    | 删除 AI 已隐含执行的步骤，`AskUserQuestion` 范式语句为白名单        |
| 3      | 省略自解释 | 🟡 中    | 需确认括号内不是边界条件，`AskUserQuestion` 选项列表为白名单        |
| 4      | 语义去重   | 🔴 高    | 需确认两项的边界条件完全相同，`AskUserQuestion` 定义+约束闭环不合并 |

**执行策略**：从低风险开始逐条检查，高风险规则如不确定，保留原文而非强行简化。

> **与其他规范的关系**：`workflow-standard.md` 定义的展开格式（树形分支、交互范式、幂等守卫等）优先级高于本文的压缩规则。当同一段文本同时满足展开和压缩条件时，保留展开格式。例如：已按 `workflow-standard.md` 展开的树形分支（`是 -> / 否 ->`）、幂等守卫（`（已合规，跳过优化）`）及依次判断前缀均不可被本文规则重新压缩为单行条件。

## 复合场景示例

以下展示一段真实 SKILL.md 段落，运用多条规则进行优化。

**原始段落：**

```markdown
1. 访问配置文件，检查配置文件是否存在；如果不存在，向用户提示"配置文件缺失"并退出；
2. 读取配置文件内容，验证配置项完整性；
3. 对缺失的配置项，检查是否存在默认值；如果存在默认值，使用默认值补充；如果不存在默认值，跳过该项；
4. 返回最终配置对象；
```

**逐规则分析：**

| 规则                   | 应用位置                  | 是否安全 | 理由                                          |
| ---------------------- | ------------------------- | -------- | --------------------------------------------- |
| 正向命令化             | 第1行“检查是否存在”       | ✅       | 纯操作转换                                    |
| 正向命令化（条件合并） | 第1行“否则提示用户并退出” | ✅       | 不涉及 AskUserQuestion 交互式确认，可安全合并 |
| 去冗余动词             | 第1行“访问”、第2行“读取”  | ✅       | AI 默认会读取目标文件                         |
| 语义去重               | 第3行两个“如果”           | ❌ 跳过  | 前者需用默认值补充，后者需跳过，是不同的操作  |

**优化后段落：**

```markdown
1. 确保配置文件存在；否则提示用户并退出；
2. 验证配置项完整性；
3. 对缺失的配置项：有默认值则补充，无则跳过；
4. 返回最终配置对象；
```

**效果**：原始 99 字 -> 优化后 58 字，缩减 41%，且语义完整。

## 常见反模式

### 反模式 1：过度合并

将不同边界条件的操作强行合并为一句。

> **错误**：`检查是否有未提交变更；检查是否有未推送提交` -> `检查 git 状态`
> **问题**：合并后 AI 可能只执行 `git status` 一次，不区分未提交和未推送
> **正确**：明确保留两个独立动作，或合并为 `检查是否有未提交变更和未推送提交`

### 反模式 2：过度省略

删掉与用户交互相关的指令，认为 AI 会“自动”沟通。

> **错误**：`若目录不存在则创建`（省略了原版中“提示用户确认”）
> **问题**：AI 直接静默创建目录，用户不知情
> **正确**：保留交互指令 `向用户确认后创建`

### 反模式 3：过度正向化

把带条件时机的指令硬转为“确保”，丢失执行顺序。

> **错误**：`拆分完成后，更新 SKILL.md 中的所有链接` -> `确保链接更新`
> **问题**：AI 可能在拆分前就更新链接，顺序颠倒
> **正确**：保留顺序关系 `拆分后更新链接`

### 反模式 4：过度去重

把“检查范围”和“执行操作”合并，丢失操作细节。

> **错误**：`扫描所有源文件；提取公共函数` -> `提取公共函数`
> **问题**：AI 只执行提取，不执行扫描，可能遗漏文件
> **正确**：保留两个动作或合并为 `扫描源文件并提取公共函数`

## 验证清单

完成文本简化后，逐一确认以下内容：

- [ ] 每条规则引用来源明确（指向对应规则编号）
- [ ] 删除的内容不属于用户交互指令
- [ ] 删除的内容不包含边界条件或例外情况
- [ ] 合并后的项确实指向同一关注点
- [ ] 执行时机/顺序关系未被淡化为“确保”
- [ ] 简化后的指令 AI 仍能执行出不重复、不遗漏的结果
