# Rules 编写标准 — 定义 Rules 的编写规范、分组建议和变量声明规范

## Overview

Rules 约束 AI 的执行行为（process），告诉 AI 执行时怎么做。

## 内容边界

- **应写入 Rules**：元数据规范、结构规范、内容规范、行为规范、防御规范、验证规范
- **不应写入 Rules**：结果验证（进 Review List）、元数据定义（进 Definitions）、编写标准（进本文件）

## 分组建议

- 条数 ≥ 10 时使用两级缩进列表分组（如 `- **分组名**`）
- 推荐按以下六个维度分组，此方案为 Rules 分组权威定义，Review List 分组应据此对齐：

| #   | 分组名         | 定位                   | 包含范围                                                      |
| --- | -------------- | ---------------------- | ------------------------------------------------------------- |
| 1   | **元数据规范** | 技能自身声明           | name、description、frontmatter 字段校验                       |
| 2   | **结构规范**   | 目录与文件结构         | 标准目录、文件结构约束                                        |
| 3   | **内容规范**   | 引用、锚点、路径、术语 | 引用层次、规范冲突、路径匹配、锚点引用、术语定义              |
| 4   | **行为规范**   | 文件操作与交互         | 文件增删改、AskUserQuestion 交互、Workflow 结构、引用执行行为 |
| 5   | **防御规范**   | 副作用与错误处理       | 副作用同步、错误分级、回滚机制                                |
| 6   | **验证规范**   | 自洽性与引用验证       | Concern 分离、Workflow-Rules-RL 对齐、Examples 自洽           |

## 变量声明规范

Workflow 中使用的跨步骤流程变量，须遵循以下规范：

- **声明位置**：在 `## Definitions` 中以带 `<a id="">` 锚点的列表条目声明；
- **命名格式**：以“是否”或“是否为”开头（如“是否跨步骤循环引用”、“是否为技能原始仓库”），名称体现变量获取规则；
- **变量描述**：在 Definitions 条目中说明获取规则（可包含文件引用），AI 据此知道如何判定该变量的值；
- **初始化位置**：在 Workflow 的 Step 0（前置检查）中使用通用模式初始化，格式为：
  ```markdown
  - 初始化全局变量 [变量A](#锚点A)、[变量B](#锚点B)、...：
    - 依次判断每个变量的条件：
      - 满足 -> 初始化该变量为 true；
      - 不满足 -> 初始化该变量为 false；
  ```
- **变量读取**：在需要读取变量值的步骤中，通过锚点链接引用 Definitions 中的变量定义；
- **作用域规则**：仅跨步骤使用的全局变量需要声明；仅在单个子步骤内使用的局部变量无需声明。

## 与 Review List 的关系

Rules 与 Review List 按 Concern 分离，不强制一一对应：

- Rules 约束 AI 的执行行为（process），Review List 验证输出质量（result）
- 纯结构类检查项不需要对应 Rule
- 纯过程类约束且 Workflow 已覆盖的不需要对应 Review List 项

## 验证清单

- [ ] 所有 Rules 约束的是 AI 执行行为（process），非结果验证
- [ ] 没有将 Review List 的检查项误放在 Rules 中
- [ ] 分组阈值合理（≥10 条时分组的建议已遵循）
- [ ] 分组使用了推荐的六个维度（元数据/结构/内容/行为/防御/验证）
- [ ] Rules 内部一致性：同组内规则间无矛盾，跨组规则在交互场景中无冲突（如行为规范的文件删除确认与编辑范围例外不应矛盾）
