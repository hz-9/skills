# Skill Evolve

## Overview

对已有 SKILL.md 执行一次性结构进化：对齐标准模板、精简冗余内容、拆分参考文档以提升可维护性。

由 skill-evolve-cycle 循环进化技能在小循环A阶段调用，也可独立运行完成单次优化。

## Definitions

- <a id="相似含义目录"></a>**相似含义目录**：与标准目录语义相同但历史命名不同的目录（如 `Description`↔`Overview`、`Checklist`↔`Review List`），由 AI 自动识别是否存在及对应的目标目录，是否采用或留空由用户确认。
- <a id="引用层次"></a>**引用层次**：SKILL.md 直接链接 `references/` 下文件为一层；`references/` 文件再链接外部资源为二层，应避免。
- <a id="secure-步骤"></a>**Secure 步骤**：Workflow 中的一组固定标准化步骤，其具体构成和结构由 [Workflow 编写规范](references/workflow-standard.md) 定义。
- <a id="抽象变量名"></a>**抽象变量名**：在 reference 文件中用于替代具体数值或路径的泛化占位符，通常包含在方括号 `[]` 中（如 `[文件名]`、`[目录名]`），确保引用内容不随被引用文件内容变更而失效。
- <a id="模板标准目录"></a>**模板标准目录**：[模板](template.md) 中通过 `##` 标题节定义的标准目录集合，是结构对齐的目标基准。
- <a id="是否跨步骤循环引用"></a>**是否跨步骤循环引用**：标记目标 SKILL.md 的 Workflow 子步骤间是否存在跨步骤循环跳转引用，控制是否需要使用数字子编号格式。根据 [Workflow 编写规范](references/workflow-standard.md#数字子编号（引用场景）) 分析引用关系后初始化。
- <a id="退出路径"></a>**退出路径**：Workflow 中终止执行的所有可能出口，包括正常完成、用户终止、错误终止和复核失败终止四种类型。
- <a id="原始内容副本"></a>**原始内容副本**：在 Workflow 前置检查阶段保存于内存中的目标 SKILL.md 初始完整内容，用作不可恢复错误时的回滚基线。
- <a id="是否自身进化"></a>**是否自身进化**：标记目标 SKILL.md 是否为 skill-evolve 自身。由步骤 0 通过比较目标路径的规范化绝对路径与本 SKILL.md 文件的规范化绝对路径是否一致来初始化，一致则为 true。

## Prerequisites

待修改和优化对象已存在，本 SKILL 负责优化，而非从零创建。如需从零创建技能，使用 `skill-create`。

## Workflow

0. **前置检查** — 确保后续任务的先决条件已达成；

  - 验证目标 SKILL.md 存在且可读取：
    - 是 -> 下一步；
    - 否 -> 报告“目标文件不存在或无法读取”，终止流程；
  - 验证 `template.md` 存在且 frontmatter/主体结构可解析：
    - 是 -> 下一步；
    - 否 -> 报告“模板文件缺失或损坏”，终止流程；
  - 验证 `references/` 下被引用的 reference 文件存在（文件列表需与 `## References` 节保持同步）：
    - 是 -> 下一步；
    - 否 -> 列出缺失文件清单，通过 AskUserQuestion 提供选项，阻塞等待用户选择：
      - 跳过缺失项继续 -> 跳过缺失文件，继续执行，执行后进入下一步；
      - 终止修复后再运行 -> 终止流程；
  - 在内存中保存目标 SKILL.md 的完整[原始内容副本](#原始内容副本)（作为回滚基线）；
  - 初始化全局变量 [是否跨步骤循环引用](#是否跨步骤循环引用)、[是否自身进化](#是否自身进化)：
    - 依次判断变量条件（参照变量定义中的获取规则）：
      - 是 -> 初始化该变量为 true；
      - 否 -> 初始化该变量为 false；

1. **元数据结构** — 检查并修正 frontmatter 内容正确性；
  - 判断 `name` 是否存在：
    - 是 -> 判断是否与所在目录名一致：
      - 是 -> 下一步；
      - 否 -> 强制修正为目录名，执行后进入下一步；
    - 否 -> 以上一级目录名为准设置 name，执行后进入下一步；
  - 判断 `description` 是否存在以及满足[元数据规范](#元数据规范)中的格式要求：
    - 是 -> 下一步；
    - 否 -> 通过 AskUserQuestion 提供选项，阻塞等待用户选择：
      - 接受修复方案 -> 更新 description，执行后进入下一步；
      - 手动指定 -> 由用户提供描述后更新，执行后进入下一步；
2. **结构对齐** — 对照模板补齐缺失目录、调整顺序；
  - 依次判断[模板标准目录](#模板标准目录)（即[模板](template.md)中定义的 `##` 标题节）是否存在：
    - 是 -> 下一步（当前内容已满足该目录的模板职责要求，标记为已合规，跳过本轮优化；当模板职责更新或内容发生实质偏离时，需重新执行此检查）；
    - 否 -> 通过 AskUserQuestion 提供选项，阻塞等待用户选择：
      - 采用 -> 重命名[相似含义目录](#相似含义目录)为[模板标准目录](#模板标准目录)名（保留原内容），执行后进入下一步；
      - 留空 -> 创建该目录节（内部填写"暂无内容"），执行后进入下一步；
  - 判断[Secure 步骤](#secure-步骤)（前置检查、复核检查、成果输出）是否都存在：
    - 是 -> 下一步；
    - 否 -> 基于[Workflow 编写规范](references/workflow-standard.md#自动补全算法)自动补全（插入/移动/重编号），无需用户确认，执行后进入下一步；
  - 非[模板标准目录](#模板标准目录)的内容 -> 参照[内容边界标准](references/content-boundary.md)迁移到 `references/` 下：
    - 判断是否存在同名文件：
      - 是 -> 通过 AskUserQuestion 提供选项，阻塞等待用户选择：
        - 覆盖 -> 以新内容覆盖同名文件，更新被迁移内容中指向原位置的内部链接，并扫描 SKILL.md 及 references/ 下所有指向该非标准目录的锚点链接更新为新路径，执行后进入下一步；
        - 合并 -> 合并新内容到同名文件，更新被迁移内容中指向原位置的内部链接，并扫描 SKILL.md 及 references/ 下所有指向该非标准目录的锚点链接更新为新路径，执行后进入下一步；
        - 跳过 -> 保留同名文件不变，跳过迁移，执行后进入下一步；
      - 否 -> 以新文件名创建文件，更新被迁移内容中指向原位置的内部链接，并扫描 SKILL.md 及 references/ 下所有指向该非标准目录的锚点链接更新为新路径，执行后进入下一步；
    - 迁移完成后删除原位置的非标准目录内容（跳过场景除外）；
  - 判断[是否跨步骤循环引用](#是否跨步骤循环引用)：
    - 是 -> 使用数字子编号格式对齐子步骤，执行后进入下一步；
    - 否 -> 使用常规项目符号格式，执行后进入下一步；
  - 调整目录顺序；
3. **格式标准化** — 对齐模板职责，统一格式，清理时效性内容；
  - 对照[模板](template.md)中各个目录内容的职责要求，判断是否已合规：
    - 是 -> 下一步（当前内容已满足该目录的模板职责要求，标记为已合规，跳过本轮优化；当模板职责更新或内容发生实质偏离时，需重新执行此检查）；
    - 否 -> 通过 AskUserQuestion 提供选项，阻塞等待用户选择：
      - [动态选项，由 AI 根据模板职责与当前内容差异生成] -> 执行对应优化，执行后进入下一步；
  - 【格式统一检查】依次对照以下参考文件的 `## 验证清单`，对目标 SKILL.md 执行逐项检查与修复（须**逐项读取**每个检查项并逐一确认，每个方面出现不符合项时，基于对应规范修复后重新检查该方面，最多 2 次，仍不通过则记录失败项，继续下一方面）：
    - Workflow 结构/格式/分支逻辑/交互范式 -> 参照 [Workflow 编写规范](references/workflow-standard.md#验证清单)
    - 标点符号/引号/括号/全角半角 -> 参照 [标点符号使用规范](references/punctuation-convention.md#验证清单)
    - 文本简化 -> 参照 [文本简化规则](references/text-optimization.md#验证清单)
    - Rules 规范 -> 参照 [Rules 编写标准](references/rules-standard.md#验证清单)
    - Review List 规范 -> 参照 [Review List 编写标准](references/review-list-standard.md#验证清单)
    - 示例格式规范 -> 参照 [示例编写规范](references/example-standard.md#验证清单)
    - [Secure 步骤](#secure-步骤)结构 -> 参照 [Workflow 编写规范](references/workflow-standard.md#固定步骤规范)
    - 目录结构一致性 -> 参照 [SKILL 目录结构](references/directory-structure.md#验证清单) 中的 references/ 文件内容规范
    - Examples 易变数字 -> 参照 [示例编写规范](references/example-standard.md#验证清单)
  - 删除时效性信息（如具体日期、版本号），保持术语一致；
  - 检查 `references/` 下文件的[抽象变量名](#抽象变量名)：引用 SKILL.md 或其他 reference 的具体可变量替换为 `[文件名]`、`[目录名]` 等抽象变量名；
  - 【术语引用完整性审计】逐一扫描 `## Definitions` 中每个定义条目，按以下流程检查全文引用：
    - 取出当前定义名称和 `<a id="">` 锚点名；
    - 扫描 SKILL.md 及 `references/` 下所有 .md 文件，定位所有包含该定义名称的出现位置；
    - 对每个出现位置判断引用格式：
      - 已使用 `[术语名](#锚点名)` 锚点级链接 -> 继续下一个出现位置；
      - 裸文本（无链接） -> 替换为 `[术语名](#锚点名)` 锚点级链接，继续下一个出现位置；
    - 全部分析完成后，统计修复数量，记录到优化报告；
  - 判断[是否自身进化](#是否自身进化)：
    - 是 -> 验证 template.md 是否因本次修改需同步更新（如新增标准目录、修改锚点 ID、变更引用路径等）：
      - 是 -> 通过 AskUserQuestion 提供选项，阻塞等待用户选择：
        - 同步更新 template.md -> 执行同步更新，执行后进入下一步；
        - 跳过，后续手动处理 -> 跳过同步，执行后进入下一步；
      - 否 -> 下一步；
    - 否 -> 下一步；
4. **内容精简** — 检查并精简 SKILL.md 内容与行数；
  - 判断行数是否超过 500 行：
    - 是 -> 优先尝试在 SKILL.md 中优化（减少空白行，其次按[文本简化规则](references/text-optimization.md)压缩内容，禁止压缩语义密度）：
      - 优化后是否仍超过 500 行：
        - 是 -> 迁移部分内容到 `references/`，更新 SKILL.md 中指向原内容的链接，执行后进入下一步；
        - 否 -> 下一步；
    - 否 -> 判断[是否自身进化](#是否自身进化)：
      - 是 -> 下一步（自身优化场景，跳过复杂度迁移）；
      - 否 -> 评估内容复杂度是否需要迁移至 references/（评估标准：引用文件数量 > 10、分支逻辑深度 > 5 层、Workflow 步骤 > 20 步，满足任一即视为复杂）：
        - 是 -> 通过 AskUserQuestion 提供选项，阻塞等待用户选择：
          - 执行迁移 -> 迁移内容至 references/ 并更新链接，执行后进入下一步；
          - 保持现状 -> 跳过迁移，执行后进入下一步；
        - 否 -> 下一步；
  - 判断[是否自身进化](#是否自身进化)：
    - 是 -> 下一步（自身优化场景，跳过目录添加建议）；
    - 否 -> 判断是否需要添加 `scripts/`、`tests/`、`assets/` 或 `schemas/` 目录：
      - 是 -> 建议引入对应目录（技能包含确定性操作、会被重复生成的代码、需显式错误处理的逻辑、需测试用例、有静态资源需存放、或需跨 Skill 数据传递），执行后进入下一步；
      - 否 -> 下一步；
5. **参考文档拆分** — 拆分 REFERENCE.md 为 references/ 下多个文件；
  - 判断 `REFERENCE.md` 是否存在：
    - 是 -> 通过 AskUserQuestion 提供选项，阻塞等待用户选择：
      - [动态选项，由 AI 根据 REFERENCE.md 内容领域生成拆分方案] -> 自动拆分，执行后进入下一步；
      - 跳过 -> 执行 References 同步检查后进入步骤 6（复核检查）；
      - 手动指定拆分方案 -> 等待用户指定，执行后进入下一步；
    - 否 -> 执行 References 同步检查后进入步骤 6（复核检查）；
  - 按领域拆分为 `references/` 下多个文件：
    - 是否存在同名文件：
      - 是 -> 通过 AskUserQuestion 提供选项，阻塞等待用户选择：
        - 覆盖 -> 以新内容覆盖同名文件，执行后进入下一步；
        - 合并 -> 合并新内容到同名文件，执行后进入下一步；
        - 跳过 -> 保留同名文件不变，执行后进入下一步；
      - 否 -> 下一步；
  - 拆分后更新 SKILL.md 中所有指向原文件的链接；
  - 拆分后逐段对比原文，确认无内容丢失（如括号内示例、注意事项等细节未被遗漏，最多重试 2 次）：
    - 有丢失 -> 补充丢失内容后返回对比操作（重新执行"拆分后逐段对比原文"），重试计数器加 1；
    - 无丢失 -> 删除根目录下的 REFERENCE.md，下一步；
    - 超限仍有丢失（重试 2 次后仍有丢失）-> 记录失败项，通过 AskUserQuestion 提供选项，阻塞等待用户选择：
      - 保留 REFERENCE.md（内容可能存在遗漏）-> 保留原文件，执行后进入下一步；
      - 确认删除（风险自担）-> 删除 REFERENCE.md，执行后进入下一步；
6. **复核检查** — 对照 [Review List](#review-list)，确认优化结果；
  - 判断 Review List 是否有内容：
    - 否 -> 直接进入下一步（成果输出）；
    - 是 -> 下一步；
  - 依次判断 [Review List](#review-list) 中每个检查项，是否通过（须**逐项输出**全部检查项结果，不得以"其余项已通过"等缩写形式跳过）：
    - 是 -> 继续下一个检查项；
    - 否 -> 记录失败检查项（基于 "复核检查示例" 显示输出内容），继续下一个检查项；
  - 验证 `## References` 节与 `references/` 目录同步性（优化后）：
    - 同步 -> 下一步；
    - 不同步 -> 尝试自动修复 References 节（最多 3 次）：
      - 修复后验证通过 -> 下一步；
      - 3 次修复后仍不同步 -> 记录失败检查项，执行后进入下一步；
  - 判断是否有任一检查失败：
    - 是 -> 按 Rules 中“防御规范”执行；
    - 否 -> 进入下一步（成果输出）；
7. **成果输出** — 输出优化总结，告知完成；
  - 输出结构化摘要（具体格式请参考“成果输出示例”）；
  - 仅列出本次有变化的维度（如行数对比、目录补齐、参考文档拆分等），无变化的维度不输出；
  - 告知优化完成；

## Rules

- <a id="元数据规范"></a>**元数据规范**
  - description 须遵循格式：第一句说明技能能做什么，第二句说明触发条件（“当...时使用”），使用第三人称，不超过 1024 字符；
  - 仅校验 `name` 和 `description` 两个字段；其他非常规 frontmatter 字段（如 `disable-model-invocation`）保持原样，不做任何处理；

- **结构规范**
  - 标准结构目录仅适用于目标 SKILL 的 SKILL.md 文件本身，不影响目录下其他文件；
  - REFERENCE.md 应该移动到 `references` 文件夹下，并拆散为多个文件；

- **内容规范**
  - [引用层次](#引用层次)不超过一层：SKILL.md 可直接链接 `references/` 下文件；`references/` 下文件不应再链接外部资源；
  - 当 `references/` 下规范文件间发生冲突时，展开格式优先于压缩规则（workflow-standard.md 的树形分支和交互范式均不被 text-optimization.md 规则重新压缩）；
  - 路径匹配时避免子串匹配方式，推荐使用精确匹配（如目录名或完整路径），防止误判相似名称；
  - Review List 引用外部验证清单时，应指向文件的 `## 验证清单` 节或子分组锚点；引用 `#验证清单` 时须确保 Workflow 中有“逐项读取”显式指令；
  - Rules 节中所有 `#锚点` 引用必须指向存在的 `##` 标题节点或 `<a id="">` 标签锚点；
  - Skill 所有文件中使用的领域术语，必须在 `## Definitions` 中有明确定义；引用的术语与 Definitions 中的定义名称必须完全一致（大小写敏感）；
  - 每个 `## Definitions` 下的条目（如 `- **模板标准目录**`）**必须**有对应的可解析锚点（`<a id="">` 标签），以便全文档通过 `[]()` 精确引用；
  - 正文中引用 Definitions 术语时，须使用锚点级链接（如 `[模板标准目录](#模板标准目录)`），不得使用无链接的裸文本；

- **行为规范**
  - 删除任何文件（除已拆分的 REFERENCE.md 外），都要通过交互式提问向使用者进行提问；
  - 编辑范围限定：仅应调整目标 SKILL 的 `SKILL.md` 和 `references/` 目录下的文件；
    - 例外：移动或删除根目录下的 `REFERENCE.md`（未拆分时）或 `template.md` -> 需通过 AskUserQuestion 提供选项，阻塞等待用户选择：
      - 确认 -> 执行操作，执行后进入下一步；
      - 取消 -> 跳过，直接进入当前步骤后续操作；
  - 拆分 REFERENCE.md 后，必须逐段对比原文，确认无内容丢失（如括号内示例、注意事项等细节未被遗漏）；
  - 禁止以纯文本追问形式向用户咨询问题，**必须**使用 `AskUserQuestion` 工具；
  - 选项流向标注规则参照 [Workflow 编写规范](references/workflow-standard.md#选项流向标注)，“跳过”类选项须显式注明跳转目标；
  - Workflow 中相邻步骤的职责应互斥，每个步骤应有明确且唯一的意图，避免描述重叠；
  - Workflow 步骤结构须遵循 [Workflow 编写规范](references/workflow-standard.md)，缺失/错误由步骤 2 自动补全；
  - 自动补全[Secure 步骤](#secure-步骤)时应先检查整个 Workflow 中是否已存在同名步骤，存在则移动而非新增，避免重复插入；
  - Workflow 中引用 `## 验证清单` 时，AI 须**逐项读取**每个检查项并逐一确认；违反上述规范应在复核检查中标记未通过；
  - 引用外部验证清单时，AI 须先将被引用文件的完整内容载入内存；

- **防御规范**
  - 每个涉及文件移动、拆分或删除的操作，必须同步处理由该操作引发的副作用（如更新链接引用、修复相对路径）；
  - 若执行过程中检测到不可恢复的错误（文件写入失败、链接更新后校验死链无法修复），使用[原始内容副本](#原始内容副本)恢复目标文件（回滚仅恢复 SKILL.md，references/ 下新建文件需手动清理；仅在当前会话内检测到的错误中生效），并告知用户恢复结果；
  - 复核检查完成后，若存在失败项，判断是否为不可恢复错误（文件写入失败、死链无法修复等）：
    - **可恢复错误** -> 记录失败检查项到优化报告，标记**复核检查未通过**，终止流程；
    - **不可恢复错误** -> 使用步骤 0 的[原始内容副本](#原始内容副本)回滚 SKILL.md（references/ 下新建文件需手动清理），告知用户后终止流程；

- **验证规范**
  - Concern 分离原则参照 [rules-standard](references/rules-standard.md)；
  - Workflow 中定义的[退出路径](#退出路径)和回滚行为必须在 Rules 和 Review List 中有对应验证项；
  - Workflow 步骤应显式覆盖 Rules 中约束性规则引用的执行逻辑；
  - 多元素迭代处理：参照 [Workflow 编写规范](references/workflow-standard.md#循环与迭代)；
  - Examples 内容必须自洽于 Rules；
  - Examples 必须包含复核检查示例；
  - 对话交互示例仅聚焦步骤 0-5；
