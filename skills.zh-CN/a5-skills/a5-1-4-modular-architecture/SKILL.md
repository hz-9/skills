# NestJS 模块化架构与动态模块

## Overview

当 AI 在 NestJS 项目中遇到模块配置相关代码时，自动执行以下工作：审查 @Module 装饰器的配置完整性，检查模块导入导出关系的合理性，评估动态模块和全局模块的使用场景是否恰当，并提供改进建议。

## Definitions

- <a id="目标代码"></a>**目标代码**：当前对话中 NestJS 的模块类代码（以 @Module 装饰器标识的类）或文件路径。
- <a id="模块元数据"></a>**模块元数据**：@Module 装饰器的四个属性 —— imports、controllers、providers、exports。
- <a id="动态模块"></a>**动态模块**：可通过静态方法（forRoot、forFeature、register 等）在运行时动态配置的模块，使用 ConfigurableModuleBuilder 模式创建。
- <a id="全局模块"></a>**全局模块**：使用 @Global() 装饰器标记、自动向所有模块暴露其提供者的模块。
- <a id="是否分析完成"></a>**是否分析完成**：标记对目标代码的分析是否已得出完整结果。

## Prerequisites

- NestJS 项目环境；
- 包含 @Module 声明或模块配置的代码文件可访问；
- 了解模块化设计的基本原则。

## Workflow

0. **前置检查** — 确保目标代码和运行环境可达；
   - 判断目标代码是否存在且可读取：
     - 是 -> 下一步；
     - 否 -> 提示用户提供目标代码或文件路径，阻塞等待用户输入；
   - 初始化全局变量 [是否分析完成](#是否分析完成)：
     - 判断代码是否完整可解析：
       - 满足 -> 初始化变量为 true；
       - 不满足 -> 初始化变量为 false；

1. **分析模块代码** — 读取并理解模块配置和依赖关系；
   - 读取目标模块代码，识别以下核心要素：
     - @Module 装饰器的四个属性（imports、controllers、providers、exports）配置；
     - 是否使用了 @Global() 全局装饰器；
     - 是否为动态模块（静态方法 forRoot、register、forFeature）；
     - 模块间的导入导出关系链；

2. **逐项审查** — 对照审查清单检查模块配置质量；
   - 依次判断以下审查项是否通过：
     - imports 中导入的模块是否是必需的（避免导入未被使用的模块增加启动开销）：
       - 通过 -> 记录通过，继续下一个审查项；
       - 未通过 -> 记录“导入了未使用的模块，建议移除”，继续；
     - exports 是否将供其他模块使用的提供者全部导出：
       - 通过 -> 记录通过，继续下一个审查项；
       - 未通过 -> 记录“部分提供者未导出，其他模块无法注入”，继续；
     - @Global() 的使用是否合理（避免滥用全局模块导致隐式耦合）：
       - 通过 -> 记录通过，继续下一个审查项；
       - 未通过 -> 记录“建议谨慎使用 @Global()，考虑通过导入共享模块代替”，继续；
     - 动态模块是否使用了正确的命名约定（forRoot 用于根模块注册、forFeature 用于功能模块注册、register 用于单次注册）：
       - 通过 -> 记录通过，继续下一个审查项；
       - 未通过 -> 记录“动态模块静态方法命名未遵循约定”，继续；
     - 是否存在循环模块引用（ModuleA 导入 ModuleB，ModuleB 导入 ModuleA）：
       - 通过 -> 记录通过，继续下一个审查项；
       - 未通过 -> 记录“检测到循环模块引用，建议使用 forwardRef()”，继续；
     - 模块职责是否单一（避免单个模块包含跨多个领域的提供者）：
       - 通过 -> 记录通过，继续下一个审查项；
       - 未通过 -> 记录“模块职责过宽，建议拆分为多个领域模块”，继续；
   - 判断是否有任何问题记录：
     - 是 -> 汇总问题列表，进入下一步；
     - 否 -> 直接进入步骤 4（复核检查）；

3. **提供修改建议** — 对发现的问题给出具体修复方案；
   - 依次对每个问题提供修复建议，包括正反对比示例；
   - 通过 AskUserQuestion 提供选项，阻塞等待用户选择：
     - 采纳全部建议 -> 生成修正后的代码，进入步骤 4；
     - 逐条确认 -> 逐条由用户决定采纳或忽略，全部确认后进入步骤 4；
     - 仅查看不修改 -> 不影响代码，直接进入步骤 4；

4. **复核检查** — 对照 [Review List](#review-list)，确认执行结果；
   - 判断 Review List 是否有内容：
     - 否 -> 直接进入下一步（成果输出）；
     - 是 -> 下一步；
   - 依次判断 [Review List](#review-list) 中每个检查项，是否通过：
     - 基于“复核检查示例”来显示输出内容；
   - 有任一检查失败，终止流程；
   - 全部通过后，进入下一步；

5. **成果输出** — 输出审查摘要，告知完成；
   - 输出结构化摘要表格（审查文件 / 审查项数 / 通过 / 发现问题 / 风险等级 / 已采纳建议）；
   - 告知审查完成；

## Rules

- **内容规范**
  - description 须遵循格式：第一句说明审查内容，第二句说明触发条件，使用第三人称，不超过 1024 字符；
  - 审查项须引用 NestJS 官方文档关于模块的标准；
  - SKILL.md 不超过 300 行；超过时拆分到 references/ 目录；
  - 引用层次不超过一层；
  - 保持术语一致，Definitions 中已定义术语在正文中通过锚点链接引用；
  - 删除任何时效性信息；

- **行为规范**
  - 审查时仅输出问题摘要和定位，不直接修改代码，除非用户通过 AskUserQuestion 明确要求；
  - 所有涉及代码修改的交互环节，必须使用 AskUserQuestion 工具；
  - 模块职责拆分建议标注为架构建议而非代码缺陷；

- **防御规范**
  - 如目标代码为空或不可读，直接报告并终止；
  - 动态模块命名约定的推荐标注为社区惯例而非官方硬性要求；

- **验证规范**
  - Examples 内容必须自洽于 Rules；
  - Examples 必须包含复核检查示例；
  - 对话交互示例仅聚焦步骤 0~3（不含复核检查和成果输出）；

## Examples

### 对话交互示例

**示例：用户请求审查模块架构设计**

```markdown
用户 > 帮我看看这个 AppModule 的配置有没有问题
AI   > 检测到用户需要 NestJS 模块化架构审查，触发 nestjs-1-4-modular-architecture 技能
AI   > 正在分析模块配置...

审查结果：
- 🟩 imports 包含 DatabaseModule、AuthModule、UserModule
- 🟥 DatabaseModule 中使用了 @Global()，仅 AuthModule 和 UserModule 使用
- 🟩 exports 中正确导出了共享提供者
- 🟥 UserModule 同时导入并提供了 EmailService（建议拆为 EmailModule）
- 🟩 无循环模块引用

总评：2 个问题需要关注
用户 > 具体的修改建议是什么？
AI   > 1. DatabaseModule 的 @Global() 可移除，改为由 AppModule 显式导出
      2. EmailService 的使用者提取为独立的 EmailModule
用户 > 先看看就好，暂时不修改
```
