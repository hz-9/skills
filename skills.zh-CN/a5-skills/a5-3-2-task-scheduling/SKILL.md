# NestJS 异步任务与定时调度

## Overview

当 AI 在 NestJS 项目中遇到定时任务或异步任务相关代码时，自动执行以下工作：审查 @nestjs/schedule 装饰器的使用规范，检查 Cron 表达式和间隔配置的正确性，评估任务并发控制和错误处理策略，并提供改进建议。

## Definitions

- <a id="目标代码"></a>**目标代码**：当前对话中 NestJS 的定时任务类、Cron 方法或任务调度配置代码。
- <a id="Cron 装饰器"></a>**Cron 装饰器**：@nestjs/schedule 提供的 @Cron、@Interval、@Timeout 装饰器，分别对应 Cron 表达式、定时间隔和一次性延迟执行。
- <a id="ScheduleModule"></a>**ScheduleModule**：@nestjs/schedule 的根模块，使用 ScheduleModule.forRoot() 在 AppModule 中注册。
- <a id="是否分析完成"></a>**是否分析完成**：标记对目标代码的分析是否已得出完整结果。

## Prerequisites

- NestJS 项目环境（包含 @nestjs/schedule 依赖）；
- 定时任务类或调度配置代码可访问；
- 了解 Cron 表达式的基本语法。

## Workflow

0. **前置检查** — 确保目标代码和运行环境可达；
   - 判断目标代码是否存在且可读取：
     - 是 -> 下一步；
     - 否 -> 提示用户提供目标代码或文件路径，阻塞等待用户输入；
   - 初始化全局变量 [是否分析完成](#是否分析完成)：
     - 判断代码是否完整可解析：
       - 满足 -> 初始化变量为 true；
       - 不满足 -> 初始化变量为 false；

1. **分析任务代码** — 读取并理解定时任务和调度配置；
   - 读取目标代码，识别以下核心要素：
     - ScheduleModule.forRoot() 是否在根模块中注册；
     - @Cron、@Interval、@Timeout 装饰器的参数配置；
     - 任务方法的异常处理逻辑；

2. **逐项审查** — 对照审查清单检查任务调度代码质量；
   - 依次判断以下审查项是否通过：
     - Cron 表达式是否正确（六位或七位格式，各部分取值在有效范围内）：
       - 通过 -> 记录通过，继续下一个审查项；
       - 未通过 -> 记录“Cron 表达式格式可能有误，请检查各部分取值范围”，继续；
     - 长时间运行的任务是否添加了防止重叠执行的配置（@Cron 的 name 和 preventOverlap）：
       - 通过 -> 记录通过，继续下一个审查项；
       - 未通过 -> 记录“长时间任务建议启用 preventOverlap 避免重叠执行”，继续；
     - 任务方法是否包含异常处理逻辑（@Cron 方法内部异常默认被吞没）：
       - 通过 -> 记录通过，继续下一个审查项；
       - 未通过 -> 记录“定时任务中的异常默认被吞没，建议添加 try/catch 并记录日志”，继续；
     - ScheduleModule 是否仅在需要调度任务的模块中导入（避免不必要的全局注册）：
       - 通过 -> 记录通过，继续下一个审查项；
       - 未通过 -> 记录“ScheduleModule 仅需在使用调度功能的模块中导入”，继续；
     - 是否在测试环境中禁用了定时任务（避免测试时触发真实任务）：
       - 通过 -> 记录通过，继续下一个审查项；
       - 未通过 -> 记录“建议在测试环境中禁用 ScheduleModule”，继续；
   - 判断是否有任何问题记录：
     - 是 -> 汇总问题列表，进入下一步；
     - 否 -> 直接进入步骤 4（复核检查）；

3. **提供修改建议** — 对发现的问题给出具体修复方案；
   - 依次对每个问题提供修复建议；
   - 通过 AskUserQuestion 提供选项，阻塞等待用户选择：
     - 采纳全部建议 -> 生成修正后的代码，进入步骤 4；
     - 逐条确认 -> 逐条由用户决定采纳或忽略，全部确认后进入步骤 4；
     - 仅查看不修改 -> 不影响代码，直接进入步骤 4；

4. **复核检查** — 对照 [Review List](#review-list)，确认执行结果；
   - 判断 Review List 是否有内容：
     - 否 -> 直接进入下一步（成果输出）；
     - 是 -> 下一步；
   - 依次判断 [Review List](#review-list) 中每个检查项，是否通过：
     - 基于“复核检查示例”来显示输出内容；
   - 有任一检查失败，终止流程；
   - 全部通过后，进入下一步；

5. **成果输出** — 输出审查摘要，告知完成；
   - 输出结构化摘要表格（审查文件 / 审查项数 / 通过 / 发现问题 / 风险等级 / 已采纳建议）；
   - 告知审查完成；

## Rules

- **内容规范**
  - description 须遵循格式：第一句说明审查内容，第二句说明触发条件，使用第三人称，不超过 1024 字符；
  - 审查项须引用 NestJS 官方任务调度文档；
  - SKILL.md 不超过 300 行；超过时拆分到 references/ 目录；
  - 引用层次不超过一层；
  - 保持术语一致，Definitions 中已定义术语在正文中通过锚点链接引用；
  - 删除任何时效性信息；

- **行为规范**
  - 审查时仅输出问题摘要和定位，不直接修改代码，除非用户通过 AskUserQuestion 明确要求；
  - 所有涉及代码修改的交互环节，必须使用 AskUserQuestion 工具；
  - preventOverlap 和测试环境禁用标注为推荐而非强制；

- **防御规范**
  - 如目标代码为空或不可读，直接报告并终止；
  - Cron 表达式仅在明显错误时报告，复杂表达式标注为需手动验证；

- **验证规范**
  - Examples 内容必须自洽于 Rules；
  - Examples 必须包含复核检查示例；
  - 对话交互示例仅聚焦步骤 0~3（不含复核检查和成果输出）；

### 对话交互示例

**示例：用户请求审查定时任务配置**

```markdown

AI   > 正在分析定时任务代码...

审查结果：
- 🟩 ScheduleModule.forRoot() 已正确注册
- 🟥 Cron 表达式 '0 0 * * *' 只有五位（缺少秒位）
- 🟥 syncData 方法未捕捉异常，失败时不会记录日志
- 🟩 preventOverlap 已启用
- 🟥 测试环境中未禁用定时任务

总评：3 个问题需要关注
用户 > 帮我修复异常处理和测试禁用问题
AI   > 添加 try/catch 日志和测试环境判断。是否应用修改？

```
