# NestJS 自定义装饰器

## Overview

当 AI 在 NestJS 项目中遇到自定义装饰器相关代码时，自动执行以下工作：审查 createParamDecorator 的使用方式，检查自定义装饰器与元数据反射（Reflector）的配合，评估装饰器组合的合理性，并提供改进建议。

## Definitions

- <a id="目标代码"></a>**目标代码**：当前对话中 NestJS 的自定义装饰器实现代码或使用自定义装饰器的控制器代码。
- <a id="参数装饰器"></a>**参数装饰器**：使用 createParamDecorator 创建的自定义参数装饰器，用于提取和转换请求中的特定数据。
- <a id="元数据装饰器"></a>**元数据装饰器**：使用 @SetMetadata 或 Reflector 创建的自定义装饰器，用于在控制器或方法上设置元数据。
- <a id="是否分析完成"></a>**是否分析完成**：标记对目标代码的分析是否已得出完整结果。

## Prerequisites

- NestJS 项目环境；
- 自定义装饰器代码文件可访问；
- 了解装饰器模式和 reflect-metadata 的基本概念。

## Workflow

0. **前置检查** — 确保目标代码和运行环境可达；
   - 判断目标代码是否存在且可读取：
     - 是 -> 下一步；
     - 否 -> 提示用户提供目标代码或文件路径，阻塞等待用户输入；
   - 初始化全局变量 [是否分析完成](#是否分析完成)：
     - 判断代码是否完整可解析：
       - 满足 -> 初始化变量为 true；
       - 不满足 -> 初始化变量为 false；

1. **分析装饰器代码** — 读取并理解自定义装饰器实现；
   - 读取目标代码，识别以下核心要素：
     - 装饰器的类型（参数装饰器 / 方法装饰器 / 类装饰器）；
     - createParamDecorator 的工厂函数实现；
     - 与 Reflector 配合使用的元数据读取逻辑；
     - 多个装饰器的组合使用方式；

2. **逐项审查** — 对照审查清单检查装饰器代码质量；
   - 依次判断以下审查项是否通过：
     - 参数装饰器的 createParamDecorator 工厂函数是否正确处理了数据提取逻辑：
       - 通过 -> 记录通过，继续下一个审查项；
       - 未通过 -> 记录“参数装饰器的数据处理逻辑不完整”，继续；
     - 自定义装饰器是否通过 SetMetadata 和 Reflector 配合使用（元数据驱动）：
       - 通过 -> 记录通过，继续下一个审查项；
       - 未通过 -> 记录“建议使用 SetMetadata + Reflector 实现元数据驱动的装饰器”，继续；
     - 装饰器组合是否保持了可读性（避免过度嵌套导致难以理解）：
       - 通过 -> 记录通过，继续下一个审查项；
       - 未通过 -> 记录“装饰器组合过于复杂，建议简化或合并”，继续；
     - 自定义装饰器是否有 TypeScript 类型声明（提供良好的 IDE 提示）：
       - 通过 -> 记录通过，继续下一个审查项；
       - 未通过 -> 记录“自定义装饰器缺少返回类型声明”，继续；
     - 装饰器的 data 参数是否提供了默认值（无参数传入时的降级行为）：
       - 通过 -> 记录通过，继续下一个审查项；
       - 未通过 -> 记录“装饰器的 data 参数未处理 undefined 的情况”，继续；
   - 判断是否有任何问题记录：
     - 是 -> 汇总问题列表，进入下一步；
     - 否 -> 直接进入步骤 4（复核检查）；

3. **提供修改建议** — 对发现的问题给出具体修复方案；
   - 依次对每个问题提供修复建议；
   - 通过 AskUserQuestion 提供选项，阻塞等待用户选择：
     - 采纳全部建议 -> 生成修正后的代码，进入步骤 4；
     - 逐条确认 -> 逐条由用户决定采纳或忽略，全部确认后进入步骤 4；
     - 仅查看不修改 -> 不影响代码，直接进入步骤 4；

4. **复核检查** — 对照 [Review List](#review-list)，确认执行结果；
   - 判断 Review List 是否有内容：
     - 否 -> 直接进入下一步（成果输出）；
     - 是 -> 下一步；
   - 依次判断 [Review List](#review-list) 中每个检查项，是否通过：
     - 基于“复核检查示例”来显示输出内容；
   - 有任一检查失败，终止流程；
   - 全部通过后，进入下一步；

5. **成果输出** — 输出审查摘要，告知完成；
   - 输出结构化摘要表格（审查文件 / 审查项数 / 通过 / 发现问题 / 风险等级 / 已采纳建议）；
   - 告知审查完成；

## Rules

- **内容规范**
  - description 须遵循格式：第一句说明审查内容，第二句说明触发条件，使用第三人称，不超过 1024 字符；
  - 审查项须引用 NestJS 官方自定义装饰器文档；
  - SKILL.md 不超过 300 行；超过时拆分到 references/ 目录；
  - 引用层次不超过一层；
  - 保持术语一致，Definitions 中已定义术语在正文中通过锚点链接引用；
  - 删除任何时效性信息；

- **行为规范**
  - 审查时仅输出问题摘要和定位，不直接修改代码，除非用户通过 AskUserQuestion 明确要求；
  - 所有涉及代码修改的交互环节，必须使用 AskUserQuestion 工具；
  - 装饰器组合方式标注为代码风格建议而非强制；

- **防御规范**
  - 如目标代码为空或不可读，直接报告并终止；
  - 类型声明缺失仅在有明确接口上下文时报告；

- **验证规范**
  - Examples 内容必须自洽于 Rules；
  - Examples 必须包含复核检查示例；
  - 对话交互示例仅聚焦步骤 0~3（不含复核检查和成果输出）；
