# NestJS 自动化测试驱动

## Overview

当 AI 在 NestJS 项目中遇到测试相关代码时，自动执行以下工作：审查测试模块配置和测试用例的结构，检查单元测试和 E2E 测试的覆盖完整性，评估 Mock 策略和测试隔离性，并提供改进建议。

## Definitions

- <a id="目标代码"></a>**目标代码**：当前对话中 NestJS 项目的测试代码（.spec.ts 或 .e2e-spec.ts 文件）。
- <a id="TestModule"></a>**TestModule**：由 @nestjs/testing 的 Test.createTestingModule 创建的测试模块，用于模拟 NestJS 的 DI 容器。
- <a id="E2E 测试"></a>**E2E 测试**：端到端测试，使用 supertest 创建 HTTP 客户端，测试完整的请求/响应链路。
- <a id="是否分析完成"></a>**是否分析完成**：标记对目标代码的分析是否已得出完整结果。

## Prerequisites

- NestJS 项目环境（包含 @nestjs/testing、jest、supertest 依赖）；
- 测试规范文件可访问；
- 了解 Jest 测试框架和测试分层原则。

## Workflow

0. **前置检查** — 确保目标代码和运行环境可达；
   - 判断目标代码是否存在且可读取：
     - 是 -> 下一步；
     - 否 -> 提示用户提供目标代码或文件路径，阻塞等待用户输入；
   - 初始化全局变量 [是否分析完成](#是否分析完成)：
     - 判断代码是否完整可解析：
       - 满足 -> 初始化变量为 true；
       - 不满足 -> 初始化变量为 false；

1. **分析测试代码** — 读取并理解测试结构和策略；
   - 读取目标代码，识别以下核心要素：
     - Test.createTestingModule 的编译配置和导入的模块；
     - 单元测试中的 Mock 策略（jest.mock、useValue 自定义提供者）；
     - 测试用例的命名规范和结构（describe / it 嵌套）；
     - E2E 测试中的 app 创建和 supertest 使用；
     - 测试覆盖的边界情况（正常路径、错误路径、边界值）；

2. **逐项审查** — 对照审查清单检查测试代码质量；
   - 依次判断以下审查项是否通过：
     - 测试是否使用了 Test.createTestingModule 而非直接实例化（利用 DI 容器）：
       - 通过 -> 记录通过，继续下一个审查项；
       - 未通过 -> 记录“建议使用 Test.createTestingModule 构建测试模块”，继续；
     - 外部依赖（数据库、HTTP 服务）是否被 Mock 替代（避免测试依赖外部服务）：
       - 通过 -> 记录通过，继续下一个审查项；
       - 未通过 -> 记录“外部依赖未被 Mock，测试可能因外部服务不稳定而失败”，继续；
     - 测试用例是否覆盖了成功路径和失败路径（异常、无效输入等边界情况）：
       - 通过 -> 记录通过，继续下一个审查项；
       - 未通过 -> 记录“错误路径未覆盖，建议补充异常场景测试”，继续；
     - E2E 测试中是否在 beforeAll 中创建 app 并在 afterAll 中关闭（确保测试隔离）：
       - 通过 -> 记录通过，继续下一个审查项；
       - 未通过 -> 记录“app 生命周期管理不完整，可能导致测试间相互影响”，继续；
     - 测试用例是否相互独立（不依赖其他测试的执行顺序）：
       - 通过 -> 记录通过，继续下一个审查项；
       - 未通过 -> 记录“测试用例间存在状态共享，建议使用 beforeEach 重置状态”，继续；
     - 是否使用了 describe 对测试按功能/模块分组（提高可读性和可维护性）：
       - 通过 -> 记录通过，继续下一个审查项；
       - 未通过 -> 记录“测试用例缺少 describe 分组，建议按功能组织”，继续；
   - 判断是否有任何问题记录：
     - 是 -> 汇总问题列表，进入下一步；
     - 否 -> 直接进入步骤 4（复核检查）；

3. **提供修改建议** — 对发现的问题给出具体修复方案；
   - 依次对每个问题提供修复建议；
   - 通过 AskUserQuestion 提供选项，阻塞等待用户选择：
     - 采纳全部建议 -> 生成修正后的代码，进入步骤 4；
     - 逐条确认 -> 逐条由用户决定采纳或忽略，全部确认后进入步骤 4；
     - 仅查看不修改 -> 不影响代码，直接进入步骤 4；

4. **复核检查** — 对照 [Review List](#review-list)，确认执行结果；
   - 判断 Review List 是否有内容：
     - 否 -> 直接进入下一步（成果输出）；
     - 是 -> 下一步；
   - 依次判断 [Review List](#review-list) 中每个检查项，是否通过：
     - 基于“复核检查示例”来显示输出内容；
   - 有任一检查失败，终止流程；
   - 全部通过后，进入下一步；

5. **成果输出** — 输出审查摘要，告知完成；
   - 输出结构化摘要表格（审查文件 / 审查项数 / 通过 / 发现问题 / 风险等级 / 已采纳建议）；
   - 告知审查完成；

## Rules

- **内容规范**
  - description 须遵循格式：第一句说明审查内容，第二句说明触发条件，使用第三人称，不超过 1024 字符；
  - 审查项须引用 NestJS 测试最佳实践；
  - SKILL.md 不超过 300 行；超过时拆分到 references/ 目录；
  - 引用层次不超过一层；
  - 保持术语一致，Definitions 中已定义术语在正文中通过锚点链接引用；
  - 删除任何时效性信息；

- **行为规范**
  - 审查时仅输出问题摘要和定位，不直接修改代码，除非用户通过 AskUserQuestion 明确要求；
  - 所有涉及代码修改的交互环节，必须使用 AskUserQuestion 工具；
  - Mock 策略和测试覆盖标注为建议而非强制，视项目测试策略而定；

- **防御规范**
  - 如目标代码为空或不可读，直接报告并终止；
  - 测试覆盖率仅检查可见的测试用例结构，不运行实际测试；

- **验证规范**
  - Examples 内容必须自洽于 Rules；
  - Examples 必须包含复核检查示例；
  - 对话交互示例仅聚焦步骤 0~3（不含复核检查和成果输出）；

### 对话交互示例

**示例：用户请求审查测试实现**

```markdown
用户 > 帮我检查这个服务的单元测试
AI   > 检测到用户需要 NestJS 测试审查，触发 nestjs-5-3-testing 技能
AI   > 正在分析测试代码...

审查结果：
- 🟩 使用 Test.createTestingModule 构建测试模块
- 🟥 数据库 Repository 未 Mock，测试直接依赖数据库
- 🟩 成功路径覆盖完整
- 🟥 未覆盖异常输入和错误返回场景
- 🟩 afterAll 中正确关闭 app

总评：2 个问题需要关注
用户 > 帮我添加 Mock 和异常测试
AI   > 使用 useValue 提供 Mock Repository 并添加异常场景测试。是否应用修改？
用户 > 全部采纳
```
