# NestJS 权限控制与守卫

## Overview

当 AI 在 NestJS 项目中遇到守卫（Guard）相关代码时，自动执行以下工作：审查守卫的实现和注册方式，检查权限检查逻辑的完整性，评估角色控制和自定义访问策略的设计，识别授权绕过风险，并提供改进建议。

## Definitions

- <a id="目标代码"></a>**目标代码**：当前对话中 NestJS 的守卫实现代码、权限元数据定义或自定义访问策略代码。
- <a id="守卫接口"></a>**守卫接口**：所有守卫需实现的 CanActivate 接口，包含 canActivate(context: ExecutionContext) 方法。
- <a id="元数据反射"></a>**元数据反射**：通过 @SetMetadata 装饰器和 Reflector 类实现的守卫间数据传输机制。
- <a id="是否分析完成"></a>**是否分析完成**：标记对目标代码的分析是否已得出完整结果。

## Prerequisites

- NestJS 项目环境；
- 守卫实现或权限控制配置代码可访问；
- 了解守卫在 NestJS 请求处理链中的执行顺序（中间件之后、拦截器之前）。

## Workflow

0. **前置检查** — 确保目标代码和运行环境可达；
   - 判断目标代码是否存在且可读取：
     - 是 -> 下一步；
     - 否 -> 提示用户提供目标代码或文件路径，阻塞等待用户输入；
   - 初始化全局变量 [是否分析完成](#是否分析完成)：
     - 判断代码是否完整可解析：
       - 满足 -> 初始化变量为 true；
       - 不满足 -> 初始化变量为 false；

1. **分析守卫代码** — 读取并理解守卫实现和权限控制逻辑；
   - 读取目标代码，识别以下核心要素：
     - 守卫类的实现方式（实现 CanActivate 接口）；
     - canActivate 方法的实现逻辑和返回值；
     - @SetMetadata 定义的权限元数据；
     - Reflector 在守卫中读取元数据的方式；
     - 守卫的注册方式（全局 / 控制器 / 方法级）；

2. **逐项审查** — 对照审查清单检查守卫代码质量；
   - 依次判断以下审查项是否通过：
     - canActivate 方法是否在所有分支返回了布尔值（true 允许、false 拒绝）：
       - 通过 -> 记录通过，继续下一个审查项；
       - 未通过 -> 记录“canActivate 方法必须返回布尔值，缺少返回值的分支会抛出异常”，继续；
     - 是否使用了 Reflector 读取控制器或方法的元数据：
       - 通过 -> 记录通过，继续下一个审查项；
       - 未通过 -> 记录“建议使用 Reflector 读取 @SetMetadata 定义的权限元数据”，继续；
     - 守卫是否执行了实际的认证/授权检查（而非仅返回 true）：
       - 通过 -> 记录通过，继续下一个审查项；
       - 未通过 -> 记录“守卫返回始终 true 可能导致授权检查缺失”，继续；
     - 角色或权限元数据是否在正确的层级（控制器或方法）上设置：
       - 通过 -> 记录通过，继续下一个审查项；
       - 未通过 -> 记录“元数据设置在错误层级，可能导致守卫无法正确读取”，继续；
     - 守卫是否正确处理了异常（如 token 过期、无权限时的错误处理）：
       - 通过 -> 记录通过，继续下一个审查项；
       - 未通过 -> 记录“守卫中异常未被捕获，建议抛出特定 HttpException”，继续；
   - 判断是否有任何问题记录：
     - 是 -> 汇总问题列表，进入下一步；
     - 否 -> 直接进入步骤 4（复核检查）；

3. **提供修改建议** — 对发现的问题给出具体修复方案；
   - 依次对每个问题提供修复建议；
   - 通过 AskUserQuestion 提供选项，阻塞等待用户选择：
     - 采纳全部建议 -> 生成修正后的代码，进入步骤 4；
     - 逐条确认 -> 逐条由用户决定采纳或忽略，全部确认后进入步骤 4；
     - 仅查看不修改 -> 不影响代码，直接进入步骤 4；

4. **复核检查** — 对照 [Review List](#review-list)，确认执行结果；
   - 判断 Review List 是否有内容：
     - 否 -> 直接进入下一步（成果输出）；
     - 是 -> 下一步；
   - 依次判断 [Review List](#review-list) 中每个检查项，是否通过：
     - 基于“复核检查示例”来显示输出内容；
   - 有任一检查失败，终止流程；
   - 全部通过后，进入下一步；

5. **成果输出** — 输出审查摘要，告知完成；
   - 输出结构化摘要表格（审查文件 / 审查项数 / 通过 / 发现问题 / 风险等级 / 已采纳建议）；
   - 告知审查完成；

## Rules

- **内容规范**
  - description 须遵循格式：第一句说明审查内容，第二句说明触发条件，使用第三人称，不超过 1024 字符；
  - 审查项须引用 NestJS 官方关于守卫的文档；
  - SKILL.md 不超过 300 行；超过时拆分到 references/ 目录；
  - 引用层次不超过一层；
  - 保持术语一致，Definitions 中已定义术语在正文中通过锚点链接引用；
  - 删除任何时效性信息；

- **行为规范**
  - 审查时仅输出问题摘要和定位，不直接修改代码，除非用户通过 AskUserQuestion 明确要求；
  - 所有涉及代码修改的交互环节，必须使用 AskUserQuestion 工具；
  - 权限策略建议标注为架构建议而非强制；

- **防御规范**
  - 如目标代码为空或不可读，直接报告并终止；
  - 守卫始终返回 true 仅在明确可确认时报告为问题（如无认证中间件前置）；

- **验证规范**
  - Examples 内容必须自洽于 Rules；
  - Examples 必须包含复核检查示例；
  - 对话交互示例仅聚焦步骤 0~3（不含复核检查和成果输出）；

### 对话交互示例

**示例：用户请求审查守卫实现**

AI   > 正在分析守卫代码...

审查结果：
- 🟩 实现了 CanActivate 接口
- 🟥 错误处理分支缺少 return false，可能未拒绝未授权访问

- 🟩 守卫注册在控制器级别，作用范围明确
- 🟥 未处理 token 过期场景，直接抛 Unknown 异常

总评：2 个问题需要关注
用户 > 请修复错误分支的返回值问题
