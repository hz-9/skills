# NestJS 高速缓存策略与 Redis

## Overview

当 AI 在 NestJS 项目中遇到缓存相关代码时，自动执行以下工作：审查缓存模块配置和缓存管理器设置，检查 CacheInterceptor 的使用场景，评估缓存失效策略和数据一致性机制，识别常见缓存陷阱，并提供改进建议。

## Definitions

- <a id="目标代码"></a>**目标代码**：当前对话中 NestJS 的缓存模块配置、缓存拦截器使用或自定义缓存管理代码。
- <a id="缓存管理器"></a>**缓存管理器**：NestJS 的 CacheManager，支持内存缓存和 Redis 等外部存储，提供 get、set、del 等操作。
- <a id="缓存拦截器"></a>**缓存拦截器**：NestJS 内置的 CacheInterceptor，自动缓存 GET 端点响应，需在 @Module 中启用。
- <a id="是否分析完成"></a>**是否分析完成**：标记对目标代码的分析是否已得出完整结果。

## Prerequisites

- NestJS 项目环境（包含 @nestjs/cache-manager 和 cache-manager 依赖）；
- 缓存配置或相关代码可访问；
- 了解缓存策略的基本概念（TTL、缓存穿透、缓存雪崩）。

## Workflow

0. **前置检查** — 确保目标代码和运行环境可达；
   - 判断目标代码是否存在且可读取：
     - 是 -> 下一步；
     - 否 -> 提示用户提供目标代码或文件路径，阻塞等待用户输入；
   - 初始化全局变量 [是否分析完成](#是否分析完成)：
     - 判断代码是否完整可解析：
       - 满足 -> 初始化变量为 true；
       - 不满足 -> 初始化变量为 false；

1. **分析缓存代码** — 读取并理解缓存配置和策略；
   - 读取目标代码，识别以下核心要素：
     - CacheModule 的注册方式（全局 / 模块级）和配置（ttl、max）；
     - Redis 存储的集成配置（store 类型、连接参数）；
     - CacheInterceptor 的应用范围（全局 / 控制器 / 方法级）；
     - 手动缓存操作（cacheManager.get、set、del）；
     - 缓存键的命名策略；

2. **逐项审查** — 对照审查清单检查缓存实现质量；
   - 依次判断以下审查项是否通过：
     - 缓存 TTL 值是否合理（避免过长导致数据陈旧或过短导致缓存失效）：
       - 通过 -> 记录通过，继续下一个审查项；
       - 未通过 -> 记录“TTL 设置不合理，建议根据数据更新频率调整”，继续；
     - 写操作（POST / PUT / DELETE）后是否执行了缓存失效（避免脏数据）：
       - 通过 -> 记录通过，继续下一个审查项；
       - 未通过 -> 记录“写操作后未清除相关缓存，可能导致返回陈旧数据”，继续；
     - Redis 配置是否正确（使用 @nestjs/cache-manager 的 store 配置）：
       - 通过 -> 记录通过，继续下一个审查项；
       - 未通过 -> 记录“Redis 存储配置不完整，请检查 store 驱动和连接参数”，继续；
     - 缓存键是否具有唯一性且包含足够的上下文信息（避免键冲突）：
       - 通过 -> 记录通过，继续下一个审查项；
       - 未通过 -> 记录“缓存键定义过于简单可能导致冲突，建议包含模块名前缀”，继续；
     - CacheInterceptor 是否仅应用于 GET 端点（非幂等方法不应自动缓存）：
       - 通过 -> 记录通过，继续下一个审查项；
       - 未通过 -> 记录“CacheInterceptor 应用于非 GET 方法可能导致意外行为”，继续；
   - 判断是否有任何问题记录：
     - 是 -> 汇总问题列表，进入下一步；
     - 否 -> 直接进入步骤 4（复核检查）；

3. **提供修改建议** — 对发现的问题给出具体修复方案；
   - 依次对每个问题提供修复建议；
   - 通过 AskUserQuestion 提供选项，阻塞等待用户选择：
     - 采纳全部建议 -> 生成修正后的代码，进入步骤 4；
     - 逐条确认 -> 逐条由用户决定采纳或忽略，全部确认后进入步骤 4；
     - 仅查看不修改 -> 不影响代码，直接进入步骤 4；

4. **复核检查** — 对照 [Review List](#review-list)，确认执行结果；
   - 判断 Review List 是否有内容：
     - 否 -> 直接进入下一步（成果输出）；
     - 是 -> 下一步；
   - 依次判断 [Review List](#review-list) 中每个检查项，是否通过：
     - 基于“复核检查示例”来显示输出内容；
   - 有任一检查失败，终止流程；
   - 全部通过后，进入下一步；

5. **成果输出** — 输出审查摘要，告知完成；
   - 输出结构化摘要表格（审查文件 / 审查项数 / 通过 / 发现问题 / 风险等级 / 已采纳建议）；
   - 告知审查完成；

## Rules

- **内容规范**
  - description 须遵循格式：第一句说明审查内容，第二句说明触发条件，使用第三人称，不超过 1024 字符；
  - 审查项须引用 NestJS 和 cache-manager 官方文档；
  - SKILL.md 不超过 300 行；超过时拆分到 references/ 目录；
  - 引用层次不超过一层；
  - 保持术语一致，Definitions 中已定义术语在正文中通过锚点链接引用；
  - 删除任何时效性信息；

- **行为规范**
  - 审查时仅输出问题摘要和定位，不直接修改代码，除非用户通过 AskUserQuestion 明确要求；
  - 所有涉及代码修改的交互环节，必须使用 AskUserQuestion 工具；
  - TTL 值建议基于数据更新频率提供参考而非固定值；

- **防御规范**
  - 如目标代码为空或不可读，直接报告并终止；
  - 缓存失效必要性仅在写操作可确认时报告，不确定的关联数据标注为需手动确认；

- **验证规范**
  - Examples 内容必须自洽于 Rules；
  - Examples 必须包含复核检查示例；
  - 对话交互示例仅聚焦步骤 0~3（不含复核检查和成果输出）；
