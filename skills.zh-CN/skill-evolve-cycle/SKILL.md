# Skill Evolve Cycle

## Overview

对目标 SKILL 执行多轮“优化->审查->修复->合并->反哺”的循环进化过程。由 skill-evolve（优化阶段）和 Code Review（审查阶段）交替驱动，支持对 skill-evolve 自身的进化。

## Definitions

- <a id="技能原始仓库"></a>**技能原始仓库**：远程仓库地址为 `git@github.com:hz-9/skills.git` 或 `https://github.com/hz-9/skills` 的仓库。当当前工作空间属于该仓库时，某些步骤（反哺）需额外执行。

- <a id="收敛"></a>**收敛**：某层级循环在当前迭代中发现 0 个新问题，满足退出条件的状态。

- <a id="是否技能仓库"></a>**是否技能仓库**：标记当前工作空间是否属于[技能原始仓库](#技能原始仓库)。由步骤 0.1 通过 `git remote get-url origin` 匹配判定并初始化，全流程缓存复用。

### 收敛条件

| 层级             | 收敛条件                                                                                                     | 判定位置                                                                                                              |
| ---------------- | ------------------------------------------------------------------------------------------------------------ | --------------------------------------------------------------------------------------------------------------------- |
| **小循环A 收敛** | 当前 skill-evolve 优化发现 **0 个新问题**                                                                    | 目标 SKILL 已对齐标准结构，无待优化项                                                                                 |
| **小循环B 收敛** | 当前 Code Review 发现 **0 个新问题**                                                                         | 分两种情况：① 首次从全新角度审查无问题 -> 目标 SKILL 已稳定；② 修复后返回全新审查无问题 -> 本轮修复正确、未引入新问题 |
| **大循环收敛**   | ① 本轮小循环A 首轮发现0个问题 **且** ② 大循环步骤 2 中小循环B 首次 Code Review 发现 **0 个新问题** | 两条均满足时说明目标 SKILL 已稳定，无待修复问题                                                                       |

> **新问题定义**：相对本轮操作的上一次等价判定结果而言，本次新发现且之前未被记录的问题。同一问题被修复后再次出现的属于复现，计入新问题。

### 严重度分级

| 级别   | 定义                                           | 示例                                        |
| ------ | ---------------------------------------------- | ------------------------------------------- |
| **高** | 结构性问题或语义错误，会导致用户误解或功能失效 | 缺少关键 section、描述格式错误、死链接      |
| **中** | 可维护性问题或非关键性遗漏                     | 标点不规范、术语不一致、超过 300 行但未拆分 |
| **低** | 优化建议，不影响正确性                         | 冗余内容、推荐的但非强制的最佳实践          |

## Prerequisites

- 目标 SKILL.md 文件已存在
- skill-evolve 技能可用
- `docs/skill-evolve-cycle/` 目录存在于当前工作空间（不存在则创建）；UTC 时间子目录由 Step 0 自动创建

## Workflow

0. **前置检查** — 检查工作空间状态、远程仓库归属和目标身份；
   0.1 尝试执行 `git remote get-url origin`：
     - 成功且匹配 [技能原始仓库](#技能原始仓库) -> 标记 `is_skill_repo = true`；下一步；
     - 成功但不匹配 -> 标记 `is_skill_repo = false`；下一步；
     - 命令失败（非 Git 仓库或无 remote） -> 标记 `is_skill_repo = false`，报告警告"无法获取远程仓库信息，假定为[非技能原始仓库](#技能原始仓库)"；下一步；
   0.2 （预留：自身进化场景判断，当前版本无特殊处理，直接下一步）
   0.3 获取当前 UTC 时间并创建 `docs/skill-evolve-cycle/{UTC时间}/` 目录（父目录不存在则一并创建），本轮运行所有报告均保存至此目录：
     - 创建成功 -> 下一步；
     - 创建失败 -> 报告"目录创建失败"，通过 AskUserQuestion 提供选项，阻塞等待用户选择：
       - 重试 -> 再次尝试创建，执行后返回 0.3；
       - 终止 -> 终止流程；

1. **小循环A（优化）** — 使用 skill-evolve 优化目标 SKILL 直至[收敛](#收敛)；
    1.1 **skill-evolve 优化** — 使用 skill-evolve 优化目标 SKILL（注意：每次运行完整的优化流程和步骤，不要进行快速检查；skill-evolve 自身流程已包含调查和修复）
        - **发现问题** -> 继续 1.2
        - **发现 0 个问题** -> **小循环A 收敛**，进入大循环步骤 2（小循环B）
        - **执行失败（返回错误）** -> 生成错误报告 `cycle{轮次}-A-error.md`，标记"小循环A-执行异常"，进入大循环步骤 3
    1.2 **生成迭代报告** — 生成 `cycle{轮次}-A-{迭代}.md` 保存到 `docs/skill-evolve-cycle/{UTC时间}/`
    1.3 **返回 1.1**，重新运行优化

    > 每轮小循环A 最大 30 次迭代。达到上限仍未收敛时，强制退出小循环A，进入大循环步骤 2，在报告中标注"小循环A-未完全收敛"。

2. **小循环B（审查）** — 通过 Code Review 审查目标 SKILL 并修复问题直至[收敛](#收敛)；

    按以下步骤运行直至[收敛](#收敛)。

    > **Code Review 执行方式（硬规则，不可绕过）**：以下规则为强制执行，覆盖 2.1 首次审查和 2.4 回归审查，不允许以任何理由绕过、降级或变通：
    > 1. **Agent 类型锁定**：必须使用 CodeReview Agent 执行审查，**严禁**因"变更量小"、"问题已修复完毕"、"当前为回归审查"等理由切换为 GeneralPurpose Agent。CodeReview Agent 不可用时，视为环境故障，流程终止并在报告中标注"CodeReview Agent 不可用"。
    > 2. **并行数硬约束**：必须并行派出 **恰好 3 个** CodeReview Agent，分别独立执行三个视角的审查（Completeness/完整性；Correctness/正确性；Impact/影响），**缺一不可**。不允许因"变更量小"、"问题数已减少"、"当前为回归审查"等任何理由缩减并行数。
    > 3. **禁止流程降级**：不允许"快速审查"、"轻量审查"、"仅验证修复完整性"等任何跳过或简化完整审查流程的行为。每个审查轮次必须严格顺序执行完整步骤链。

    2.1 **Code Review** — 以全新的视角全面审查目标 SKILL（并非对于上一轮小循环B 的回归审查），记录所有发现的问题
        - **发现 0 个问题** -> **小循环B 收敛**，等待大循环步骤 3 判定去向
        - **发现问题** -> 继续 2.2
    2.2 **记录问题** — 将每个问题记录到本轮报告中，标注严重度（高/中/低）
    2.3 **修复问题** — 逐个修复所有发现的问题
        - 每次修复后生成修复报告 `cycle{轮次}-B-{迭代}-fix{修复轮次}.md`，记录修复方式和结果
        - 修复轮次自增，进入 2.4
    2.4 **回归审查与循环控制** — 回归审查已修复的问题：
        - 检查以下内容，及其他问题
            - 修复是否正确
            - 修复是否引入了新问题
        - **发现任何问题** -> 回到 2.2，将新问题记录到本轮报告后进入 2.3 继续修复（修复轮次+1）
        - **发现 0 个问题** -> 继续 2.5
    2.5 **生成迭代报告** — 汇总本轮所有修复记录，生成 `cycle{轮次}-B-{迭代}.md` 保存到 `docs/skill-evolve-cycle/{UTC时间}/`
    2.6 **返回 2.1** — 重置修复轮次计数，从全新角度重新审查

    > 每轮小循环B 最大 30 次外层循环迭代（2.1->...->返回 2.1 计为一次）。内层修复循环（2.2->2.3->2.4->2.2）单独计数，**每次外层迭代中，修复轮次最多 30 次**。内层修复循环超限时强制退出小循环B，**进入大循环步骤 3**（报告标注**"小循环B-修复卡死"**）。外层循环达到 30 次上限仍未收敛时强制退出小循环B，**进入大循环步骤 3**（报告标注**"小循环B-未完全收敛"**）。

3. **大循环[收敛](#收敛)判定** — 检查是否满足大循环[收敛](#收敛)条件（小循环A首轮0问题且小循环B首轮0问题）；
  3.1 是否本轮大循环的小循环 A 首轮发现0个问题，小循环 B 首轮审查发现 0个问题
    - 是 -> 继续步骤 7
    - 否 -> 继续步骤 4
4. **合并与整理** — 汇总问题并按类型统计，按流程分岔路由；

   汇总本轮小循环A 和 小循环B 发现的所有问题，按类型统计。

   **流程分岔**：
   - `is_skill_repo = true` -> 继续步骤 5
   - `is_skill_repo = false` -> 跳过步骤 5，继续步骤 6

5. **反哺到 skill-evolve** — 将审查经验按反哺路由规范分流写入；

   ⚠️ 本步骤仅当当前仓库为[技能原始仓库](#技能原始仓库)时执行。会修改 skill-evolve 的 SKILL.md 或 references/ 下文件。

   5.1 依据 Rules 中"反哺路由规范"执行前置判断，判断是否存在有效反哺内容：
    - 是 -> 查询 [内容边界标准](../skill-evolve/references/content-boundary.md) 确定去向，通过 AskUserQuestion 提供选项，阻塞等待用户选择：
      - [动态选项，由 AI 根据 Code Review 内容 + 前置判断 + 所有权表生成] -> 执行对应反哺项（写入正确的 reference 文件或 SKILL.md），执行后进入下一步；
      - 按用户确认的反哺项执行更新；
      - 反哺后同步更新 SKILL.md 的 `## References` 节，执行后进入下一步；
    - 否 -> 自动跳过，进入下一步；
   5.2 是否进行了至少一项反哺：
    - 是 -> 重新读取 Skill Evolve 的 SKILL.md 及 references/ 文件（使新反哺规则生效），继续步骤 1
    - 否 -> 下一步

6. **本轮成果输出** — 汇总报告并返回小循环A 开始下一轮大循环；

   汇总本轮所有报告到 `docs/skill-evolve-cycle/{UTC时间}/` 目录，生成 `cycle{轮次}-summary.md`。返回步骤 1，开始下一轮大循环（轮次+1）。

   > 每轮大循环最大 30 次外层迭代（步骤 1->6 计为一次）。达到上限仍未触发大循环收敛时，终止全部流程，汇总所有轮次报告，标记最终状态为"大循环-未完全收敛"。

7. **复核检查** — 对照 [Review List](#review-list)，确认优化结果；
   - 判断 Review List 是否有内容：
     - 否 -> 直接进入下一步（成果输出）；
     - 是 -> 下一步；
   - 依次判断 [Review List](#review-list) 中每个检查项，是否通过：
     - 是 -> 继续下一个检查项；
     - 否 -> 记录失败检查项（基于"复核检查示例"显示输出内容）；
   - 判断是否有任一检查失败：
     - 是 -> 汇总本轮所有报告到 `docs/skill-evolve-cycle/{UTC时间}/`，标记最终状态为**复核检查未通过**（注明失败的检查项），终止全部流程；
     - 否 -> 全部通过，进入下一步（成果输出）；
   - 全部通过后，进入下一步（成果输出）；

8. **成果输出** — 输出优化总结，告知完成；
   - 输出最终报告（具体格式请参考"成果输出示例"），保存到 `docs/skill-evolve-cycle/{UTC时间}/final-summary.md`，标记 **大循环收敛**，终止全部流程；
   - 输出结构化摘要（具体格式请参考"成果输出示例"）；
   - 告知优化完成；

## Rules

- **行为规范**
  - **自动持续运行**：大循环之间必须自动推进下一轮次，不得中断等待用户确认后手动触发。
  - **交互模式继承**：调用 skill-evolve 时，AI 遵循 skill-evolve Workflow 中定义的内在逻辑来判定是否需要触发 AskUserQuestion。若 skill-evolve 触发了 AskUserQuestion，则 skill-evolve-cycle 必选触发这个 AskUserQuestion，不允许通过 AI 自行决策；
  - **小循环收敛即推进**：小循环A 收敛后进入大循环步骤 2；小循环B 收敛后由大循环步骤 3 判定去向（大循环收敛或进入步骤 4）。均不允许中断；
  - **Agent 类型锁定规则**：整个流程中，小循环B 的 Code Review 阶段（含 2.1 首次审查和 2.4 回归审查）**必须**使用 CodeReview Agent 执行审查。严禁以任何理由（包括但不限于"变更量小"、"问题已修复完毕"、"当前轮次仅需快速确认"、"回归审查无需新 Agent"）切换为 GeneralPurpose Agent 或其他 Agent 类型。若环境客观上不支持 CodeReview Agent，流程终止并在报告中标注"CodeReview Agent 不可用"，不得降级替代后继续执行。
  - **并行数硬约束规则**：每轮 Code Review（含 2.1 首次审查和 2.4 回归审查）必须并行派出 **恰好 3 个** CodeReview Agent，分别独立覆盖 Completeness、Correctness、Impact 三个视角。不允许因任何理由（包括但不限于"变更量小"、"问题数已减少"、"当前轮次为回归审查"、"平行 Agent 资源紧张"）缩减并行数。3 个 Agent 的审查结果均需独立记录到本轮报告中，不可合并为单个 Agent 的输出。
  - **禁止流程降级规则**：各步骤必须严格按照 Workflow 定义的顺序、内容和执行方式执行，不允许以任何理由跳过、简化、合并或加速任何步骤。特别地，小循环B 中不允许：
    - 以"所有问题已修复完毕"为由跳过 2.1 完整审查，直接进入"快速验证完整性"
    - 以"变更量小"、"问题数已减少"为由将 3 个并行 Agent 降级为 1 个 Agent 或 AI 自行审查
    - 以"当前为回归审查"为由省略 2.4 回归审查中的三个视角全量检查
    - 将 2.4 回归审查与 2.1 后续轮次审查合并执行
- **内容规范**
  - **报告持久化**：每个小循环、每轮大循环的合并报告，均保存到 `docs/skill-evolve-cycle/{UTC时间}/` 目录。命名规则：
    - 小循环A 报告 `cycle{轮次}-A-{迭代}.md`（迭代指每次从步骤 1 重新运行时生成）
    - 小循环A 错误报告 `cycle{轮次}-A-error.md`（skill-evolve 执行失败时生成）
    - 小循环B 修复报告 `cycle{轮次}-B-{迭代}-fix{修复轮次}.md`（每次修复后生成，修复轮次从 1 开始自增）
    - 小循环B 迭代报告 `cycle{轮次}-B-{迭代}.md`（迭代指外层循环次数，每次从步骤 1 重新审查时生成）
    - 大循环合并报告 `cycle{轮次}-summary.md`
    - 最终汇总报告 `final-summary.md`
  - **禁止引入流程外标签**：流程执行和报告中**严禁**出现 Workflow 未定义的流程标签或状态标识，包括但不限于"最终 Code Review"、"最后一次审查"、"最终确认"、"完结审查"、"最终轮次"。所有审查轮次统一使用 Workflow 定义的命名和标识（如 `cycle{N}-B-{迭代}.md`、`cycle{N}-B-{迭代}-fix{修复轮次}.md`）。收敛判定仅依据 Workflow 中定义的收敛条件和状态标签。
- **验证规范**
  - **[技能原始仓库](#技能原始仓库)判断**：通过 `git remote get-url origin` 的输出匹配已知仓库地址，匹配后缓存结果 `is_skill_repo`，全流程复用；
  - **反哺路由规范**：反哺阶段（步骤 5）按以下流程决定审查经验是否反哺及去向：
    - **前置判断**：
      - 是否为本次 Code Review 特有、不可复现的噪声 → 跳过；
      - 是否已在 SKILL.md 或 references/ 下存在（含同语义表述） → 更新已有内容，跳过反哺；
      - 未存在且值得反哺 → 继续路由判断；
    - **路由判断**：查询 [内容边界标准](../skill-evolve/references/content-boundary.md)：
      - 匹配 reference 文件 → 写入该文件，不追加到 SKILL.md；
      - 匹配"保留在 SKILL.md" → 写入 SKILL.md 对应节；
      - 无匹配 → 通过 AskUserQuestion 询问用户；
    - 反哺后同步更新 SKILL.md 的 `## References` 节；

## Examples

### 对话交互示例

```markdown
用户 > 对 skills.zh-CN/skill-evolve/SKILL.md 执行循环进化

AI > 开始 skill-evolve-cycle：

      Cycle 1 — 发现问题，未收敛：
      步骤 0  前置检查…is_skill_repo = true。
      步骤 1  小循环A（优化）…优化自身，发现 3 个问题，修复后收敛。
      步骤 2  小循环B（审查）…发现 2 个问题，修复后收敛。
      步骤 3  大循环收敛判定…小循环A首轮发现问题 → 进入步骤 4。
      步骤 4  合并与整理…is_skill_repo = true → 进入步骤 5。
      步骤 5  反哺到 skill-evolve…已同步 3 条新规则。
      步骤 6  本轮输出…已保存 cycle1-summary.md，进入下一轮。

      Cycle 2 — 收敛轮次：
      步骤 1  小循环A（优化）…全面优化，发现 0 个新问题。
      步骤 2  小循环B（审查）…全新角度，发现 0 个新问题。
      步骤 3  大循环收敛判定…小循环A首轮0问题 + 小循环B首轮0问题 → 是，进入步骤 7。
      步骤 4-6 此处被跳过（收敛无需执行合并/反哺/输出）
      步骤 7-8  复核检查与成果输出通过。
```

### 复核检查示例

```markdown
AI > 准备进行复核检查，对照 Review List 逐项检查：
```
