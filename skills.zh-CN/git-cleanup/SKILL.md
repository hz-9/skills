# Git Cleanup

## Overview

系统化清理 Git 仓库中废弃的 Worktree、分支和 Tag。按"全面扫描 -> 一次确认 -> 统一删除"流程操作，先一次性扫描全部类别（Worktree/分支/Tag），再通过一次确认让用户决策，最后统一执行删除，再经二次确认后统一删除远程引用。执行前自动创建备份。

## Definitions

- <a id="废弃分支"></a>**废弃分支**：符合以下任一条件的分支：
    - **已合并分支**：已被合并到主分支的本地分支；
    - **孤儿分支**：执行 `git fetch -p` 后，远程跟踪分支已不存在的本地分支（显示 `: gone]`）；
- <a id="废弃 Tag"></a>**废弃 Tag**：符合以下任一条件的 Tag：
    - **非版本 Tag**：不匹配语义化版本号模式（如 `v1.2.3`、`1.2.3`、`v1.2.3-beta.1`）的 Tag；
    - **孤儿 Tag**：指向的 commit 没有任何本地或远程分支引用的 Tag；
- <a id="废弃 Worktree"></a>**废弃 Worktree**：非当前 Worktree（用户当前所在工作目录之外的 Worktree）；
- <a id="当前 Worktree"></a>**当前 Worktree**：用户当前所在工作目录对应的 Worktree，自动跳过、不可删除；
- <a id="主分支"></a>**主分支**：仓库的主要开发分支（如 `main` 或 `master`），由 AI 自动检测后向用户确认；
- <a id="保护分支"></a>**保护分支**：主分支及 `develop` 分支，自动跳过、不可删除；
- <a id="当前检出分支"></a>**当前检出分支**：`HEAD` 指向的分支，自动跳过、不可删除。

## Prerequisites

- Git 2.0+；
- 当前在 Git 仓库目录中（可通过 `git rev-parse --git-dir` 验证）；
- `jq`（JSON 处理工具，check-env.sh / scan.sh / delete.sh / setup.sh 等脚本依赖此命令）
- 用户有对远程仓库的推送权限（如需执行远程删除）。

## Workflow

0. **前置检查** — 确保环境已就绪；
  0.1 执行环境检查（`bash scripts/check-env.sh`）：
    - 判断脚本是否执行成功：
      - 成功 -> 解析 JSON，判断是否包含 "error" 字段：
        - 是 -> 报告脚本错误（如缺少 jq 依赖），终止流程；
        - 否 -> 逐项报告检查结果：
      - "in-git-repo" 未通过 -> 报告"当前不在 Git 仓库中"，终止流程；
      - "git-version" 未通过 -> 提示升级 Git 至 2.0+，终止流程；
      - "fetch-prune" 未通过 -> 记录失败原因，继续执行；
      - "has-remote" -> 存储结果（步骤 4.1 远程删除时使用）；
    - 全部检查通过 -> 进入步骤 0.2；
  0.2 执行环境准备（`bash scripts/setup.sh`）：
    - 判断脚本是否执行成功：
      - 成功 -> 解析 JSON，判断是否包含 "error" 字段：
        - 是 -> 报告脚本错误（如缺少 jq 依赖），终止流程；
        - 否 -> 读取以下信息：
      - main_branch_candidate -> 自动检测到的主分支名；
      - current_worktree -> 当前 Worktree 路径；
      - current_branch -> 当前检出分支；
      - backup_created / backup_path -> 备份状态和路径；
    - 判断备份是否创建成功：
      - 成功 -> 进入步骤 0.3；
      - 失败 -> 报告"备份创建失败"，终止流程；
  0.3 确认主分支 — 通过 AskUserQuestion 提供选项，阻塞等待用户选择：
    - 检测到主分支 -> 提供选项：
      - 确认主分支名称 -> 使用该名称，进入步骤 1；
      - 手动指定其他分支 -> 由用户输入分支名称，进入步骤 1；
    - 未检测到主分支 -> 提供选项：
      - 手动指定主分支名称 -> 由用户输入分支名称，进入步骤 1；

1. **全面扫描** — 执行扫描脚本，输出三个类别 JSON；
  1.1 执行扫描脚本（`bash scripts/scan.sh --main-branch <主分支>`）：
    - 判断脚本是否执行成功：
      - 成功 -> 解析 JSON，判断是否包含 "error" 字段：
        - 是 -> 报告脚本错误（如缺少 jq 依赖），终止流程；
        - 否 -> 获取 worktrees、branches、tags 三个数组；
      - 失败 -> 报告"扫描脚本执行异常"，终止流程；
  1.2 判断三个数组是否全部为空：
    - 是 -> 报告"未发现待清理的废弃引用"，终止流程；
    - 否 -> 进入步骤 2；

2. **分类确认** — 逐类展示扫描结果，分别确认删除；
  2.1 展示扫描结果 — 将三个 JSON 数组分别渲染为 Markdown 表格：
      - Worktree 表格（列：路径 / 关联分支 / 是否当前）；
      - 分支表格（列：分支名 / 类型 / 原因）；
      - Tag 表格（列：Tag 名 / 类型 / 原因）；
  2.2 确认 Worktree — 通过 AskUserQuestion 提供选项，阻塞等待用户选择：
      - 是，删除全部非当前 Worktree -> 标记全部为待删除，进入 2.3；
      - 选择部分删除 -> 提示用户输入路径（逗号分隔），确认后进入 2.3；
      - 跳过 -> 进入 2.3；
  2.3 确认分支 — 通过 AskUserQuestion 提供选项，阻塞等待用户选择：
      - 是，删除全部废弃分支 -> 标记全部为待删除，进入 2.4；
      - 选择部分删除 -> 提示用户输入分支名（逗号分隔），确认后进入 2.4；
      - 跳过 -> 进入 2.4；
  2.4 确认 Tag — 通过 AskUserQuestion 提供选项，阻塞等待用户选择：
      - 是，删除全部废弃 Tag -> 标记全部为待删除，进入 2.5；
      - 选择部分删除 -> 提示用户输入 Tag 名（逗号分隔），确认后进入 2.5；
      - 跳过 -> 进入 2.5；
  2.5 检查是否有标记为待删除的引用：
      - 是（存在待删除项） -> 进入步骤 3；
      - 否（全部跳过） -> 报告"未选择任何待删除项"，终止流程；

3. **统一执行** — 使用删除脚本执行所有已确认删除；
  3.1 执行删除脚本（`bash scripts/delete.sh --worktrees '<json>' --branches '<json>' --tags '<json>'`）：
    - 判断脚本是否执行成功：
      - 失败 -> 报告"删除脚本执行异常"，跳转至步骤 5（异常退出处理）；
      - 成功 -> 解析 JSON；
      - 传入数据结构示例：
        - `--worktrees '[{"path":"/path/to/wt"}]'`
        - `--branches '[{"name":"feature/old","type":"merged"},{"name":"fix/temp","type":"orphan"}]'`
        - `--tags '[{"name":"v0.1-alpha","type":"non-version"},{"name":"temp-tag","type":"orphan"}]'`
    - 解析返回的 JSON 结果数组，逐条输出删除结果（成功 / 失败）；
    - 当 Worktree 删除失败时，通过 AskUserQuestion 询问用户是否尝试强制删除（`git worktree remove --force <path>`）：
      - 是 -> 执行强制删除并输出结果；
      - 否 -> 跳过该 Worktree，继续处理后续删除；
  3.2 输出本地删除摘要表格（统计成功/失败数量），参见 <a id="本地删除摘要示例"></a>[本地删除摘要示例](#本地删除摘要示例)；
  3.3 进入步骤 4；

4. **远程删除** — 二次确认后执行远程删除；
  4.1 获取步骤 0.1 check-env 的 has-remote 结果，判断是否有本地删除：
      - 远程存在且有本地删除 -> 通过 AskUserQuestion 提供选项，阻塞等待用户选择：
        - 确认推送远程删除 -> 进入 4.2；
        - 仅保留本地删除 -> 进入 4.3；
      - 远程不存在或本地无删除 -> 报告"无变更需要同步到远程"，进入步骤 6；
  4.2 执行远程删除脚本（`bash scripts/remote-delete.sh --branches '<json>' --tags '<json>'`）：
    - 判断脚本是否执行成功：
      - 失败 -> 报告"远程删除脚本执行异常"，跳转至步骤 5（异常退出处理）；
      - 成功 -> 解析 JSON；
      - 传入数据结构示例：
        - `--branches '[{"name":"feature/old-login"}]'`
        - `--tags '[{"name":"temp-tag-123"}]'`
      - 解析返回的 JSON，逐条输出删除结果；
  4.3 输出最终清理报告（本地 + 远程删除统计）；

5. **异常退出处理** — 输出恢复指引，在任一删除步骤异常退出时跳转至此；
  5.1 判断检查步骤 0.2 setup.sh 输出的 backup_created 结果：
    - 是 -> 输出备份路径和恢复命令，参见 <a id="备份恢复示例"></a>[备份恢复示例](#备份恢复示例)；
    - 否 -> 报告"无可用备份，请检查仓库状态"；
  5.2 输出本次已完成的删除操作摘要（如有已删除的 Worktree/Branch/Tag 列表）；
  5.3 告知用户已安全终止，提供手动恢复建议和备份路径信息，终止流程。

6. **复核检查** — 对照 [Review List](#review-list)，确认执行结果；
  6.1 判断 Review List 是否有内容：
    - 否 -> 直接进入下一步（成果输出）；
    - 是 -> 下一步；
  6.2 依次判断 [Review List](#review-list) 中每个检查项，是否通过（基于"复核检查示例"显示输出内容）：
    - 是 -> 继续下一个检查项；
    - 否 -> 通过 AskUserQuestion 提供选项，阻塞等待用户选择：
      - 重试失败的步骤 -> 返回对应步骤重新执行；
      - 跳过失败项继续 -> 标记为未完成，进入成果输出；
      - 终止流程 -> 终止流程；
  6.3 全部通过后，进入下一步（成果输出）；

7. **成果输出** — 输出清理摘要、备份路径和回滚指引，告知完成；
  7.1 输出结构化摘要（包含本次清理的各维度统计数量）；
  7.2 输出备份路径和恢复命令（参见 [备份恢复示例](#备份恢复示例)）
  7.3 提示用户在回滚前确认当前工作区状态，以及备份路径的正确性；
  7.4 告知清理完成；

## Rules

- **元数据规范**
  - description 遵循格式：第一句说明技能能做什么，第二句说明触发条件（“当...时使用”），使用第三人称，不超过 1024 字符；
- **结构规范**
  - 标准目录仅适用于 SKILL.md 本身，不影响目录下其他文件；
- **内容规范**
  - 扫描结果和删除摘要均以表格形式呈现；
  - 表格列名清晰，包含“引用名 / 类型 / 原因 / 状态”等必要维度；
  - 最终清理报告包含各维度的统计数量（已删除 / 已跳过 / 失败等）；
- **行为规范**
  - 备份失败时不允许跳过，必须备份成功或终止流程；
  - [当前 Worktree](#当前-worktree)和[当前检出分支](#当前检出分支)自动跳过，不进入扫描列表；
  - [Worktree 绑定分支](#worktree-绑定分支)自动跳过，不进入删除候选列表；
  - [保护分支](#保护分支)（[主分支](#主分支) + `develop`）自动跳过，不进入扫描列表；
  - 远程删除必须经过二次确认，禁止在首次确认后直接推送删除；
  - 所有涉及用户抉择的交互环节（确认删除、选择操作方式、确认[主分支](#主分支)等），**必须**使用 `AskUserQuestion` 工具，禁止以纯文本追问代替；将问题与选项结构化传入 AskUserQuestion，每次调用不超过 4 个问题；
  - 先统一扫描所有类别，再统一执行删除，删除阶段按 Worktree → Branch → Tag 顺序执行；
  - 先删除本地引用，再删除远程引用（分两阶段）；
  - 整体完成后输出删除摘要，再进行远程删除二次确认；
  - 删除命令执行失败时，记录失败原因并继续执行后续删除，不中断流程；
- **防御规范**
  - 删除任何文件，都要通过交互式提问向用户进行确认；
- **验证规范**
  - 示例内容应与最新 Workflow 步骤一致；

### 复核检查示例

```markdown
AI > 进入复核检查，Review List 包含 16 个检查项，开始逐项验收：

**内容检查**
  - 🟩 所有用户确认删除的 Worktree 已成功删除
  - 🟩 所有用户确认删除的分支已成功删除
  - 🟩 所有用户确认删除的 Tag 已成功删除

  - 🟩 扫描结果以表格形式展示，列名清晰
  - 🟩 ...（其余检查项均通过）

**行为检查**
  - 🟩 执行前已创建 `cp -a` 备份，路径和命名正确
  - 🟩 已执行 `git fetch -p`，远程跟踪已修剪

  - 🟩 保护分支未被列入删除列表
  - 🟩 远程删除前经过二次确认
```
