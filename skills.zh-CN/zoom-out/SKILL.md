# Zoom Out

## Overview

当用户对某段代码不熟悉时，AI 触发 zoom-out 视角：拉远视角向上抽象一层，生成相关模块与调用者的全局地图，使用项目领域术语表词汇，帮助用户理解代码在整体架构中的位置。

## Definitions

- <a id="zoom-out"></a>**Zoom Out**：从当前代码片段拉远视角向上抽象，展示模块间关系、调用链和架构角色的全景视角。
- <a id="模块地图"></a>**模块地图**：列出当前文件/模块的上下游依赖、调用者和被调用者的结构化视图。

## Prerequisites

- 当前项目有可浏览的代码库，agent 可访问相关源文件。
- 项目存在领域术语表或约定俗成的命名规范更佳，非必须。

> **建议**：如项目已有 `CONTEXT.md` 或领域文档，zoom-out 将能据此使用更准确的领域术语组织模块地图。

## Workflow

0. **前置检查** — 确保 zoom-out 条件已满足；
    - 确保目标代码可访问：
      - 是 -> 下一步；
      - 否 -> 报告“目标代码不存在或不可访问”，终止流程；
    - 判断项目是否存在领域术语表：
      - 是 -> 加载术语表用于后续输出；
      - 否 -> 根据代码命名惯例推断术语；

1. **理解目标代码** — 分析用户指定的目标代码上下文；
    - 识别目标代码所属模块及其架构角色；

2. **分析调用关系** — 搜索并列出调用链；
    - 使用代码搜索工具查找该模块被哪些文件引用；
    - 使用 LSP 或代码导航功能获取调用链；
    - 标注模块分层位置（infrastructure / domain / application）；

3. **生成模块地图** — 使用项目领域术语组织输出（最大重试 3 次）；
    - （首次执行时初始化重试计数为 0）
    - 判断输出是否为结构化地图形式：
      - 是 -> 下一步；
      - 否 -> 重试计数加 1：
        - 未超过 3 次 -> 重新组织为地图形式，返回步骤 3 起始判断重新验证；
        - 已达 3 次 -> 报告“无法生成结构化地图”，终止流程；

4. **复核检查** — 对照 [Review List](#review-list)，确认输出内容；
    - （首次进入时初始化复核重试计数为 0）
    - 判断 Review List 是否有内容：
        - 否 -> 直接进入下一步（成果输出）；
        - 是 -> 下一步；
    - 依次判断 [Review List](#review-list) 中每个检查项，是否通过：
        - 基于“复核检查示例”来显示输出内容；
        - 有任一检查失败 -> 复核重试计数加 1：
          - 未超过 3 次 -> 重新生成模块地图，返回步骤 3；
          - 已达 3 次 -> 报告“复核检查未通过”，终止流程；
        - 全部通过 -> 下一步；

5. **成果输出** — 向用户呈现[模块地图](#模块地图)，说明目标代码如何融入全局；
    - 输出结构化摘要（所属模块、上游调用者数量、下游依赖数量等）；
    - 告知处理完成；

## Rules

- **内容规范**
  - 始终向上抽象一层，不深入到实现细节；
  - 输出以结构化的地图形式呈现，而非长篇描述。
- **术语规范**
  - 使用项目已有的领域术语，避免引入新术语；
  - 若项目无领域术语表，根据代码命名惯例推断术语；
- **验证规范**
  - Review List 检查项覆盖 Rules 中所有约束性规则，确保 Rules 约束的行为在 Review List 中有对应的输出质量验证（遵循 Concern 分离原则）；

## Examples

### 对话交互示例

**示例：AI 根据用户请求触发 zoom-out**

```markdown
用户 > 这段代码我不太熟悉，帮我 zoom out 一下

AI > 触发 zoom-out 视角，开始分析...

目标代码：UserAuthService.authenticate()
所属模块：auth（认证模块）

┌─────────────────────────────────────────┐
│              调用者                      │
│  LoginController (web/login)            │
│  ApiGateway (gateway/)                  │
│  SessionManager (core/session)          │
└──────────────┬──────────────────────────┘
               │
    UserAuthService.authenticate()
               │
┌──────────────┴──────────────────────────┐
│              依赖                       │
│  PasswordHasher (crypto/)               │
│  UserRepository (data/user)             │
│  TokenService (auth/token)              │
│  AuditLogger (core/logging)             │
└─────────────────────────────────────────┘

UserAuthService 位于 auth 模块，负责用户身份验证。
上游由 LoginController、ApiGateway 等入口调用；
下游依赖加密、数据访问和令牌服务。
```

### 复核检查示例

```markdown
AI > 进入复核检查，对照 Review List 逐项确认：

**内容完整性检查**
  - 🟩 输出包含所属模块及架构角色
  - 🟥 未列出上游调用者和下游依赖 — 调用链缺失
  - 🟩 调用者和依赖均附带了路径信息
**格式规范检查**
  - 🟩 使用了项目领域术语
  - 🟥 输出为长篇描述而非结构化地图形式

（AI 运行时将逐项输出全部检查项结果）

**！！！以下检查项未通过！！！**
  - 🟥 内容完整性：调用链信息缺失
  - 🟥 格式规范：输出非地图形式

终止流程，返回步骤 3 重新生成模块地图。
```

### 成果输出示例

```markdown
| 维度 | 说明 |
|------|------|
| 目标代码 | UserAuthService.authenticate() |
| 所属模块 | auth（认证模块） |
| 上游调用者 | 3 个 |
| 下游依赖 | 4 个 |
| Abstract | 向上抽象一层 |
| 输出形式 | 结构化模块地图 |
```

### 执行成功示例

```markdown
| 维度 | 说明 |
|------|------|
| 目标代码 | UserAuthService.authenticate() |
| 所属模块 | auth（认证模块） |
| 上游调用者 | 3 个（LoginController、ApiGateway、SessionManager） |
| 下游依赖 | 4 个（PasswordHasher、UserRepository、TokenService、AuditLogger） |
| 输出形式 | 结构化模块地图 |
```

## Review List

完成 zoom-out 后，验证以下内容：

- **内容完整性检查**
    - [ ] 输出包含目标代码所属模块及架构角色
    - [ ] 列出了上游调用者和下游依赖
    - [ ] 调用者和依赖均附带了路径/位置信息
- **格式规范检查**
    - [ ] 使用了项目领域术语，而非引入新术语
    - [ ] 当无项目领域术语表时，术语来源为代码命名惯例推断，而非自创新词
    - [ ] 输出为结构化地图形式，非长篇描述
    - [ ] 抽象层级正确：仅向上抽象一层，未深入到实现细节
