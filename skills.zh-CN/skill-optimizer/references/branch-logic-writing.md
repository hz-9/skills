# 分支逻辑写作规范

## 概述

规范 SKILL.md 中条件分支判断的写法，将平铺的 if-else 转为树形箭头格式（`是 -> / 否 ->`），确保每种分支结果对应明确、可执行的操作指令。每个分支动作行末必须以 `；` 结尾，作为分支结束标记；引入子条件或子操作列表的非终止分支行末以 `：` 结尾。

## 规则 1：分支逻辑树形化

条件分支必须以树形箭头格式呈现，每种结果（是/否）独占一行、缩进表达层级，避免将多个分支结果压缩在同一行。

> **标准格式**：
> ```markdown
> - 判断条件：
>     - 是 -> 执行动作 A；
>     - 否 -> 执行动作 B；
> ```

### 示例 1：简单二元判断

```markdown
**原文**

- 若缺失或不规范，交互式确认修复方案；

**优化后**

- 确认 `name` 是否存在：
    - 是 -> 下一步；
    - 否 -> 通过 AskUserQuestion 提供几个选项（文件夹名 / 基于 SKILL.md 整理出来的文件名），阻塞等待用户选择；
```

> **关键改进**：原文只有"若缺失"的隐含分支，AI 不知道满足条件时该做什么；优化后明确定义了"是→下一步"。

### 示例 2：嵌套分支

```markdown
**原文**

- 非模板标准目录的内容 → 迁移到 `references/` 下（若存在同名文件，交互式确认覆盖、合并或跳过）；

**优化后**

- 非模板标准目录的内容 → 迁移到 `references/` 下：
    - 是否存在同名文件：
        - 是 -> 通过 AskUserQuestion 提供选项（覆盖 / 合并 / 跳过），阻塞等待用户选择；
        - 否 -> 下一步；
```

> **关键改进**：原文将"迁移"和"同名文件处理"压缩在一行，AI 可能混淆执行顺序；优化后通过缩进表达了父子判断关系。

### 示例 3：存在性检测

```markdown
**原文**

- 若存在 `REFERENCE.md`，先展示拆分方案，确认后执行：

**优化后**

- 检测 `REFERENCE.md` 是否存在：
    - 是 -> 通过 AskUserQuestion 提供拆分选项（由 AI 根据 REFERENCE.md 内容生成），阻塞等待用户选择；
    - 否 -> 下一步；
```

> **关键改进**：原文缺少"不存在时"的行为定义；优化后补充了 `否 → 下一步`，消除 AI 的行为盲区。

## 规则 2：幂等守卫

对"是否已符合某规范"的判断，必须在「是」分支标注跳过理由，防止二次执行时重复触发优化。

> **标准格式**：
> ```markdown
> - 判断 [元素] 是否已使用 [规范] 标准范式：
>     - 是 -> 下一步（已合规，跳过优化）；
>     - 否 -> 基于 [规范] 进行优化；
> ```

### 示例：交互环节幂等检查

```markdown
**原文**

- 判断 `## Workflow` 中是否存在涉及用户抉择的交互环节
    - 是 -> 基于交互式操作写作规范进行优化
    - 否 -> 下一步

**优化后**

- 判断 `## Workflow` 中是否存在涉及用户抉择的交互环节：
    - 是 -> 判断是否已使用 `AskUserQuestion` 标准范式：
        - 是 -> 下一步（已合规，跳过优化）；
        - 否 -> 基于 [交互式操作写作规范] 进行优化；
    - 否 -> 下一步；
```

> **关键改进**：原版缺少"已合规→跳过"的二级判断，导致二次执行时会把已优化为正确格式的内容再次"优化"，可能造成格式变形。

## 规则 3：依次判断（集合迭代）

当需要对集合中每个元素逐一判断时，使用"依次判断...每个..."前缀，明确迭代范围。

> **标准格式**：
> ```markdown
> - 依次判断 [集合] 中每个 [元素]，是否已 [条件]：
>     - 是 -> 下一步（已合规，跳过优化）；
>     - 否 -> 基于 [规范] 进行优化；
> ```

### 示例：批量交互环节检查

```markdown
**原文**

- 判断 `## Workflow` 中是否存在涉及用户抉择的交互环节
    - 是 -> 基于交互式操作写作规范进行优化
    - 否 -> 下一步

**优化后**

- 依次判断 `## Workflow` 中每个存在涉及用户抉择的交互环节，是否已使用 `AskUserQuestion` 标准范式：
    - 是 -> 下一步（已合规，跳过优化）；
    - 否 -> 基于 [交互式操作写作规范] 进行优化；
```

> **关键改进**："依次判断...每个..."相比"判断...是否存在"，将"存在性检测"升级为"逐项合规检查"，AI 不会在第一个合规项后就停止检查。

## 分支动作类型

树形分支的 `是 ->` / `否 ->` 后可接以下动作类型：

| 类型 | 格式 | 含义 |
|------|------|------|
| 下一步 | `下一步；` | 继续当前步骤内的下一个判断，或进入下一步骤 |
| 终止流程 | `终止流程；` | 停止执行，不再继续后续步骤 |
| 跳过优化 | `下一步（已合规，跳过优化）；` | 当前项已合规，跳过并继续 |
| 多步动作块 | `迁移到 references/：` + 缩进子列表 | 执行一组有序子操作 |
| 引入建议 | `建议引入对应目录；` | 给出建议但不强制执行 |
| 阻塞交互 | `通过 AskUserQuestion 提供选项（...），阻塞等待用户选择；` | 暂停执行，通过 UI 组件向用户提供结构化选项并等待选择 |

### 示例：终止流程

```markdown
- 验证 target 是否存在：
    - 是 -> 下一步；
    - 否 -> 报告"目标不存在"，终止流程；
```

> **与"下一步"的区别**：`终止流程` 是流程级别的结束，后续所有步骤不再执行；`下一步` 仅结束当前判断，继续同一步骤内的后续操作。

## 危险场景分析

> **场景 1：缺少否定分支**
> *示例*：`若文件存在则读取内容` → 缺少"不存在时"的行为
> *问题*：AI 遇到文件不存在时可能报错或跳过，行为不可预测
> *正确*：明确 `否 -> 提示用户文件不存在并退出`
>
> **场景 2：多分支压缩**
> *示例*：`若 A 则 X，若 B 则 Y` → 两个独立条件写在同一行
> *问题*：AI 难以判断 A 和 B 是否互斥，可能同时执行 X 和 Y
> *正确*：拆分为两个独立的树形判断块
>
> **场景 3：层级混淆**
> *示例*：
> ```markdown
> - 若 A 成立则执行 B，若 C 成立则执行 D
> ```
> *问题*：C 是 A 的子条件还是同级条件？AI 无法判断
> *正确*：用缩进区分父子层级
>
> **场景 4：规范文件自身违反规范**
> *示例*：本文早期版本中示例 3 的 `- 否 -> 下一步` 曾缺少行末分号（现已修复），但标准格式和示例 1 均以 `；` 结尾
> *问题*：AI 以规范文件为基准学习格式时，会因示例不一致而产生随机行为（有时加分号有时不加）
> *正确*：规范文件自身必须通过其定义的验证清单，确保示例之间格式严格一致

## 常见反模式

### 反模式 1：隐式 else

只有"是"分支，缺少"否"分支。

> **错误**：`若 name 缺失，交互式确认` → 缺少"否"分支（name 存在时该如何？）
> **正确**：
> ```markdown
> - 判断 name 是否存在：
>     - 是 -> 下一步；
>     - 否 -> 通过 AskUserQuestion 提供确认选项（由 AI 根据上下文生成），阻塞等待用户选择；
> ```

### 反模式 2：内联嵌套

将嵌套条件写在括号或同一行中。

> **错误**：`迁移到 references/ 下（若存在同名文件，交互式确认覆盖、合并或跳过）`
> **正确**：用缩进层级展开：
> ```markdown
> - 迁移到 references/ 下：
>     - 是否存在同名文件：
>         - 是 -> 通过 AskUserQuestion 提供选项（覆盖 / 合并 / 跳过），阻塞等待用户选择；
>         - 否 -> 下一步；
> ```

### 反模式 3：箭头缺失

有分支内容但未标注 `是 ->` / `否 ->`。

> **错误**：
> ```markdown
> - 检查文件是否存在；
>     - 读取内容；
>     - 提示用户；
> ```
> **问题**：缩进不等于分支，"不存在时"还提示用户吗？
> **正确**：显式标注箭头

### 反模式 4：缺少幂等守卫

对"是否已符合规范"的判断不检查现有合规状态，直接把所有匹配项送入优化流程。

> **错误**：`判断 Workflow 中是否存在交互环节 → 是 -> 基于交互式操作写作规范进行优化`
> **问题**：已合规的内容被二次"优化"，可能导致格式变形或产生新问题
> **正确**：增加二级判断 `已使用 AskUserQuestion 标准范式？→ 是 -> 下一步（已合规，跳过优化）`

- [ ] 每个分支指向具体、可执行的操作指令
- [ ] 不存在括号内嵌套条件（反模式 2）
- [ ] 不存在隐式 else（反模式 1）
- [ ] 不存在多个独立条件写入同一行的情况（危险场景 2）

- [ ] 集合迭代使用"依次判断...每个..."前缀，非"判断...是否存在"
- [ ] 每个终止分支行末以 `；` 结尾；引入子条件或子操作列表的非终止分支行末以 `：` 结尾
