# 交互式操作写作规范

## 概述

规范 SKILL.md 中涉及用户抉择的交互环节的写法，确保 AI 正确使用 `AskUserQuestion` 工具进行阻塞式 UI 确认，而非以纯文本追问代替。

## 规则 1：Rules 中的交互式确认规则

`## Rules` 中必须包含使用 `AskUserQuestion` 的行为兜底。

**必须包含的规则**：

```markdown
- 所有涉及用户抉择的交互环节（确认修复方案、选择处理方式、确认删除/覆盖/合并等），**必须**使用 `AskUserQuestion` 工具，禁止以纯文本追问代替；将问题与选项结构化传入 AskUserQuestion，每次调用 ≤4 个问题；
```

目的：为未进行明确设定的交互操作进行兜底；

## 规则 2：Workflow 中的交互编写范式

Workflow 中涉及用户抉择的步骤，必须以固定范式编写，确保 AI 生成的结构化选项清晰可执行。

> **标准范式**：`通过 AskUserQuestion 提供选项（a / b / c），阻塞等待用户选择`

### 示例 1：单条件交互

```markdown
**原文**

交互式确认覆盖、合并或跳过；

**优化后**

通过 AskUserQuestion 提供选项（覆盖 / 合并 / 跳过），阻塞等待用户选择；
```

### 示例 2：确认性交互

```markdown
**原文**

由用户确认是否采用、留空或跳过；

**优化后**

通过 AskUserQuestion 提供选项（采用 / 留空 / 跳过），阻塞等待用户选择；
```

> **危险场景**：使用模糊表述替代范式
> *示例*：`询问用户如何处理` -> AI 可能输出纯文本问题而非调用工具
> *正确*：`通过 AskUserQuestion 提供选项（...），阻塞等待用户选择`

## 规则 3：动态选项

当选项内容依赖上下文分析结果、无法在 SKILL.md 中预先枚举时，使用动态选项模式。

> **标准范式**：`通过 AskUserQuestion 提供选项（由 AI 根据 [分析对象] 生成 N 个选项），阻塞等待用户选择`

### 示例：依赖上下文生成的选项

```markdown
**原文**

需要用户确认 description 格式的修复方案；

**优化后**

通过 AskUserQuestion 提供选项（由 AI 根据 description 缺失类型生成修复方案），阻塞等待用户选择；
```

> **适用条件**：选项取决于待优化内容的具体特征（如缺失类型、不符合项内容、拆分领域），预定义不可行时使用。

## 常见反模式

### 反模式 1：纯文本追问

用普通消息替代 `AskUserQuestion` 工具调用。

> **错误**：`向用户确认是否删除`
> **问题**：AI 以文本输出代替 UI 确认，可能不阻塞等待
> **正确**：`通过 AskUserQuestion 提供选项（删除 / 保留），阻塞等待用户选择`

### 反模式 2：选项描述缺失

只提"询问用户"但未给出具体选项。

> **错误**：`询问用户选择处理方式`
> **问题**：AI 不知道应提供哪些选项，可能遗漏关键选项
> **正确**：`通过 AskUserQuestion 提供选项（覆盖 / 合并 / 跳过），阻塞等待用户选择`

### 反模式 3：超量选项

单次调用超过 4 个问题。

> **错误**：一个 `AskUserQuestion` 调用包含 6 个问题
> **问题**：UI 展示混乱，用户决策负担重
> **正确**：拆分为多次调用，每次 ≤4 个问题

## 验证清单

- [ ] Rules 中包含使用 `AskUserQuestion` 的行为约束规则
- [ ] Workflow 中每个交互环节使用标准范式 `通过 AskUserQuestion 提供选项（...），阻塞等待用户选择`
- [ ] 每个交互步骤给出了明确的 2-4 个选项，或标注了动态选项模式（由 AI 根据上下文生成）
- [ ] 动态选项模式正确标注了 AI 生成选项的依据和范围
- [ ] 每次 `AskUserQuestion` 调用 ≤4 个问题
- [ ] 不存在以纯文本代替交互确认的表述
