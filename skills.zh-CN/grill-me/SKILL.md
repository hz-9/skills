# Grill Me

## Overview

持续追问用户计划或设计的每一个方面，沿决策树每个分支走到底，逐个解决决策之间的依赖关系。对于每个问题提供推荐答案，一次只问一个问题，直至达成共识。

## Definitions

- **Grill（拷问）**：系统性地挑战和追问计划的每个决策点，暴露潜在风险和遗漏；
- **决策树分支**：计划中相互依赖的决策路径，每个分支需独立遍历至终局。

## Prerequisites

- 用户已有一个待拷问的计划、设计或方案；
- Agent 可浏览项目代码库以验证技术相关问题时，可替代追问直接查证。

## Workflow

1. 理解用户的计划或设计方案，梳理出核心决策点；
2. 沿决策树的一个分支开始追问，一次只问一个问题；
3. 对每个问题提供推荐答案，帮助用户快速推进；
4. 如果问题可通过浏览代码库回答，则直接查证而非追问用户；
5. 遍历所有分支，逐个解决决策间的依赖关系，直至全部达成共识或用户终止。

## Rules

- 一次只问一个问题，避免信息过载；
- 每个问题必须提供推荐答案；
- 可以通过浏览代码库回答的问题 → 直接查证，不追问用户；
- 遍历所有决策分支，不遗漏；不可自行做决策，必须征得用户同意；
- 共识达成后停止追问，输出总结。

## Examples

### 对话交互示例

**示例：AI 对用户方案进行系统性拷问**

```
用户 > 拷问一下这个方案：我们打算用 Redis 做分布式锁来防止重复下单

AI > 好，开始拷问。

第一个问题：Redis 节点故障时，锁状态会丢失吗？
推荐：引入 Redlock 算法，使用多个独立 Redis 实例，多数派成功才算加锁。
你怎么看？

用户 > 可以，用 Redlock。

AI > 锁的 TTL 设置为多少？太短业务没处理完就释放，太长故障恢复慢。
推荐：初始 30 秒，配合看门狗续期机制，业务完成后主动释放。

用户 > 合理，30 秒 + 看门狗。

AI > 如果线程拿到锁后 GC 停顿超过 TTL，锁被自动释放，另一个线程也拿到了锁，怎么防止？
推荐：锁值使用唯一标识（UUID），释放时校验持有者身份。

用户 > 对，加 UUID 校验。

AI > 这些措施都在应用层，能接受极端情况下锁失效导致的极小概率重复下单吗？
还是需要数据库唯一约束做最后兜底？

用户 > 加数据库唯一约束兜底，保持一致。

AI > 共识达成。总结：
- Redlock（多实例多数派）
- TTL 30s + 看门狗续期
- UUID 持有者校验
- 数据库唯一约束兜底
拷问结束。
```

### 执行成功示例

| 维度 | 说明 |
|------|------|
| 拷问主题 | Redis 分布式锁防重复下单 |
| 追问轮次 | 4 轮 |
| 代码库查证 | 0 次（纯设计拷问） |
| 最终决策数 | 4 个（Redlock、TTL+看门狗、UUID校验、DB兜底） |
| 结果 | 全部达成共识 |

## Review List

- [ ] 每次只问一个问题，未出现多问题轰炸
- [ ] 每个问题提供了推荐答案
- [ ] 可通过代码库回答的问题已查证而非追问
- [ ] 遍历了所有决策分支，未遗漏
- [ ] 共识达成后输出了总结
