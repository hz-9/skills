# Grill Me

## Overview

持续追问用户计划或设计的每一个方面，沿[决策树分支](#decision-tree-branch)每个分支走到底，逐个解决决策之间的依赖关系。对于每个问题提供推荐答案，一次只问一个问题，直至达成共识。

## Definitions

- <a id="grill"></a>**Grill（拷问）**：系统性地挑战和追问计划的每个决策点，暴露潜在风险和遗漏。
- <a id="decision-tree-branch"></a>**决策树分支**：计划中相互依赖的决策路径，每个分支需独立遍历至终局。

## Prerequisites

- 用户已有一个待拷问的计划、设计或方案。
- Agent 可浏览项目代码库以验证技术相关问题时，可替代追问直接查证。

## Workflow

0. **前置检查** — 确保拷问条件已满足；
    - 判断用户是否已有一个待拷问的计划或方案：
        - 是 -> 进入步骤 1（理解计划）；
        - 否 -> 通过 AskUserQuestion 提供选项，阻塞等待用户选择：
          - 自行描述计划 -> 等待用户输入计划内容，接收后返回步骤 0 重新检查；
          - 由 AI 引导 -> AI 逐步提问引导用户梳理计划，完成后返回步骤 0 重新检查；

1. **理解计划** — 梳理核心决策点；
    - 优先选择依赖关系最多、风险最高的分支作为起始分支；
    - 理解完成后 -> 进入步骤 2（开始追问）；

2. **开始追问** — 沿[决策树分支](#decision-tree-branch)逐层深入；
    - 判断当前待追问问题是否可通过浏览代码库回答：
        - 是 -> 直接查证而非追问用户，查证后判断是否仍有待追问问题：
          - 仍有问题 -> 继续按本步骤后续流程追问；
          - 所有问题已解决 -> 跳至步骤 7（成果输出）；
        - 否 -> 进入下一环节；
    - 通过 AskUserQuestion 提供推荐答案与选项，阻塞等待用户选择（一次只问一个问题）：
      - 同意推荐答案 -> 记录决策，进入共识判断；
      - 不同意 -> 通过 AskUserQuestion 追问具体原因，提供备选方案，阻塞等待用户选择：
        - [动态选项，由 AI 根据用户反馈生成] -> 记录决策，进入共识判断；
    - **共识判断** — 判断当前是否已与该分支达成一致：
        - 是 -> 下一步；
        - 否 -> 记录分歧点并继续后续追问，避免卡死；

3. **范围检查** — 判断当前追问是否超出范围；
    - 判断问题是否涉及基础设施或底层技术选型且不影响计划可行性：
        - 是 -> 通过 AskUserQuestion 提供选项，阻塞等待用户选择：
          - 继续深入 -> 继续追问，执行后进入下一步；
          - 跳过分支 -> 跳过当前分支，执行后进入下一步；
        - 否 -> 下一步；
    - 判断追问是否偏离核心目标超过两层依赖：
        - 是 -> 通过 AskUserQuestion 提供选项，阻塞等待用户选择：
          - 继续深入 -> 继续追问，执行后进入下一步；
          - 回到主线 -> 回到核心目标，执行后进入下一步；
        - 否 -> 下一步；

4. **回溯检查** — 判断是否发现与先前决策存在冲突；
    - 判断是否发现与先前决策存在冲突：
      - 是 -> 回溯至相关分支重新协商：
        - 回溯后判断是否再次遇到已记录分歧：
          - 是 -> 将分歧升级为阻塞项，输出总结并建议用户单独讨论；终止流程；
          - 否 -> 继续追问；
      - 否 -> 下一步；

5. **遍历检查** — 判断是否已遍历所有决策分支；
    - 判断是否已遍历所有决策分支：
      - 是 -> 下一步；
      - 否 -> 返回步骤 2，选择下一个决策分支；

6. **复核检查** — 对照 [Review List](#review-list)，确认执行结果；
    - 判断 Review List 是否有内容：
        - 否 -> 直接进入下一步（成果输出）；
        - 是 -> 下一步；
    - 依次判断 [Review List](#review-list) 中每个检查项，是否通过：
        - 基于“复核检查示例”来显示输出内容；
        - 有任一检查失败 -> 终止流程；
        - 全部通过 -> 下一步；

7. **成果输出** — 输出决策总结；
    - 判断是否已达成共识：
        - 是 -> 输出最终决策总结；
        - 否（用户终止）-> 输出当前已达成的共识与未解决分歧总结；

## Rules

### 追问规范
    - 一次只问一个问题，避免信息过载。
    - 每个问题必须提供推荐答案。
### 决策规范
    - 可以通过浏览代码库回答的问题 -> 直接查证，不追问用户。
    - 遍历所有决策分支，不遗漏。不可自行做决策，必须征得用户同意。
### 分歧处理
    - 用户不同意推荐答案时，追问具体原因，提供备选方案。
    - 若仍无法达成一致，记录分歧点并继续后续追问，避免卡死。
### 回溯规范
    - 发现与先前决策存在冲突时，必须回溯至相关分支重新协商，直至所有分支的决策保持逻辑一致。
    - 回溯后若再次遇到已记录分歧，将分歧升级为阻塞项，输出总结并建议用户单独讨论，避免无限循环。
### 范围规范
    - 拷问范围限定在用户计划的直接决策层面，不追问基础设施或底层技术选型的原因，除非该原因直接影响计划可行性。
    - 如果追问偏离计划核心目标超过两层依赖，提醒用户是否继续深入。
### 终止规范
    - 共识达成后停止追问，输出最终决策总结。
    - 用户主动终止时，输出当前已达成的共识和未解决的分歧点总结。
### 交互规范
    - 所有涉及用户抉择的交互环节，**必须**使用 `AskUserQuestion` 工具，禁止以纯文本追问代替；将问题与选项结构化传入 AskUserQuestion，每次调用不超过 4 个问题；
### 自洽性规范
    - Review List 检查项必须覆盖 Rules 中所有约束性规则，确保 Rules 与 Review List 一一对应；

## Examples

### 对话交互示例

**示例：AI 对用户方案进行系统性拷问（使用 AskUserQuestion）**

``` markdown

用户 > 拷问一下这个方案：我们打算用 Redis 做分布式锁来防止重复下单

AI > 好，开始拷问。

AI 通过 AskUserQuestion 提供选项，阻塞等待用户选择：
  - 使用 Redlock（多实例多数派） -> 记录决策，继续；
  - 有其他想法 -> 追问具体方案；

用户 > 使用 Redlock（多实例多数派）

AI > 已记录：使用 Redlock。

AI 通过 AskUserQuestion 提供选项，阻塞等待用户选择：
  - TTL 30s + 看门狗续期 -> 记录决策，继续；
  - 有其他想法 -> 追问具体方案；

用户 > TTL 30s + 看门狗续期

AI > 已记录：TTL 30s + 看门狗。

AI 通过 AskUserQuestion 提供选项，阻塞等待用户选择：
  - UUID 持有者校验 -> 记录决策，继续；
  - 有其他想法 -> 追问具体方案；

用户 > UUID 持有者校验

AI > 已记录：UUID 持有者校验。

AI 通过 AskUserQuestion 提供选项，阻塞等待用户选择：
  - 数据库唯一约束兜底 -> 记录决策，结束追问；
  - 接受极端情况 -> 记录决策，结束追问；

用户 > 数据库唯一约束兜底

AI > 共识达成。总结：
- Redlock（多实例多数派）
- TTL 30s + 看门狗续期
- UUID 持有者校验
- 数据库唯一约束兜底
拷问结束。
```

## Review List

完成拷问后，验证以下内容：

- **追问质量检查**
    - [ ] 每次只问一个问题，未出现多问题轰炸
    - [ ] 每个问题提供了推荐答案
- **完整性检查**
    - [ ] 可通过代码库回答的问题已查证而非追问
    - [ ] 遍历了所有决策分支，未遗漏
    - [ ] 未自行做决策，每个决策均征得了用户同意
    - [ ] 发现决策冲突时已回溯并重新协商
- **分歧与回溯检查**
    - [ ] 用户不同意时已追问原因并提供备选方案
    - [ ] 无法达成一致时已记录分歧点并继续后续追问
    - [ ] 回溯中再次遇到已记录分歧时，已升级为阻塞项并建议单独讨论
- **范围检查**
    - [ ] 未追问基础设施或底层技术选型（除非直接影响计划可行性）
    - [ ] 追问偏离核心目标超过两层依赖时，已提醒用户是否继续
- **收尾检查**
    - [ ] 共识达成后输出了最终决策总结
    - [ ] 用户主动终止时输出了当前共识与分歧总结
- **交互规范检查**
    - [ ] 交互规范：所有用户抉择交互使用 AskUserQuestion，禁止纯文本追问
    - [ ] 自洽性：Review List 检查项与 Rules 中所有约束性规则一一对应，不遗漏
